Purpose

This document defines the authoritative design-level specification for the new SMASH shared power system.

It is a conceptual model plus the currently hardened implementation-facing translation layer.

It is not final schema code, final UI art direction, runtime tracking design, or a code patch plan.

Its purpose is to freeze the rules language and implementation-safe boundaries that later implementation must obey.


Phase Boundary

This specification defines the power foundation plus the hardened outcomes from Phase 2, Phase 3, Phase 4, and the current Phase 5 implementation-facing translation work.

Core Design Truth

SMASH is imagination-first.

Players should imagine the character and power first, then realize that fantasy in a mechanically viable and balanced form.

The power system must therefore be structurally clear without collapsing into either vague flavour language or rigid taxonomy theatre.


Power Structure Overview

A power consists of:

Power-Level Fields

one Descriptor Chassis

one or more Effect Packets

This is the shared conceptual model for powers.


1. Shared Power Core

The Shared Power Core is the full cross-system model used to define powers.

It includes:

Power-Level Fields

Descriptor Chassis

Effect Packets

Primary Packet defence gating

the timing model built from Lifespan, Effect Timing, and Effect Duration

This core is intended for Summoning Circle first and later reuse in the Character Builder.


2. Power-Level Fields

Power-Level Fields define the overall cast context and common rules grammar of the power. They apply to the power as a whole rather than to an individual packet.

They answer:

what kind of power this is

how it is used

what shared context applies to all packets inside it

Current agreed examples include:

Name

Description

Delivery Method

Range Category

Targeting information

AoE data where relevant

Dice

Counter

Lifespan where relevant

Commitment Modifier

Power-Level Fields define how the power is used. They do not define the full contents of each packet.


2.1 Commitment Modifier

Commitment Modifier is a Power-Level Field that defines whether the user must or may continue spending Power Actions to build, sustain, or operate the power beyond the default one-action use pattern.

The builder should present Commitment Modifier as an always-visible selector with the following options:

Standard

Channel

Charge

Standard is the default.

Channel means the user may or must continue consuming later Power Actions to maintain, reapply, or sustain the power after initial use.

Charge means the user must or may consume one or more later Power Actions building power before Packet 1 resolves, or may grow the power’s eventual result while the power remains held before cast or release depending on chassis.

Commitment Modifier is not a Descriptor Chassis.

It is a separate power-level layer that changes action commitment and likely cost, but does not replace the underlying structural form of the power.

Commitment Modifier must not be collapsed into Lifespan, Effect Timing, or Effect Duration. Those fields answer different questions.

Commitment Modifier also does not create new packet timing categories. However, it may constrain legal Packet 1 resolution windows and operational windows.

In first implementation:

Channel uses one behaviour model only as an action-commitment modifier. It does not automatically reroll Packet 1 or create repeated packet resolution by itself.

Authored Effect Timing determines when each Effect Packet resolves.

Charge may constrain when Packet 1 is allowed to resolve and how long a power may remain in a usable pre-resolution state, but it does not replace the meaning of Lifespan, Effect Timing, or Effect Duration.

First-pass Channel support is limited to Immediate, Field, and Attached powers. Trigger + Channel and Reserve + Channel are deferred.

In first implementation, Charge uses one active Charge Type only:

Delayed Cast

Build Power

Delayed Cast is the design-facing term. The current code may still use the implementation-internal enum/value DELAYED_RELEASE for stability.

Build Power uses bonus dice per charged turn.

Delayed Cast is not Reserve-only.

Delayed Cast cannot be Counter-enabled.

Reserve + Charge delayed-cast:

becomes primed after the required charge turns

once primed, may be held in reserve for up to its authored Reserve Lifespan

if not released before that Reserve hold window ends, it is lost and goes on cooldown

Reserve keeps release permission and release-flavoured wording.

Non-Reserve delayed-cast:

becomes primed after the required charge turns

must be cast with a Power Action before the end of the user’s next turn

if not cast, is lost

Non-Reserve delayed-cast does not inherit Reserve release framing.

For non-Reserve delayed-cast, Packet 1 timing and defence wording follow the authored Packet 1 Effect Timing rather than a forced ON_RELEASE bridge.

Build Power remains separate from Delayed Cast. On non-Reserve chassis its player-facing wording uses cast framing such as before cast, on cast, and when the power is cast.

In the current builder surface, Immediate + Standard Packet 1 uses On Cast only.

If an Immediate Packet 1 resolves On Cast and does not create a beyond-turn carrier, later Immediate packets are also restricted to On Cast in first implementation.

Secondary scaling from the Primary Packet result class is derived rather than manually selected in the current builder surface.


2.2 Counter

Counter is a Power-Level Field that defines whether the power may be used as a Counter when the user spends a Response on self-defence.

The builder should present Counter as an always-visible selector with the following options:

No

Yes

No is the default unless later costing rules say otherwise.

Counter is not a Descriptor Chassis.

Counter is not the same thing as Trigger.

Counter is not the same thing as the player-facing Response resource.

Responses are tracked by the player and may be spent on Assists or Counters.

Assist is universal by system rule and is not authored per power.


2.3 Response Window / Incarnate Stack

The response window is the shared resolution model for hostile actions, Assists, Counters, active defences, and cleanup.

When a hostile action is declared, resolve the stack in this order:

1. Apply already-active effects.

2. Declare the hostile action and targets.

3. Declare legal Assists.

4. Declare legal Counters.

5. Roll and apply Assists.

6. Roll and apply Counters.

7. Roll the triggering hostile action.

8. If no Counter was declared by a defender and the hostile action produced one or more actionable hostile successes, that defender rolls the chosen normal active defence.

9. Calculate prevention and effects.

10. Apply the incoming result and the Counter result.

11. Process defeats and cleanup.

All declared actions, Assists, Counters, and Responses resolve before defeat and cleanup unless a power explicitly interrupts or cancels the triggering action.

Attack Counter damage does not cancel the triggering action unless the Counter explicitly has interrupt or cancel text.

Zero-success hostile action rule

When a legal hostile action is declared against a target, it creates the normal declaration/reaction window for Counters, Assists, and declaration-triggered effects before the hostile roll is made.

Actionable hostile successes are incoming successes or hostile units that create something the defender must actually oppose, prevent, avoid, cancel, or suffer. Examples include attack successes, wound-bearing successes, hostile effect successes, control/debuff units, ongoing-effect application successes, or resistible hostile units. If the incoming hostile action produces zero actionable hostile successes, normal defence does not roll and does not degrade.

If the hostile action later produces zero actionable hostile successes, it still counts as an attack or targeting event, but it is not a hit and applies no successful hostile effect. "When attacked" and "when targeted by a hostile action" effects may trigger from a zero-success hostile action. "When hit", "when damaged", "when wounded", "when affected", or "when successfully struck" effects do not trigger from a zero-success hostile action.

The defender does not roll normal Dodge, Physical Defence, Mental Defence, or active Resist, does not commit defensive-pool points, and does not suffer resolution-based defence degradation from normal defence against that action.

Counters, Assists, and declaration-triggered responses that were declared still count as used and resolve according to their own rules unless they specifically require the incoming action to hit, wound, damage, affect, or succeed.

Design intent: characters react to a declared threat before knowing whether the attacker rolled well, but defence degradation represents meaningful defensive exertion against actual pressure rather than merely raising a shield against an attack that whiffs before contact.

Independent defence-lane degradation

Defence degradation is tracked independently by defence lane. A defence lane degrades only when that specific lane actually resolves against one or more actionable hostile successes.

Physical Defence degradation does not affect Mental Defence. Mental Defence degradation does not affect Physical Defence. Dodge uses its own degradation track and does not degrade Physical Defence or Mental Defence.

Active Resist degradation applies only to the relevant resisted lane or attribute unless an explicit rule states otherwise.

Repeated pressure in one defensive channel does not automatically weaken unrelated defensive channels. Blocking or defending against several physical attacks does not make the defender worse at resisting a later mental attack. Resisting several mental attacks does not make the defender worse at blocking or defending against later physical attacks.

Design intent: defence degradation represents repeated pressure in the same defensive channel, not generic exhaustion that weakens every defensive response. This preserves lane identity and prevents mixed-threat groups from creating abusive cross-lane defence stripping.

Counter replacement rule

Any Counter replaces the defender's normal active defence against that trigger.

If a Counter is declared, the defender does not also roll Dodge, Physical Defence, Mental Defence, or Resist unless the Counter explicitly grants or modifies that defence.

Already-active passive effects, static protection, buffs, debuffs, and prepared states still apply because they were established before the response window.

Preparation stacks. Panic buttons replace.

Counter-enabled powers and normal use

Counter: Yes adds a legal Counter mode. It does not make the power Counter-only.

The same power may still be used normally with a Power Action.

Normal use follows the normal resolution path and may allow the normal defence gates generated by the Primary Packet.

Counter use follows the response-window Counter path and replaces the defender's normal active defence against that trigger.

Counters and Assists use the same cooldown lifecycle as the power being executed.

Assist ordering

Assists are declared before Counters and roll/apply before Counters.

An Assist may modify a Counter roll, a normal defence if no Counter is declared, or another legal response outcome.

The assisted actor still chooses one active response path: normal defence or Counter.

Attack Counters

Attack Counters are legal.

They roll during the Counter step before the triggering hostile action rolls.

Their damage or effect is known before final result application.

The triggering hostile action still rolls and resolves unless the Attack Counter explicitly interrupts or cancels it.

The triggering attacker does not get an active defence roll against the Attack Counter.

Already-active passive or static protection still applies.

Defence Counters

Defence Counters are specialist alternative defensive responses.

They replace normal Dodge, Physical Defence, Mental Defence, or Resist rather than stacking with those normal gates by default.

They may grant or modify a normal defence only when the power explicitly says so.

Mixed defensive Counters

If a Counter contains multiple active defence packets, the actor chooses one eligible active defence path when declaring the Counter, before the incoming roll and before the Counter roll are made.

Only the chosen active defence path resolves.

Unchosen active defence packets do not apply, do not create pools, do not provide successes, do not mitigate, and do not act as fallback protection against that incoming packet.

This is the default for mixed defensive Counters.

Counter declaration includes the active defence path choice when the Counter has more than one active defence path. The choice is made after the incoming hostile action is declared, but before incoming and Counter rolls are made. The actor may use known declared information, such as attack type, target, damage type, range, and visible threat context, to choose the mode. The actor may not wait to see the roll result before choosing unless the power explicitly has Adaptive Counter wording.

Terminology:

- Single-lane Counter: a Counter with one active defence path. Examples include Dodge Counter, Physical Defence / Block Counter, Mental Defence / Block Counter, and Resist Counter.
- Modal Defence Counter: the default for a Counter with multiple active defence paths. The actor chooses one eligible active defence path when declaring the Counter. Only the chosen path resolves. Unchosen active defence packets do not apply.
- Adaptive Counter: a premium / explicit exception. The power explicitly allows the actor to choose the active defence path after seeing the relevant roll/result. This is not default behaviour and is future / explicit authoring only.
- Layered Defence Counter: a premium / explicit exception. The power explicitly allows more than one active defence path to apply against the same incoming packet. Example: try to Dodge; if Dodge fails, also Block. This is not default behaviour and is future / explicit authoring only.

Example: a Counter contains both a Dodge packet and a Physical Defence / Block packet. When the Counter is declared, the actor chooses Dodge mode or Block mode. If Dodge mode is chosen and the Dodge result is not enough to avoid the incoming packet, the actor may not then switch to Block. If Block mode is chosen, the Dodge packet does not apply.

Rationale: Dodge and Block have different lane identities. Dodge is high-ceiling, all-or-nothing avoidance. Block / Defence is more reliable partial mitigation. Allowing the actor to roll first and choose afterwards would create post-result optimisation. That would turn mixed Dodge + Block Counters into Dodge upside plus Block insurance, making single-lane Counters easier to obsolete and harder to price fairly. Mixed defensive Counters are still valuable because they provide matchup versatility at declaration time, not post-roll insurance.

Passive/static mitigation remains governed by existing mitigation rules. This rule does not change Attack Counters, Movement Counters, Control Counters, Assist rules, or the global Counter timing order except where those powers also contain multiple active defence packets. The Modal Defence Counter choice happens inside "Counter is declared".

Movement Counters

Movement Counters can avoid or cancel a trigger only if both timing and geometry are sufficient.

First-pass Movement Counter timing is a Clash against the triggering action successes.

The Movement Counter must strictly exceed the trigger successes to avoid or cancel the trigger. Ties fail.

The movement component may still occur even when the avoid/cancel result fails if the power text says the movement happens.

Control Counters and Control Assists

Control Counters and Control Assists can halt an incoming action only when Control is the Primary packet / Primary intention of that Counter or Assist, and only when the chosen Control effect could logically stop the incoming action.

An interrupting Control Counter or Assist succeeds only if its Control result strictly exceeds the incoming action's successes. Ties fail. If the interrupting Control result does not strictly exceed the incoming action, the interrupt fails completely: the incoming action resolves normally, and its successes, wounds, potency, stacks, duration, movement, target count, and other outputs are not reduced. Failed interrupting Control is not partial mitigation and does not become weaker Block, Resist, Dodge, tax, or damage reduction.

The attacker does not make a separate Resist roll against an interrupting Control Counter or Assist by default. The incoming action's successes are the interruption contest value because they represent the strength, momentum, or quality of the action being interrupted.

Force No Main Action can interrupt an incoming Main Action if it strictly exceeds. Force No Power Action can interrupt an incoming Power Action if it strictly exceeds. A movement-stopping Control may interrupt a charge or movement-dependent action if it logically prevents the action. A Control that only affects later movement cannot cancel a melee attack that is already in range unless explicitly authored. A Control that only applies an after-effect condition cannot retroactively stop the triggering action.

Control as a Secondary packet is after-effect Control by default. It cannot interrupt, cancel, halt, weaken, reduce, or partially mitigate the triggering incoming action. It applies only after the triggering action resolves, if powered by the Primary result and if its own timing / legality conditions are met.

If a Counter has Attack Primary + Control Secondary, the Control does not use strict-exceed interruption. The Attack may resolve as counter damage, and the Control is an after-effect only; the incoming action is not halted.

If a Counter has Control Primary + Attack Secondary, the Control may use strict-exceed interruption. The Attack Secondary does not resolve regardless; it resolves only if powered by the successful Primary Control result, unless the power explicitly says otherwise.

If a successful interrupting Control also applies persistent Control stacks, duration, or an ongoing effect, those follow normal Primary-result scaling. Default scaling uses applied / net interrupt success, not raw successes, unless explicitly authored otherwise.

Dodge and Resist defensive power options

The shared builder needs explicit defence power options for Block, Dodge, and Resist.

These options must be shared by Summoning Circle and Character Builder.

Dodge and Resist Counter powers do not stack with normal Dodge or normal Resist. They are chosen active response paths. Resist is authored against one core Attribute lane and remains distinct from Cleanse cleanup/removal powers. One Defence -> Resist packet selects exactly one Resisted Attribute: Attack, Guard, Fortitude, Intellect, Synergy, or Bravery. Potency means Resists applied per successful power-roll success, so total Resists equals roll successes multiplied by Potency. Multi-attribute Resist requires multiple effect packets. Movement powers may be Counter-enabled and use Movement Counter timing, but Movement/Reposition is not a Defence mode.


Roll modifiers, failures, and success quality

Modifiers are applied to individual die results before success quality is determined, with one exception: a natural 1 is always a hard failure and cannot be modified into a success.

A natural 2 or 3 is a recoverable failure. Positive modifiers may raise a natural 2 or 3 to the normal success threshold, but a rescued recoverable failure can produce at most 1 normal success and cannot become a greater, critical, or higher-tier success.

A natural roll that already meets the normal success threshold is a natural success. Natural successes may be modified normally and may be upgraded or downgraded according to the current success-tier thresholds. Negative modifiers may reduce natural successes, including reducing them below the success threshold.

This rule preserves modifier usefulness for low-die characters while preventing positive modifiers from erasing all failure risk or granting spike output to rolls that were not natural successes.

Do not change the existing success thresholds, dice sizes, maximum modifier, or success-tier definitions as part of this ruling.


Duration defensive pools

Duration Dodge, Physical Defence Block, Mental Defence Block, and Resist powers create finite defensive pools.

Power descriptors use the canonical pool terms only: Physical Block Pool, Mental Block Pool, and Dodge Pool. The descriptor clause is "creates a [Pool Name] with N point(s) per success"; detailed pool operation remains in the shared rules.

They do not reroll or refresh their full value every time the protected actor is attacked unless explicitly authored to do so.

When the power applies, roll it once.

Pool size = successful power-roll successes x packet potency.

The pool remains until:

- the pool is spent to 0
- the effect duration expires
- the effect is removed, cleansed, or resisted away
- the source / target lifecycle ends the effect

The standard per-trigger commitment cap is packet potency unless the power explicitly modifies that cap.

Defensive pools may be used as pool-only protection or as a supplement to normal active defence. For each incoming hostile action, the defender chooses one legal path: take the hit/effect, commit one matching defensive pool only, roll the matching normal Dodge / Physical Defence / Mental Defence / active Resist only, or commit one matching defensive pool and roll the matching normal defence.

Commit timing:

1. The incoming action is declared and rolled.
2. If the incoming action produced one or more actionable hostile successes, the defender chooses whether to take the result, commit a matching pool only, roll the matching normal defence only, or use both.
3. If a pool is used, commit pool points up to the per-trigger cap and remaining pool before any normal defence or active Resist roll is made.
4. If a normal defence or active Resist is chosen, the defender rolls normal Dodge, Physical Defence, Mental Defence, or active Resist against the remaining hostile pressure.
5. Committed pool points add to the relevant result when a normal defence or active Resist is also rolled. In pool-only use, they directly provide their lane-specific cancellation or prevention.
6. Committed pool points are spent whether the final outcome succeeds or fails.
7. Pool-only commitment does not roll or resolve Dodge, Physical Defence, Mental Defence, or active Resist, so it does not cause resolution-based degradation.
8. If a normal defence or active Resist is rolled, resolution-based degradation applies only to the independent defence lane that actually resolves against one or more actionable hostile successes.

Dodge Pool points add to normal Dodge successes when Dodge is also rolled. In pool-only use, they count as Dodge successes for that incoming packet without rolling Dodge or degrading the Dodge lane. A Dodge Pool can be used only against physical attacks or actions that explicitly allow Dodge. It does not make mental attacks dodgeable.

Final Dodge with a normal Dodge roll = normal Dodge successes + committed Dodge Pool points. Pool-only Dodge result = committed Dodge Pool points. If the final Dodge result equals or exceeds incoming successes, the incoming packet is avoided. If the final Dodge result is lower than incoming successes, the Dodge fails and the incoming packet resolves normally.

Physical Block Pool and Mental Block Pool points add directly to wounds prevented in their matching lane, whether used pool-only or alongside normal Physical Defence / Mental Defence. A Physical Block Pool does not block mental wounds. A Mental Block Pool does not block physical wounds.

Resist Pool points add to the matching authored active Resist result when active Resist is also rolled. In pool-only use, each pool point cancels one hostile success, stack, ongoing unit, or equivalent unit in the authored resisted Attribute lane without rolling active Resist or degrading that lane. Resist Pool packets are not generic unless explicitly authored as generic, and one packet maps to one resisted Attribute.

Counter Dodge remains a one-shot Counter by default. It rolls on declaration, applies successes x potency to that one trigger, and does not create a persistent pool unless the power also explicitly has a duration effect that creates a Dodge Pool. A Counter still replaces normal active defence unless the Counter explicitly grants or modifies that defence. Already-active passive/static effects still apply.

Examples:

- Dodge Pool plus normal Dodge: A duration Dodge power rolls 4 successes with potency 3. It creates a Dodge Pool of 12. The protected actor is attacked for 5 incoming successes. Before rolling normal Dodge, they commit 3 Dodge Pool points because potency caps the per-trigger spend at 3. They roll normal Dodge and get 2 successes. Final Dodge = 5, so the attack is avoided. The pool drops from 12 to 9 and the Dodge lane degrades because normal Dodge resolved.
- Dodge Pool only: The protected actor is attacked for 2 incoming successes and commits 2 Dodge Pool points without rolling normal Dodge. Final Dodge result = 2, so the attack is avoided. The pool drops by 2, and the Dodge lane does not degrade because no normal Dodge roll resolved.
- Failed Dodge Pool spend: The protected actor commits 3 Dodge Pool points against 6 incoming successes. They roll normal Dodge and get 1 success. Final Dodge = 4, so the attack lands. The 3 committed pool points are still spent.
- Block Pool: A Physical Defence duration power creates a Physical Block Pool of 10. The protected actor commits 3 points before rolling Physical Defence. Normal Physical Defence prevents 8 wounds. The committed pool adds 3 more prevention. Total prevention = 11 wounds. The pool drops from 10 to 7 and the Physical Defence lane degrades because normal Physical Defence resolved.
- Block Pool only: A protected actor commits 3 Physical Block Pool points against 7 physical wounds without rolling Physical Defence. The pool prevents 3 wounds, 4 physical wounds remain, and Physical Defence does not degrade.
- Resist Pool: A Fortitude Resist duration power creates a Resist Pool of 8. A hostile Fortitude-lane ongoing effect applies 3 units. The protected actor commits 2 Resist Pool points before rolling Fortitude Resist. They roll 1 Resist success. Final cancelled units = 3, so the hostile units are cancelled. The pool drops from 8 to 6 and active Resist degrades because it resolved.
- Resist Pool only: The protected actor commits 2 Fortitude Resist Pool points against 2 Fortitude-lane ongoing units without rolling active Resist. Both units are cancelled, the pool drops by 2, and active Resist does not degrade.


Defensive pool chassis interactions

Trigger

Trigger changes when a packet is delivered. Once a defensive pool packet is delivered, it follows the normal defensive pool rules.

If Trigger delivers a duration Dodge, Block, or Resist effect:

- create a finite pool
- pool size = delivered power successes x potency
- per-trigger commit cap = potency unless explicitly modified
- duration begins when the Trigger packet is delivered unless the power explicitly says otherwise
- repeated delivery from the same source power, packet, and protected target uses refresh/replace, not additive stacking by default

Trigger does not get a special defensive pool exception.

Reserve

Reserve changes how a packet is prepared, held, and released. Once the packet is released, it follows the normal defensive pool rules.

If Reserve releases a duration Dodge, Block, or Resist effect:

- create a finite pool on release
- use the released packet's final dice and potency
- duration begins on release
- repeated release/application from the same source power, packet, and protected target uses refresh/replace, not additive stacking by default

Reserve does not get a special defensive pool exception.

Field

Field powers reapply or deliver effects according to authored chassis and Effect Timing.

When a Field delivers a defensive pool to legal targets:

- use the Field's existing roll/delivery cadence
- each legal target receives its own target-specific pool instance
- same source power + same packet + same protected target can have only one active pool instance at a time
- if the same Field packet reapplies to a target that already has that pool, roll the new application and use refresh/replace

Field refresh/replace means:

- active remaining pool becomes the higher of current remaining pool points or newly generated pool points
- duration/availability refreshes from the latest legal application
- pools do not add together by default

Field pools are location-bound by default:

- the pool is usable only while the protected actor remains a legal target of the Field, unless the power explicitly says the pool lingers after leaving
- when the Field ends, pools granted by that Field end unless explicitly authored to linger
- if a target leaves the Field, the Field pool ends or becomes unavailable according to authored Field rules
- if protection should follow the target independently, author it as Attached or as an explicit lingering rider

Attached

Attached powers follow the attached target while the attachment persists.

When an Attached defensive pool reapplies to the attached target:

- use the Attached power's authored delivery timing
- same source power + same packet + same protected target can have only one active pool instance at a time
- if reapplied, use refresh/replace
- when the attachment ends, the pool ends unless the power explicitly says the pool lingers

Refresh/replace rule

For repeated applications from the same source power and packet to the same protected target, default behaviour is:

- do not add pools together
- roll the new pool
- compare new pool points to current remaining pool points
- keep the higher value
- refresh duration/availability from the latest legal application

Pure replacement can make a beneficial reapplication worse. Additive stacking creates an armour/avoidance battery exploit. Higher-of-current-or-new preserves benefit without allowing uncontrolled accumulation.

Additive or recharging defensive pools

Additive defensive pools are not default behaviour.

A power may explicitly author additive or recharging pool behaviour, but this is a special mechanic requiring additional cost and limits. Likely constraints include:

- maximum pool cap
- clear expiry
- explicit stack/recharge wording
- likely lower per-trigger cap or separate cap
- heavy Power Tuning surcharge
- clear interaction with Field, Attached, and Channel delivery

Multiple pool interaction

A defender may commit from only one matching defensive pool per incoming action unless a power explicitly allows combined pool spending.

Examples:

- If two Dodge Pools are active, the defender chooses one Dodge Pool to commit from for that incoming attack.
- If one Block Pool and one Dodge Pool are active, the chosen defence mode determines which pool can be used.
- Defensive pools can still matter across multiple attacks because the defender may use different pools on different incoming actions.

Channel

Channel requires ongoing action economy sacrifice to maintain the power.

Default Channel defensive pool rule:

- Channelled defensive pools reapply while the channel is maintained according to authored Effect Timing
- each reapplication rolls normally
- if the same target already has a pool from the same source power and packet, use refresh/replace higher-of-current-or-new
- Channelled defensive pools do not add by default
- if the channel ends, channel-dependent pools end unless the power explicitly says they linger
- Channel receives pricing credit/discount for action economy sacrifice, but repeated defensive refresh remains expensive

Channel should not compound by default. Compounding or layering Channel powers require explicit additive/recharging authoring and heavy pricing.

Charge

Charge requires action economy sacrifice up front before release.

Default Charge defensive pool rule:

- Charged defensive pool powers use final charged values at release
- if Charge grants bonus dice, those dice increase the release roll
- if Charge grants bonus potency, final charged potency increases both pool size and per-trigger commit cap
- duration begins on release, not during charge
- charge turns before release do not count against pool duration
- cooldown follows the existing lifecycle after the released effect fully resolves / after duration ends, according to existing cooldown rules

Charge dice versus Charge potency:

- bonus dice increase reliability and expected pool depth
- bonus potency increases total pool depth and per-trigger burst protection
- charged potency is especially expensive for Dodge and Resist because it increases denial/cancellation capability, not just total output


3. Shared Rules Grammar

Shared Rules Grammar is the common language used across powers for readable mechanical communication.

Examples include:

Range

Targeting

Dice

Counter

Shared grammar provides consistency across powers.

It is not replaced by Delivery Method.

Counter remains shared power grammar.

It answers whether a power may be used as a Counter when the user spends a Response on self-defence.

Counter is not the same thing as Trigger.

Assist is universal by system rule and does not need authored representation in the power builder.

Responses remain the player-facing resource and are not the authored power field.


3.1 AoE placement, self-origin wording, and overlap

AoE count, shape, and footprint remain part of shared cast context.

Where an AoE sphere uses cast range 0 from the user's own position, descriptor wording must distinguish between static placement and attached presence.

For static self-origin area text, wording should describe the sphere or spheres as centered on the user's current space.

For Attached self-origin area text, wording should describe the sphere or spheres as attached to the user.

This wording distinction exists to keep preview text honest without bloating every power with handbook text.

Overlapping areas from the same packet at the same timing resolve as one packet with multiplied intensity, not as multiple separate defended entry points.

The power rolls once.

The packet's hostile result is determined once.

That hostile result is multiplied by the number of distinct authored areas from that packet affecting that target at that timing.

One defence roll is then resolved against the multiplied result.

The resulting applied value is the Packet 1 applied result for that target.

Dodge avoids the whole overlapped packet on that target.

Protection-based defence resolves against the multiplied wound total.

Resist cancels against the multiplied hostile successes or other hostile effect units.


4. Delivery Method and Descriptor Chassis

Delivery Method is the authored structural choice.

Descriptor Chassis is the rules structure created by that choice.

Use the distinction this way:

Delivery Method = the selected structural identity from the authoring side

Descriptor Chassis = the mechanical form that identity creates

A chassis changes:

the mechanical sentence structure

the required supporting fields

the broad resolution pattern

the likely relevance of Lifespan

The current practical Descriptor Chassis matrix is:

Immediate

Field

Attached

Trigger

Reserve

These are the chassis that currently justify themselves by materially changing descriptor structure, required supporting fields, or broad resolution pattern.

Counter is not retained as a separate chassis.

Counter is now a shared Power-Level Field permission used to indicate whether a power may be used as a Counter.

Trigger remains the structural chassis for pre-armed automatic resolution.

Assist is universal by system rule and does not need authored per-power representation.

Conversion is deferred.

Sequence is deferred.

These remain later mechanics only if they eventually justify themselves as real structural forms rather than packet patterns or helper labels.


4.1 Chassis Standard

A chassis must change real authored behaviour.

If it does not materially alter descriptor structure, required fields, or broad resolution pattern, it should be revised, merged, or removed.


4.2 Chassis Tooltip Guidance

The builder-facing helper meaning of each chassis should read broadly as follows:

Immediate — This power resolves as part of use now. It does not create an ongoing structure of its own.

Field — This power creates a placed area that persists and resolves effects through that area.

Attached — This power attaches to a target, object, or state and resolves through that attachment.

Trigger — This power is armed now and fires automatically later if its trigger condition is met.

Reserve — This power is held now and released later when the user chooses or when its release rule says so.


4.3 Chassis Matrix

Immediate

Mechanical role:
The baseline use-now chassis.

Descriptor meaning:
The power resolves now through Packet 1 and does not create a meaningful ongoing chassis presence of its own.

no chassis-local required fields beyond shared Power-Level Fields and Packet 1

Optional supporting fields:
Secondary Packets
later packet timing
shared AoE data where relevant

Advanced-only concerns:
Application Mode only when Intention plus Specific do not settle the gate

Timing profile:
Lifespan is normally not structurally relevant
Packet 1 usually resolves on cast or equivalent immediate use timing
Packet 1 may still produce a lingering Effect Duration after successful application


Field

Mechanical role:
A spatially placed or projected structural presence that persists for a Lifespan and resolves effects through that footprint.

Descriptor meaning:
The power creates a meaningful placed area or ongoing field presence.

meaningful placement rule through shared range / targeting grammar
meaningful footprint through shared AoE / area grammar

Descriptor meaning:

meaningful attachment host or anchor interpretation through shared targeting / range grammar
Lifespan
for hostile delayed-entry cases, clear hostile-entry structure where needed

detach or expiry condition

trigger condition

5. Effect Packets

Each intention inside a power becomes an Effect Packet.

An Effect Packet is the smallest authored mechanical outcome unit inside the power.

Effect Packets are currently expected to contain:

Intention

Specific

Potency

Effect Timing

Effect Duration

any required type or subtype

supporting application details where relevant

Packets describe individual effects, not the shared cast context of the power.

Primary Packet

Packet 1 is always the Primary Packet.

The Primary Packet is the power’s defended entry point.

It sets the defence gate.

Secondary Packets

Any packet after Packet 1 is a Secondary Packet.

Secondary Packets are contingent follow-through effects.

They do not set separate defence gates.

They do not create separate defended entry points.

They only resolve if the Primary Packet lands.

This means the key property of a Secondary Packet is not merely that it is additional.

Its defining property is contingency.


6. Timing Model

The timing model uses three separate concepts.


6.1 Lifespan

Lifespan is how long the chassis-level structure remains active, armed, attached, deployed, sustained, or otherwise available to resolve.

Examples include:

how long a field exists

how long a trigger remains armed

how long an attached effect remains attached

how long a reserve may be held

Lifespan belongs to the structural presence of the power form.

The exact implementation schema ownership of Lifespan remains unresolved. The current docs preserve both of the following truths:

Lifespan is structurally meaningful

Lifespan still appears in current Power-Level Field examples where relevant

That ownership boundary must not be falsely treated as settled canon yet.


6.2 Effect Timing

Effect Timing defines when an Effect Packet resolves relative to the power’s use and lifecycle.

Examples include:

on cast

on hit

on trigger

at start of turn

at end of turn

on expiry

Effect Timing answers:

when does this packet happen


6.3 Effect Duration

Effect Duration defines how long the resolved result remains after successful application.

Examples include:

instant

until target’s next turn

X turns

passive until removed

Effect Duration answers:

once this packet has applied, how long does the result persist


6.4 Timing Triad Rule

Lifespan, Effect Timing, and Effect Duration are separate concepts.

They answer different questions:

Lifespan = how long the chassis exists

Effect Timing = when the packet resolves

Effect Duration = how long the packet’s result persists

They must not be collapsed into one generic timing field.


6.5 Commitment Modifier and the Timing Model

Commitment Modifier is separate from the timing triad.

It answers:

how the user must continue spending turns to operate the power

This is distinct from:

how long the chassis exists

when packets resolve

how long applied results remain

Commitment Modifier does not create new packet timing categories, but it may constrain legal Packet 1 resolution windows and operational windows.

Channel may require continued Power Action spend to keep a structure or effect operating across turns.

Charge may require or allow continued Power Action spend before Packet 1 resolves, and may scale the eventual result during that period.


6.6 Persistent states versus repeating direct effects

Persistent states and repeating direct effects do not use the same cleanup model.

Persistent states such as Control stacks, stack-based Debuffs, and other stack-bearing conditions apply now and then persist as a state.

Repeating direct effects such as ongoing damage, healing-over-time, or repeated direct restoration apply on a cadence and are tracked through ongoing value plus duration rather than stack count.

For Immediate + Standard packets that resolve On Cast and use a non-instant duration for a repeating direct effect, the effect applies initially on cast and then reapplies at the start of the affected target’s next turns for the authored duration.

This first-pass default cadence is a convenience rule for currently supported immediate repeating direct effects. It is not a new universal timing category.


6.7 Legal Main Action choices and cleanup model

During a Main Action, an actor may choose exactly one legal Main Action option:

- Attack with one currently equipped weapon.
- Interact with an object or use an item, such as interacting with an artifact, pulling a lever, reloading a weapon, drinking a medicinal tonic, or accessing a computer.
- Rush, moving an additional distance up to half the actor's maximum move distance.
- Brace, allowing each failed die in a Physical Defence or Mental Defence roll to be rerolled once while the Brace benefit is available. A die rerolled by Brace cannot be rerolled again.
- Resist, rolling the relevant Resist Attribute value against persistent stacks or one chosen ongoing damage effect.

Power use is not a Main Action by default. Powers are declared using a Power Action or legal Response unless a more specific future rule explicitly creates an exception.

The Resist Main Action has two cleanup lanes:

- Persistent-state cleanup removes stack count.
- Ongoing-damage cleanup removes ongoing damage units from one chosen ongoing damage effect.

This keeps normal active defence Resist, Defence -> Resist powers, Cleanse effects, and Main Action cleanup Resist separate.

Example: An actor rolls 5 Physical Defence dice. Three dice generate successes and two dice fail. Brace allows the actor to reroll the two failed dice once. The three successful dice remain unchanged.

Persistent-state cleanup uses the following first-pass model:

- the target may use the Main Action Resist option to attempt the authored cleanup Resist Attribute and remove stacks equal to rolled successes
- Cleanse effects remove stacks per success
- persistent-state stacks naturally degrade by 1 at the end of each turn unless a more specific rule overrides this
- when the source duration ends, the remaining stacks are removed

Ongoing-damage cleanup uses the following first-pass model:

- ongoing damage is not tracked as stacks
- Cleanse removes ongoing damage units from the target per success
- the target may use the Main Action Resist option to attempt a cleanup Resist roll against one chosen ongoing damage effect
- Physical ongoing damage uses Fortitude Resist by default for this cleanup roll
- Mental ongoing damage uses Bravery Resist by default for this cleanup roll
- each cleanup-roll success removes 1 ongoing damage unit from the chosen effect
- each ongoing damage unit removes the source effect's wounds-per-success / wounds-per-band value from the stored ongoing amount
- ongoing damage does not naturally degrade by 1 each turn
- expiry ends the ongoing damage effect
- same-source reapplication may refresh duration without adding potency unless explicitly authored otherwise


6.8 Cooldown lifecycle and complete resolution

Cooldown does not begin when a power is merely declared, cast, armed, attached, placed, held, or released.

Cooldown begins only after the power has completely resolved.

Once a power has completely resolved, place a cooldown die on that power equal to the derived or authored cooldown value currently being used by the implementation.

At the end of each subsequent turn the owning character takes, reduce the cooldown die by 1.

If the cooldown die is showing 1 when it would be reduced, remove the die instead.

Once the die is removed, the power is ready and may be used on the owner's next turn or legal Response window, subject to the power's normal permissions.

Immediate chassis cooldown timing

An Immediate chassis power completely resolves once:

- the caster has rolled to determine application successes
- any required Dodge, Protection, or Resist resolution has occurred
- results have been compared
- unblocked effects, wounds, successes, stacks, heals, or other result units have been applied
- any continuing Effect Duration tracking has been noted separately

An Immediate power may create an effect with a continuing Effect Duration. That continuing effect is tracked separately. It does not prevent the Immediate power itself from resolving and entering cooldown once its immediate resolution sequence is complete.

Non-Immediate Lifespan Turns cooldown timing

For Field, Attached, Trigger, and Reserve powers with Lifespan Turns X, the cast, execution, or release turn is Turn 0.

The power remains active through X subsequent caster-turn rounds.

The power completely resolves and enters cooldown only after the final resolution window of its last lifespan turn has closed.

The final resolution window closes once the last legal target has finished the relevant turn, trigger, entry, expiry, or interaction that could be affected by the power during that final lifespan turn.

For Field powers, the caster rolls on cast and at the start of each caster turn while the field persists where the authored timing requires that roll. Those rolls determine the successes applied by the field for that round.

For Attached powers, authored Effect Timing determines whether dice are rolled on cast, at the start of the caster's turn, at the end of the caster's turn, on expiry, or when a legal trigger condition occurs during the lifespan.

For Trigger and Reserve powers, the same final-resolution-window rule applies. Once the last legal target has resolved on the final turn of the lifespan, the power completely resolves and enters cooldown.

Passive Lifespan cooldown timing

A Passive Lifespan power does not enter cooldown merely because time passes.

A Passive Lifespan power enters cooldown only when:

- the caster chooses to end the power as a Response
- the caster chooses to end the power as a Power Action on their turn
- a Cleanse Power or Resist Action reduces the power's currently active successes to 0

When one of those events occurs, the power immediately completely resolves and the cooldown die is placed.

Counters and Assists

Counters and Assists use the same cooldown lifecycle as other executions of the same power.

The Response used to execute the Counter or Assist is treated as Turn 0 for lifespan tracking.

The power enters cooldown only when it completely resolves under the normal Immediate, Lifespan Turns, Passive, Charge, and final-resolution-window rules.

Charge and cooldown lifecycle

Charge is front-loaded before the power's release or cast.

After the authored Charge Value has been completed, the power becomes Primed.

Once Primed, the power must be used as a Response or Power Action before the end of the subsequent turn unless a more specific Reserve rule says otherwise.

The release or cast of the primed power is Turn 0 for lifespan tracking.

Charge does not cause the lifespan clock to begin until the power is actually released, cast, or otherwise executed.

7. Defence Gating

Response-window cross-reference:

Dodge, Physical Defence, Mental Defence, and Resist are normal active defence gates.

If a defender declares a Counter, that Counter replaces the normal active defence gate against that trigger unless the Counter explicitly grants or modifies the gate.

Already-active passive effects, static protection, buffs, debuffs, and prepared states still apply before the response window.

Preparation stacks. Panic buttons replace.

Only the Primary Packet is defended against.

The Primary Packet sets the Defence Gate.

The Defence Gate is the allowed defence choice-set generated by the Primary Packet’s nature.

The defence choices for the Primary Packet are not fully freeform.

They are gated based on packet nature, and may use a supporting Application Mode concept where Intention plus Specific is not enough to produce sensible defence options.

The system must prevent inappropriate combinations by default.


8. Mixed-Intention Powers

Multi-intention powers are supported through multiple Effect Packets.

Mixed-intention powers do not share one universal potency, one universal timing, or one universal duration in all cases.

Each packet may have its own potency, timing, and duration where required.

Power-Level Fields define the shared cast context.

Packet-level fields define the individual effects.

14. Phase 2 Addendum — Defence Gating, Resist Attributes, Dodge Resolution, and Application Mode

Defence Gating is determined from the Primary Packet only.

Packet 1 is always the Primary Packet.

If a power contains any hostile packet, the Primary Packet must be hostile.

A non-hostile packet may not occupy Packet 1 if any later packet is hostile.

This prevents mixed hostile / beneficial powers from bypassing the single defended entry point by placing a beneficial packet first.

A hostile Primary Packet always resolves to a defence gate.

NONE is reserved for non-hostile Primary Packets only.

The Defence Gate is assigned in the following order:

determine whether the Primary Packet is hostile or non-hostile

if non-hostile, assign NONE

if hostile, determine whether the defended event is external or internal

if external, determine whether the Primary Packet deals wounds as part of its defended result

if external and wound-dealing, determine whether Dodge is sensibly possible, whether Protection is sensibly possible, or whether both are sensible

if internal, assign Resist and determine the correct Resist Attribute

use Application Mode only if Intention plus Specific still do not produce a sensible gate result

A hostile Primary Packet never resolves to NONE under normal power-building rules.

Structural chassis presence may exist before the hostile defended entry point if the power is authored that way.

This means an Attached, Trigger, Field, or Reserve chassis may exist structurally before the Primary Packet resolves, provided that no earlier packet is being treated as a separate defended hostile entry point.

The system still permits only one defended entry point per power use.

Primary Packet defence results currently resolve as follows:

Non-hostile Primary = NONE

Hostile External, no wounds = Dodge

Hostile External, deals wounds, both responses sensible = Dodge or Protection

Hostile External, deals wounds, Dodge not sensible = Protection

Hostile External, deals wounds, Protection not sensible = Dodge

Hostile Internal / take-hold = Resist

Dodge means the target may avoid the hostile packet entirely during the standard response window.

Dodge is only a legal gate option if the target could plausibly avoid the hostile packet’s threatened point, line, path, or area during that response window.

If the packet’s area, spread, persistence, or contact state makes that implausible, Dodge is not a legal gate option.

Dodge Resolution Rule

When Dodge is the legal defence gate, the defender rolls their dodge dice.

If the defender’s successes match or exceed the attacker’s successes on the hostile Primary Packet, the hostile packet is avoided entirely.

If the defender’s successes do not match or exceed the attacker’s successes, the packet is not avoided.

This is a locked rules truth.

Dodge no longer requires the defender to exceed the attacker’s successes. Matching is sufficient.

Protection means the target resolves the hostile packet through a protection-based defence rather than full evasion.

Where the packet is physically harmful, Protection means Physical Defence.

Where the packet is mentally harmful, Protection means Mental Defence.

Protection is appropriate where wounds are being dealt and the hostile packet is being met from the outside rather than resisted as an internal take-hold effect.

Resist is used when the hostile packet’s defended event is an internalizing or take-hold hostile effect rather than an incoming external event.

Resist cancels hostile successes rather than blocking wounds.

Resist Attribute Matrix

A Resist-gated Primary Packet must target one Resist Attribute.

That attribute is chosen by the thing being attacked in the fiction, not by weak-stat fishing.

If the Primary Packet is a debuff whose main result is to reduce or suppress a specific attribute, that same attribute is the Resist Attribute.

This specific-attribute override takes priority over softer thematic interpretation.

Fortitude Resist is used when the hostile force attacks bodily endurance, toxin tolerance, disease resistance, physical shutdown resistance, or raw internal survivability.

Examples include toxins, paralysis onset, suffocation, bodily decay, disease, necrotic withering of the body, and internal burning.

Intellect Resist is used when the hostile force attacks cognition, focus, mental navigation, perception parsing, memory coherence, or illusion disentangling.

Examples include mind mazes, confusion, thought loops, memory scramble, mental puzzles, and sensory deception that must be mentally parsed.

Bravery Resist is used when the hostile force attacks courage, fear threshold, panic resistance, dominance resistance, or aggressive resolve.

Examples include fear, terror, panic, dread, intimidation, supernatural awe, and command-through-dominance.

Synergy Resist is used when the hostile force attacks relational grounding, belonging, trust, empathy, ally-bonding, or interpersonal anchoring.

Examples include isolation effects, bond-breaking curses, belonging collapse, trust corrosion, and loneliness-based mental assault.

Attack Resist is legal but narrower. It is used when the hostile force attacks offensive output, striking force generation, or the target’s ability to translate aggression into attack execution.

Examples include offensive-force drain, suppression of striking limbs, or direct dampening of attack output.

Guard Resist is legal but narrower. It is used when the hostile force attacks guard posture, evasive readiness, balance, defensive coordination, or internal motor control tied to defensive function.

Examples include vertigo curses, balance collapse, reflex jam, or internal motor desync that primarily damages defensive function.

Priority rules for ambiguous Resist cases are:

specific-attribute debuff override first

then the narrowest truthful fiction lane

do not collapse every physical internal effect into Fortitude

do not collapse every emotional effect into Bravery

do not use Attack or Guard merely to make the matrix look symmetric

one Resist Attribute per Resist-gated Primary Packet

Application Mode is a narrow disambiguation layer, not a general packet layer.

It is required only where Intention plus Specific still leave one or more of the following unclear:

whether the Primary Packet is external or internal

whether the defended result deals wounds

whether Dodge is physically plausible

whether Protection is materially sensible

which Resist Attribute is being attacked

whether the hostile entry point is on attach or on payload

If changing the take-hold method would not change the sensible gate result, Application Mode is unnecessary.

Builder-Facing Application Mode Rule

Application Mode should be hidden by default.

It should only surface in authoring when Intention plus Specific do not already determine the gate cleanly.

Simple powers should not ask the user for Application Mode.

Where Application Mode is required, it should present as a narrow guided choice that answers the unresolved mechanical question rather than as a broad flavour taxonomy.

In practice this means:

use no Application Mode control for obvious cases such as standard strikes, straightforward heals, obvious toxins, obvious fear effects, and direct specific-attribute debuffs

surface a small guided selector only for ambiguous cases such as blind, silence, restrain, poison delivery timing, mark application, or other packets where take-hold method changes the defence result

prefer context-sensitive selection chips for small binary or ternary choices

use a dropdown only when the ambiguous mode list is long enough that chips become clutter

The UI should not ask, what is the flavour of this effect

It should ask the smallest rules question needed to settle the gate.

Examples include:

is the hostile entry on attach or on payload

is this effect external contact or internal take-hold

is this a projected assault or an internal corruption effect

Many awkward hostile powers should resolve through honest packet ordering rather than bloating the gate logic.

The Primary Packet is whichever hostile result the design wants the target to defend against.

This means attack-first is common, but not mandatory.

Control-primary into damage-secondary remains legal.

Debuff-primary into damage-secondary remains legal.

Damage-primary into rider-secondary remains legal.

Secondary Packets may resolve at the same moment as the Primary Packet or later, but never earlier.

Secondary Packets never create a second defence gate.

Secondary Packets only resolve if the Primary Packet lands.

For Attached or delayed-hostility powers, two lawful hostile-entry patterns exist:

gate on attach, where the hostile contact or attachment is the Primary Packet and later consequences are Secondary follow-through

gate on payload, where the chassis attaches or arms structurally and the later hostile payload is the Primary Packet

These are distinct authored packet structures and should not be treated as the same power with cosmetic wording differences.

Transformation and Summoning remain deferred special cases.

They must not break the hostile-primary rule, the single defended entry point rule, or the contingent Secondary Packet rule, but their full packet treatment is postponed until their linked builder systems are ready.


15. Phase 4 Addendum — Packet Resolution Consolidation and Hardening

Universal Packet Resolution Sequence

The power-resolution sequence is:

1. declare power use using a Power Action or Response
2. declare legal targets or areas from the power’s shared cast context
3. establish the chassis state if the selected chassis creates one
4. when Packet 1’s timing occurs, roll the power dice
5. if Packet 1 is hostile, resolve only Packet 1’s legal defence gate
6. determine whether Packet 1 lands or fails
7. if Packet 1 lands, apply Packet 1 and start any Effect Duration it creates
8. if Packet 1 lands, resolve or schedule Secondary Packets at the same moment or later according to their own Effect Timing
9. if Packet 1 fails, discard all contingent Secondary Packets
10. track Lifespan separately for any chassis structure that remains active, armed, attached, placed, or held

Packet 1 Lands

Packet 1 lands if, after its required Trial or Clash resolution, it applies at least one operative result unit.

For non-wound packets, operative result is measured in net applied successes, stacks, heals, or other effect units.

For wound-dealing packets, operative result is measured in net wounds inflicted.

Applied Result Value

Secondary Packets survive only if Packet 1 lands.

If Packet 1 lands, contingent Secondary Packets may resolve immediately or later according to their own Effect Timing.

Where a Secondary Packet scales from the magnitude of a non-wound Primary Packet, use the Primary Packet’s net applied successes.

Where a Secondary Packet scales from the magnitude of a wound-dealing Primary Packet, use:

Applied Success Bands = CEILING(Net Wounds / Wounds Per Success)

This provides a stable success-equivalent scaling value after wound blocking has been resolved.

For default wound-band scaling, Wounds Per Success is the Primary Packet's total
simultaneous wounds per success across all damage components. Multi-damage
Primary Packets increase the band size; they do not generate separate bands per
damage type unless explicitly authored and priced.

Example 1 - single damage type:

Primary Packet deals 6 physical Slashing wounds per success. A Secondary Packet
that scales from Primary wound bands uses one band per 6 applied Primary wounds,
rounding up.

Example 2 - three simultaneous damage types:

Primary Packet has Potency 3 and deals Fire, Ice, and Slashing wounds. Each type
deals 6 wounds per success. Total simultaneous wounds per success is 18. A
default wound-band Secondary uses one band per 18 wounds in the Primary result,
rounding up.

Example 3 - explicit component-specific exception:

A power may explicitly say "for every 6 Slashing wounds in the Primary result".
That uses only the Slashing component and must be authored and priced as a
component-specific rider.

Example 4 - explicit per-component additive exception:

A power may explicitly say each damage component generates its own wound bands.
That is not default and must be treated as a special multi-component band engine
with additional pricing.

Unresolved future design debt - overkill-scaling and wound-band source:

Should Primary wound-band scaling use actual wounds suffered after
remaining-health cap, or net wounds before overkill is discarded? Default should
probably use actual wounds suffered to avoid farming riders from overkilling 1
HP targets. Overkill-scaling may be allowed later only as an explicit premium
mechanic.

Vulnerability, Resistance, Protection, and wound-band scaling

Vulnerability, Resistance, and Protection trigger from damage types on the
incoming packet. If the incoming packet has a triggering damage type, apply the
relevant Vulnerability / Resistance modifiers and Protection dice to the one
defence roll against that packet, then calculate the Primary result normally.

Vulnerability applies a negative modifier to the relevant defence roll. It does
not increase wounds directly, multiply wounds, or alter the wound-band
denominator.

Resistance applies a positive modifier to the relevant defence roll. It does not
reduce wounds directly, divide wounds, or alter the wound-band denominator.

Protection adds additional dice to the relevant defence roll. It does not reduce
wounds directly, subtract wounds, or alter the wound-band denominator.

Vulnerability and Resistance compare and combine as defence-roll modifiers. For
example, Vulnerability Ice 2 and Resistance Ice 1 against an incoming Ice packet
yields net -1 to the defence roll. The result is not extra Ice wounds or
reduced Ice wounds; it is a net defence modifier before the defence roll is
resolved.

If an incoming packet has multiple damage types, each relevant V/R/P entry may
trigger. Apply all relevant defence modifiers and bonus dice before the one
defence roll against that packet. Do not split the packet into separate defence
rolls by damage type unless an explicit future rule says otherwise.

For Secondary Packet wound-band scaling, resolve the Primary Packet normally,
including any V/R/P effects on the defence roll, then calculate wound bands from
the resulting Primary wound total using the Primary Packet's authored total
simultaneous wounds per success as the default band size.

Example 5 - Vulnerability and Resistance:

An incoming Primary Packet includes Ice. The target has Vulnerability Ice 2 and
Resistance Ice 1. Net defence modifier is -1. Resolve one defence roll with that
modifier. The resulting Primary wound total is then used for any Secondary
wound-band scaling.

Example 6 - Protection:

An incoming Primary Packet includes Fire. The target has Protection Fire 2. Add
2 dice to the relevant defence roll. Protection does not subtract wounds
directly. After defence is resolved, the resulting Primary wound total is used
for Secondary wound-band scaling.

Example 7 - Multi-damage Primary with V/R/P:

Primary Packet deals 6 Fire, 6 Ice, and 6 Slashing wounds per success. Total
authored simultaneous wounds per success is 18. The target has Vulnerability
Ice 2, Resistance Fire 1, and Protection Slashing 1. Apply -2 defence modifier
from Ice Vulnerability, +1 defence modifier from Fire Resistance, and +1 defence
die from Slashing Protection. Then resolve one defence roll against the packet.
After the Primary result is calculated, default Secondary wound-band scaling
still uses one band per 18 wounds in the Primary result, rounding up.

Example 8 - no active immunity rule:

Do not say "Fire immunity reduces Fire wounds to 0" as an active rule. Damage
immunity is not currently part of the active Vulnerability / Resistance /
Protection system. Future Monster Trait immunities must define their interaction
with defence rolls, wound results, and Secondary wound-band scaling when
implemented.

Secondary Packet timing and payload source

Secondary Packets are always powered by the Primary Packet result. They never
roll independently and never create their own defended entry point. If Packet 1
does not apply an operative result, later Secondary timing does not fire.

For Field, Attached, Triggered, and other recurring Primary delivery, the first
matching Primary trigger in a round rolls once and stores that roll for the
round. Later matching Primary triggers in that same round reuse the stored roll.
Each affected Primary Target defends separately, so target-local applied
Primary results may differ.

If a Secondary Packet targets the Primary Target, use that target's own applied
Primary result. If a Secondary Packet targets Self, Allies, Field context, an
attached host's allies, or another non-primary recipient, the first successful
Primary application in that round sets the Secondary payload for that round. A
fully defended first trigger sets no payload; the next successful trigger may
set it. Once set, later same-round Primary triggers do not replace, improve,
reduce, or add to that payload unless explicitly authored.
Renderer descriptors for non-primary Secondary payloads should refer to the
power's Primary result this round rather than cumulative wounds or successes
across all same-round triggers.

For Field triggers, per-recipient area timing is target-local by default. If a
target enters, starts in, ends in, or leaves the area, the Primary affects that
target unless the packet explicitly authors area-wide delivery. If an ally
starts or ends their turn in the area, the Secondary affects that ally unless
the packet explicitly authors all-allies delivery.

Secondary Packets own their own Effect Timing and Effect Duration. They do not
inherit the parent power duration, Packet 1 duration, or chassis lifespan as
their own packet duration. An Instant Secondary Packet must not render as
lasting for 1 turn.

Do not repeat the global one-roll-per-round stored Primary result rule in every
individual descriptor. For split-timing Secondary payloads, prefer "When
[recipient] subsequently [timing], and [Power] [Primary condition] this
round..." and avoid "then when...".

Mixed-Intention Packet Resolution

Mixed-intention powers are legal through multiple Effect Packets.

If a power contains any hostile packet, Packet 1 must be hostile.

A non-hostile Packet 1 may not occupy the first slot if any later packet is hostile.

Secondary Packets remain contingent regardless of intention.

If Packet 1 fails, all contingent Secondary Packets are discarded.

If Packet 1 lands, Secondary Packets may resolve immediately or later according to their own Effect Timing, but never earlier.

Mixed intention does not create additional defence gates. Only Packet 1 is defended against.

A Secondary Packet may use a different target class or local targeting pattern only if that targeting is explicitly authored at packet level.

In the current builder-facing model, the relevant packet-local recipient classes may include Primary Targets, Allies, and Self. Allies includes self by default. A power that should exclude the user must do so through an explicit authored restriction rather than by redefining Allies as not-self.

Range, target legality, and target count are separate parts of the shared rules grammar.

Range defines reach or area. Target legality defines who or what may be selected. Target count defines how many legal targets may be selected.

Range does not by itself decide whether self is legal or illegal. Self is legal only when the power's legal target set allows self.

Self can be selected through Self range, Melee range, Ranged range, or AoE / area placement when self is legal, self is inside the relevant range or area geometry, self is selected as one of the allowed target slots, and the selected target count does not exceed the power's authored target count.

Self consumes one target slot like any other target unless the power explicitly says otherwise.

Examples:

- A Ranged power that chooses 2 legal targets may choose self plus 1 ally if self and allies are legal targets.
- That same Ranged 2-target power may not choose self plus 2 allies.
- A Melee power that chooses 1 legal target may choose self only if self is legal and the geometry permits it.
- An AoE placed somewhere that does not include self cannot affect self.
- An AoE that includes self may affect self only if self is a legal target.

Self range is the cheapest and cleanest way to author self-only effects, but Self range is not the only way self can ever be targeted.

Broader target flexibility should be priced through the normal range, target legality, target count, and area cost axes. Do not add a hidden legality layer that forbids Melee or Ranged powers from selecting self when self is otherwise legal.

Packet-local recipient classes must remain legal within the power's shared cast context unless an explicit advanced local targeting override truthfully authorizes a different recipient lane.

In the current builder surface:

Self-only powers collapse packet-local recipient selection to Self only.

Single-target Melee and single-target Ranged powers do not automatically exclude self. They may select self only when self is legal, within range / geometry, and counted as the one selected target.

These restrictions are builder-surface legality and anti-redundancy rules. They do not redefine the broader meaning of Allies, which still includes self by default where Allies is legally available.

If a legally authored packet has no legal recipient when it would resolve, that packet simply does not apply to that recipient.

Legal mixed-intention patterns include hostile-primary into hostile-secondary, hostile-primary into beneficial-secondary, and non-hostile-primary into non-hostile-secondary.

Non-hostile-primary into hostile-secondary is illegal.

Packet Timing Legality by Chassis

Immediate

Packet 1 legal timings are on cast / on use now and, where needed, on hit.

Immediate Secondary Packets may resolve at the same moment as Packet 1 or later only if Packet 1’s successful application creates an ongoing result that can carry that later timing.

If an Immediate Packet 1 resolves On Cast and creates no beyond-turn carrier, later Immediate packets remain On Cast only in first implementation.

Immediate Packet 1 may not use start of turn, end of turn, on trigger, on release, or on expiry timings as if it had a hidden persistent chassis.

In the current builder surface, Immediate + Standard Packet 1 is restricted to On Cast only unless a more specific later implementation opens an honest On Hit path.

Field

Field Packet 1 may use on create, on enter, start of turn, end of turn, and on expiry while the field exists.

Field Secondary Packets may resolve at the same moment as Packet 1 or at later legal field interaction timings while the field remains active.

Field may not use on trigger or on release as its chassis-defining Packet 1 timing.

Attached

Attached Packet 1 may use gate on attach or later attached-state payload timings such as start of turn, end of turn, movement, action, or expiry.

Attached powers must explicitly distinguish gate on attach from gate on payload for hostile delayed-entry cases.

Attached Secondary Packets may resolve at the same moment as Packet 1 or at later attached-state timings while the attachment persists.

Trigger

Trigger Packet 1 resolves on trigger.

Trigger Secondary Packets may resolve at the same moment as Packet 1 or later from the moment Packet 1 landed.

Trigger may not use on cast or on release as its Packet 1 timing.

Reserve

Reserve Packet 1 resolves on release, or on reserve expiry only if the authored release behaviour explicitly says so.

Reserve Secondary Packets may resolve at the same moment as Packet 1 release or later from the moment Packet 1 landed.

Reserve may not use on cast or on trigger as its Packet 1 timing.

Secondary timing carrier rule

Secondary Packets must obey both chassis legality and carrier legality.

If Packet 1 has Effect Duration Instant or Until the target's next turn, Secondary Packets are restricted to same-moment or expiry-style timings such as On Cast, On Trigger, or On Expiry, subject to the selected chassis.

Start of Turn, End of Turn, and other later recurring timings are only legal where Packet 1 creates a real beyond-resolution carrier such as Turns or Passive, again subject to the selected chassis.

This rule narrows legal Secondary timing choices. It does not add new timings to any chassis.

Channel First-Pass Rules

In first implementation, Channel is an action-commitment modifier only.

Do not offer alternate channel modes in first implementation.

First-pass Channel support is limited to Immediate, Field, and Attached powers.

Authored Effect Timing determines when each Effect Packet resolves while the power is being maintained. Channel does not automatically reroll Packet 1 or create repeated packet resolution by itself.

If a power is intended to apply effects repeatedly while maintained, those repeated effects must be authored through legal packet timings such as start of turn, end of turn, expiry, or other legal timings for the selected chassis.

Reapplying live stacks from the same power/source refreshes them on application rather than creating separate expiry tracks for each stack instance.

A Channelled power ends when the user stops maintaining it, can no longer maintain it, its Lifespan ends, or its sustaining anchor is broken.

Attached Range Rule

An Attached power requires legal range for initial application.

After legal initial application, host-bound persistence may continue even if later range differences arise.

If an Attached power is also Channelled, maintained reinforcement or reapplication requires the target to remain within legal cast range from the source.


16. Phase 5 Addendum — Implementation-Facing Translation of the Hardened Power Model

Purpose of this addendum

This addendum translates the hardened power model into builder-facing and schema-facing guidance for Summoning Circle without reopening the underlying architecture.

Its goal is not to make the builder exhaustive on day one.

Its goal is to make the builder structurally honest, implementation-safe, and fast for normal powers while still supporting the hardened chassis and packet rules already locked.

16.1 Locked canon

The following points are already treated as rules truth for implementation-facing work:

a power consists of Power-Level Fields, one Descriptor Chassis, and one or more Effect Packets

Shared Rules Grammar remains separate from Delivery Method

Delivery Method is implemented through Descriptor Chassis

the hardened chassis matrix is Immediate, Field, Attached, Trigger, and Reserve

Packet 1 is always the Primary Packet

only the Primary Packet sets the Defence Gate

Secondary Packets never create separate defended entry points and only resolve if the Primary Packet lands

Packet 1 lands if it applies at least one operative result unit

wound-based Secondary scaling uses Applied Success Bands = CEILING(Net Wounds / Wounds Per Success)

mixed-intention legality is constrained by hostile-primary ordering

Lifespan, Effect Timing, and Effect Duration are separate concepts

Counter is an authored Power-Level Field with No / Yes

Assist is universal by system rule

Responses are the player-facing resource

Response Window / Incarnate Stack order is locked: active effects, hostile declaration, Assist declaration, Counter declaration, Assists, Counters, trigger roll, normal active defence only if no Counter and one or more actionable hostile successes were produced, prevention/effects, result application, then cleanup

Counter use replaces the defender's normal active defence unless the Counter explicitly grants or modifies that defence

Counter-enabled powers remain normally usable; Counter: Yes adds an extra response mode, not exclusivity

Attack Counters do not cancel the triggering action unless they explicitly interrupt or cancel it

Movement interrupt Counters must strictly exceed triggering successes; ties fail

Control Counters and Control Assists can interrupt only when Control is Primary, logically capable of stopping the incoming action, and strictly exceeds triggering successes; Secondary Control is after-effect only

Preparation stacks. Panic buttons replace.

Commitment Modifier is a Power-Level Field with Standard / Channel / Charge

Channel is an action-commitment modifier only in first implementation; authored packet timing determines when effects occur

first-pass Channel support is limited to Immediate, Field, and Attached

Attached powers require legal initial range, but host-bound persistence may continue after later range changes

Channelled Attached reinforcement requires the target to remain within legal cast range from the source

Immediate + Standard Packet 1 uses On Cast only in the current builder surface

If an Immediate Packet 1 resolves On Cast and creates no beyond-turn carrier, later Immediate packets remain On Cast only in first implementation

Secondary scaling from the Primary result class is derived rather than manually authored in the current builder surface

packet-local recipient class may distinguish Primary Targets, Allies, and Self where a Secondary Packet needs a different target class from Packet 1

Allies includes self by default unless an explicit authored restriction says otherwise

range, target legality, and target count are separate: self may be selected through Self, Melee, Ranged, or AoE only when self is legal, inside the relevant geometry, and counted against the authored target count

packet-local recipient class must remain legal within the shared cast context unless an explicit advanced local targeting override authorizes a different recipient lane

Persistent states use stacks and a stack-cleanup model, while repeating direct effects use ongoing value plus duration

Physical ongoing-damage cleanup defaults to Fortitude Resist and mental ongoing-damage cleanup defaults to Bravery Resist

Duration Dodge, Block, and Resist powers create finite Dodge Pool, Block Pool, and Resist Pool resources; they roll once on apply and spend down per matching trigger unless explicitly authored to refresh or reroll

Repeated defensive pool applications from the same source power, packet, and protected target use refresh/replace higher-of-current-or-new by default; additive/recharging pools require explicit special authoring

Ambiguous Cleanse targeting of Active Power or Channelled Power may use a narrow Cleanse Theme disambiguator to resolve one exact cleanup Resist Attribute

Trigger powers may distinguish Trigger Method as Arm and then target versus Target and then arm while still resolving Packet 1 only on trigger

Secondary Packet timing is further narrowed by Packet 1 carrier legality; Instant or Until the target's next turn do not support later recurring Secondary timings

Immediate remains the only chassis that can honestly have no Lifespan requirement in the current builder surface; Field, Attached, Trigger, and Reserve require Turns or Passive and default to Turns

Cooldown begins only after the power has completely resolved, not merely when it is cast, armed, attached, placed, held, or released

Immediate powers enter cooldown after their immediate resolution sequence is complete and any continuing Effect Duration is noted separately

Non-Immediate powers with Lifespan Turns X treat cast / execution / release as Turn 0 and enter cooldown only after the final resolution window of the last lifespan turn closes

Passive Lifespan powers enter cooldown only when voluntarily ended by the caster or when Cleanse / Resist reduces currently active successes to 0

Counters and Assists use the same cooldown lifecycle, treating the executing Response as Turn 0

Charge is front-loaded; lifespan tracking begins only when the charged power is released, cast, or otherwise executed

Self-origin sphere wording must distinguish static areas centered on the user's current space from Attached areas attached to the user

Overlapping areas from the same packet at the same timing multiply hostile result before one defence roll rather than creating extra defended entry points

Dodge succeeds on match or exceed, not exceed-only

natural 1s are hard failures, natural 2s and 3s are recoverable failures, and only natural successes can produce greater / critical / higher-tier spike results

simple powers must stay simple

live preview is mandatory

Application Mode is hidden by default and should surface only when inference fails

hostile delayed-entry Attached powers must explicitly distinguish gate on attach from gate on payload

16.2 Implementation recommendation

The following are strong implementation-facing recommendations, but are not yet falsely promoted into source-level canon:

shared cast context should be authored from Power-Level Fields

structural state should be represented through chassis-local fields

packet outcomes should be represented through packet-local fields

resolved gates, legality warnings, and preview wording should be treated as derived metadata rather than co-equal authored schema truth

the builder should expose only the minimum chassis-local controls required by the selected chassis

local origin and local targeting overrides should remain advanced-only

for UI clarity, Lifespan may still be rendered near top-level controls even if eventual schema ownership lands chassis-local

16.3 Still unresolved / needs hard rule

The following implementation-facing questions remain open and should not be falsely written as settled canon:

broader Trigger + Charge + Delayed Cast canon where the current first-pass implementation has not fully hardened it

the final schema ownership boundary for Lifespan

the exact schema boundary between Power-Level Fields, chassis-local fields, packet-local fields, and derived metadata

longer-term timing taxonomy beyond the current implemented preview surface

the exact cost integration model for chassis complexity plus Commitment Modifier

16.8 Design law for implementation translation

Do not let authored data, inferred rules, and preview prose masquerade as the same thing.

Authored data defines what the power is.

Resolver logic determines what rules follow from that authored data.

Preview text reports those results clearly.

If those layers are blurred, the builder will lie.

That is unacceptable.


Design Goal

This architecture exists to preserve the speed and consistency of descriptor-driven authored content while allowing powers to feel mechanically richer, cleaner, and more distinct than the legacy one-template approach.

The model must support expressive fantasy without letting authored powers dissolve into mush.
