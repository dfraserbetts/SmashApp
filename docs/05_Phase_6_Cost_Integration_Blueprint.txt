05_Phase_6_Cost_Integration_Blueprint.txt

Purpose

This document records the implementation-facing blueprint for Phase 6 cost integration.

It is not a design-theory document.

It exists to define:

- what Phase 6 is trying to accomplish
- how the cost model should be structured
- which parts now already exist in code
- which parts are still unfinished
- which boundaries future chats must preserve when continuing the work


Phase 6 Goal

Phase 6 exists to price the hardened shared power system honestly.

That means the cost model must reflect:

- shared cast context burden
- structural chassis burden
- access / action-commitment burden
- packet identity
- packet magnitude
- packet timing
- packet duration
- packet recipient
- specific mechanic burden
- packet count complexity
- cross-packet leverage

The cost model must support both:
- future Character Builder scalar spend
- Summoning Circle monster-side derived axis contribution

There must still be one underlying cost truth.


What Phase 6 Already Has

The current implementation already includes:

1. Canonical power-cost resolution
- a live Phase 6 power-cost resolver exists
- packet-level and total power breakdowns are inspectable
- canonical power contribution now feeds the monster-side calculator through a cleaned integration path

2. Versioned Power Tuning
- active / draft / archived sets
- grouped admin UI
- friendly labels / descriptions
- searchable current keys
- comparison and reset workflow

3. Versioned Outcome Normalization
- active / draft / archived sets
- grouped admin UI
- current radar-curve / display tuning exposed separately from power truth

4. Versioned Combat Tuning / Monster Baseline Tuning
- active / draft / archived sets
- grouped admin UI
- non-power monster baseline contributors exposed separately from power truth and radar normalization

5. Canary Harness
- shared approved canary catalog
- shared approved scenario catalog
- baseline-versus-candidate comparison for:
  - Power Tuning
  - Outcome Normalization
  - Combat Tuning
- shared stripped Summoning Circle baseline so live editor and harness now align

6. Cleaned radar pipeline
- canonical power contribution is derived first
- non-power monster contribution is added second
- outcome normalization is applied last
- legacy independent power heuristics have been removed or bypassed


What Phase 6 Must Still Preserve

Even though major tooling now exists, Phase 6 must still preserve the following hard rules:

- formula shape lives in code
- tunable numeric values live in admin-editable config
- one canonical underlying power-cost truth
- no duplicate independent balance systems
- cooldown is derived after BasePowerValue
- Cooldown Reduction is not part of the base formula and remains future work
- simple powers must stay simple
- not every cost node automatically becomes a visible radar lane


Canonical Output Split

Character Builder side:
- shows a total Power Point pool
- consumes a scalar spend value per power
- supports players allocating that pool across few or many powers

Summoning Circle side:
- consumes a derived axis contribution vector from powers
- combines that vector with non-power monster contribution
- then applies outcome normalization for radar display

This means:
- one shared costing truth
- multiple downstream consumption layers
- no duplicate independent tuning system


Current Phase 6 Layer Split

The current implementation now has three distinct tuning layers.

Power Tuning
- canonical power truth

Combat Tuning / Monster Baseline Tuning
- non-power monster baseline contribution

Outcome Normalization
- downstream radar interpretation and display

Future chats must not collapse these into one blurred tuning page or one blurred mental model.


Cooldown Rule For Phase 6

Cooldown is not a cost input.

Cooldown is a derived output from the base power value.

The intended cost sequence remains:
- calculate BasePowerValue
- derive cooldown band / cooldown value from BasePowerValue
- optionally allow additional spend to reduce that cooldown where legal

Cooldown Reduction is a separate paid modifier.

Cooldown Reduction is not part of the base authored power value and should be layered on top as additional spend later.

Cooldown value and cooldown lifecycle are separate concepts.

Cooldown value answers:
- what number is placed on the cooldown die once cooldown begins

Cooldown lifecycle answers:
- when the cooldown die is placed
- when it is reduced
- when the power becomes ready again

Cooldown does not begin merely because a power is cast, armed, attached, placed, held, or released.

Cooldown begins only when the power has completely resolved.

Immediate chassis powers completely resolve once application successes have been rolled, required defence / protection / dodge / resist resolution has occurred, results have been compared, unblocked results have been applied, and any continuing duration tracking has been noted separately.

Non-Immediate powers with Lifespan Turns X treat the cast / execution / release turn as Turn 0 and remain active through X subsequent caster-turn rounds.

Those powers enter cooldown only after the final resolution window of the last lifespan turn has closed.

The final resolution window closes once the last legal target has finished the relevant turn, trigger, entry, expiry, or interaction that could be affected by the power during that final lifespan turn.

Passive Lifespan powers do not enter cooldown because time passes. They enter cooldown only when voluntarily ended by the caster as a Response or Power Action, or when Cleanse / Resist reduces currently active successes to 0.

Counters and Assists use the same lifecycle and treat the executing Response as Turn 0.

Charge is front-loaded. Charge turns occur before release or cast. Lifespan tracking begins only when the charged power is released, cast, or otherwise executed.

Once a cooldown die is placed, reduce it at the end of each subsequent turn the owning character takes. If the die is showing 1 when it would be reduced, remove it instead. Once removed, the power is ready on the owner's next turn or legal Response window, subject to the power's normal permissions.

Combat Lab, runtime, and encounter tooling must consume the derived cooldown value but must not treat cooldown as starting on cast unless the power also fully resolves on cast.


Canonical Cost Formula Shape

The base formula shape for Phase 6 is:

BasePowerValue =
  SharedContextFormula
  + StructuralFormula
  + AccessFormula
  + CombinedEffectPacketFormula

Cooldown is derived outside the base formula.

If cooldown reduction is purchased in later work, then:

FinalPowerSpendValue = BasePowerValue + CooldownReductionFormula


SharedContextFormula

SharedContextFormula prices the shared cast context that applies across the whole power.

It contains:
- Range category
- target count choices
- ranged distance
- AoE cast range
- AoE count
- AoE shape
- AoE dimensions

SharedContextFormula should be additive.

It is a real cost burden, but it should not be allowed to dominate packet identity.

Targeting flexibility cost guidance

Self-only targeting is the cheapest and cleanest targeting profile.

A power that can affect self and another ally / target is more flexible than a self-only power and should pay for that flexibility through target legality / recipient flexibility where the current architecture supports it.

A power that can choose multiple targets is more flexible than a single-target power.

Target count remains meaningful because self consumes one selected target slot unless a power explicitly says otherwise.

Broader range, area, or flexible target profiles should be valued by the cost model rather than blocked by an extra authoring rule that Melee or Ranged powers cannot target self.


StructuralFormula

StructuralFormula prices structural burden created by the chassis and its presence pattern.

It contains:
- chassis base cost
- structural presence / lifespan burden
- trigger / release structure burden where relevant
- delayed-payload / carrier burden where relevant

StructuralFormula should be mostly additive.

This is where premium non-immediate leverage must be priced honestly.


AccessFormula

AccessFormula prices use access and action-commitment burden.

It contains:
- Counter access burden
- Commitment Modifier burden
- charge-turn burden where relevant

AccessFormula should be mostly additive.

Build Power bonus output may also influence packet magnitude through a local packet-level scaling step rather than only this top-level access formula.


Counter And Response Cost Guidance

Counter: Yes is an added use permission, not an exclusivity tradeoff.

Do not discount a power because Counter-enabled use replaces normal active defence; the power remains normally usable as a Power Action.

Attack Counters carry a high premium because they create off-turn damage or effects and the triggering attacker does not receive an active defence roll against them. Already-active passive/static protection still applies.

Defence Counters are priced as specialist active-response alternatives, not as free stacks on top of Dodge, Physical Defence, Mental Defence, or Resist.

Movement and Control interrupt Counters need a high premium because a strict timing win can avoid, cancel, or deny an incoming action. First-pass timing requires strictly more successes than the trigger; ties fail.

Persistent or passive setup remains valuable because it is established before the response window and can stack with later response choices.

Preparation stacks. Panic buttons replace.

Counter, Movement Counter, and Control interrupt Counter costs should be revisited after Combat Lab and runtime behaviour faithfully implement the response-stack rules.


CombinedEffectPacketFormula

CombinedEffectPacketFormula is the heart of the power cost model.

It contains:
- PrimaryPacketFormula
- SumOfSecondaryPacketFormulae
- PacketCountComplexityFormula
- CrossPacketSynergyFormula


PrimaryPacketFormula

The Primary Packet is the main defended entry point and should be priced as the full leading effect.

PrimaryPacketFormula contains:
- EffectFamilyBaseFormula
- MagnitudeFormula
- PacketTimingFormula
- PacketDurationFormula
- RecipientFormula
- SpecificMechanicFormula
- PersistenceCleanupFormula
- ScalingFormula where relevant

MagnitudeFormula uses explicit Power Tuning keys for each supported discrete authoring value. Dice count and potency are discrete magnitude controls, so every legal authored value must be represented by a key such as packet.magnitude.dice.N or packet.magnitude.potency.N. Authoring surfaces must not expose unpriced dice/potency values, because bucket fallback is intended for continuous/range-style tuning lookups, not for flattening legal packet magnitude.

Required packet fields are validity gates, not alternate pricing branches. For example, Attack packets without a selected damage type are incomplete and must not produce a valid monster threat/radar output or player spend comparison.

For a fixed legal power shape, increasing supported dice count or potency must not reduce BasePowerValue, the canonical power-axis contribution, or the final normalized radar output unless the output is already clamped at its documented cap. Player level-sensitive spend scaling remains deferred until canonical cost and radar monotonicity are stable.


SecondaryPacketFormula

Secondary Packets are real value but contingent.

Each SecondaryPacketFormula should therefore be calculated as its own packet value and then reduced by a contingency factor rather than treated as either free or fully independent.

SecondaryPacketFormula_i =
  (PacketCoreFormula_i + PacketModifierFormulae_i)
  × SecondaryContingencyMultiplier_i


PacketCountComplexityFormula

This formula prices the burden of having multiple packets regardless of what those packets do.

It exists because:
- more packets increase exploit surface
- more packets increase complexity
- more packets increase leverage density

This is cost-only by default.

It should not emit axis by itself.


CrossPacketSynergyFormula

This formula prices additional leverage that emerges from packet combinations rather than from isolated packet values.

Examples include:
- hostile-primary into beneficial rider
- delayed-payload leverage
- recurring leverage created by a real carrier
- result-scaling follow-through
- other packet combinations that are stronger together than as isolated dropdown values

This is additive by default.

Axis emission from this node is conditional and must follow actual battlefield shape rather than defaulting blindly into Synergy.


Cost-Only / Modifier-Only / Axis-Carrying Split

The Phase 6 model now explicitly distinguishes three types of cost node:

Cost-Only
- adds scalar cost
- emits no axis directly

Modifier-Only
- modifies other value expressions
- may add scalar cost
- should not emit large raw axis value by itself

Axis-Carrying
- adds scalar cost
- may also emit derived axis contribution where honest

This matters because not every expensive thing should automatically become a visible radar spike.

Examples:
- Packet Set Complexity is cost-only
- Shared Cast Context is modifier-only
- Structural Presence may emit modest Pressure / Presence
- Cross-Packet leverage may emit axis only where the combination truly changes battlefield shape


Current Translation Order

The current intended order is:

1. derive canonical cost breakdown
2. derive canonical per-use power axis contribution
3. add non-power monster contribution
4. apply outcome normalization
5. display the final monster radar

Later work may still add:
- cadence / availability translation improvements
- monster-side realization modifier layer
- other missing downstream translation seams

But those belong after canonical power truth, not inside it.


Current Open Phase 6 Questions

The following are still real open questions inside Phase 6:

- which current costs should remain cost-only versus later translated into cadence, structural pressure, or realization
- how the later monster-side realization modifier layer should gate typed contributor output through monster attributes
- which current downstream translation steps still under-report or over-report real battlefield strength
- whether additional debug or comparison surfaces are still needed for faster tuning iteration

These are now implementation-facing tuning questions, not permission to reopen the whole formula architecture.


Immediate Practical Phase 6 Direction

The immediate Phase 6 work should stay focused on:

- tuning specific powers against the aligned live/harness baseline
- tuning non-power monster baseline contribution where stripped monsters are too weak or too strong
- tuning outcome normalization where radar display saturates too early or too late
- preserving clean separation between:
  - Power Tuning
  - Combat Tuning
  - Outcome Normalization
- beginning the monster-side realization modifier layer only when current tuning layers are stable enough to justify it
