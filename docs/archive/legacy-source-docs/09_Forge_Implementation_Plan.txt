Forge Implementation Plan
=========================

Purpose
-------

This document plans the first Forge implementation tranche after Natural
Baseline v1. It translates the Natural Baseline output-band work into a
minimal, ordered Forge plan without starting Forge code changes.

This plan is based on the current repo state and the committed source docs:

- docs/07_Natural_Baseline_Output_Bands.txt
- docs/08_Forge_Output_Bands_Blueprint.txt
- docs/05_Phase_6_Cost_Integration_Blueprint.txt
- docs/03_Current_Phase_And_Next_Steps.txt
- docs/04_Implementation_Handoffs.txt

Core constraints:

- Output first, origin second.
- Item level drives core throughput.
- Rarity mostly buys breadth, geometry, alternate profiles, riders,
  versatility, and premium features.
- Strength x 2 is the table-facing wounds per success rule.
- Total wounds per success = Strength x 2 x damage type count.
- Damage type count consumes output budget.
- Shields split budget between attack and defence.
- Armour slot weighting is deferred.
- Power Tuning, Combat Tuning / Monster Baseline Tuning, and Outcome
  Normalization remain separate layers.
- Summoned Item, Summoned Creature, Transformation, and other deferred
  builders are not part of this plan.

Current Item Model Summary
--------------------------

The existing `ItemTemplate` model already supports a broad Forge item shape.

### Common item fields

- `level`
- `rarity`
- `type`
- `size`
- descriptive fields
- campaign ownership
- tags
- custom item attributes
- global attribute modifiers

### Weapon attack profile fields

The model supports separate melee, ranged, and AoE attack profiles:

- melee:
  - `meleeTargets`
  - `meleePhysicalStrength`
  - `meleeMentalStrength`
  - melee damage type join rows
  - melee greater-success attack effects
- ranged:
  - `rangedTargets`
  - `rangedDistanceFeet`
  - `rangedPhysicalStrength`
  - `rangedMentalStrength`
  - ranged damage type join rows
  - ranged greater-success attack effects
- AoE:
  - `aoeCenterRangeFeet`
  - `aoeCount`
  - `aoeShape`
  - shape-specific dimensions
  - `aoePhysicalStrength`
  - `aoeMentalStrength`
  - AoE damage type join rows
  - AoE greater-success attack effects

Legacy/global strength fields also exist:

- `physicalStrength`
- `mentalStrength`

The newer per-profile strength fields are the clearer output-profile source
for Forge output-band work.

### Armour and shield fields

The current model supports:

- `armorLocation`
- `ppv`
- `mpv`
- `auraPhysical`
- `auraMental`
- armour attributes
- defence effects
- shield attack flag:
  - `shieldHasAttack`
- shield attributes
- VRP entries

The current Forge item model stores PPV/MPV directly, but it does not expose a
standalone item output profile for Physical Defence String Value or Mental
Defence String Value. Those values are currently better understood when an item
is evaluated as part of a monster/character output path.

### Current budget/spend fields

The repo currently calculates Forge Points rather than output-band profile
levels:

- `calculateTotalFp` combines item level, rarity scalar, and item multiplier.
- `calculateItemMultiplier` handles item type, weapon size, armour location,
  item location, and consumable settings.
- `calculateForgeTotals` reports:
  - total FP
  - spent FP
  - remaining FP
  - percent spent
  - multiplier

There is no current stored or calculated field for:

- total wounds per success by profile
- total pressure by target count / AoE
- damage type budget classification
- effective item profile level
- over-band readout
- Core Functionality vs Features & Versatility lane split


Current Forge / Output Calculation Path
---------------------------------------

### What already exists

The current Forge UI prices authored selections through Forge Point costs:

- Strength has cost rows in `ForgeCosts.csv`.
- Range category, target count, range distance, AoE geometry, and damage type
  count have cost categories.
- Greater-success effects and defensive effects have cost categories.
- PPV/MPV are priced as Stat category rows.
- Attributes and VRP options have cost/readout paths.
- Rarity and item size/location influence the total budget through config
  multipliers/scalars.

The current descriptor output path already renders table-facing damage using
Strength x 2:

- `lib/descriptors/descriptorEngine.ts`
  - `toDamageAmount = (strength) => strength * 2`
- `lib/descriptors/renderers/forgeRenderer.ts`
  - consumes the descriptor-engine damage output and labels the conversion as
    `FORGE_STRENGTH_TO_WOUNDS_V1`.

### What does not yet exist

The current Forge path does not yet calculate or display the output-band data
defined by docs/07 and docs/08:

- weapon total wounds per success
- damage type count multiplier readout
- single-target vs multi-target pressure readout
- AoE / geometry output pressure readout
- PPV/MPV package context
- defence string output or package contribution estimate
- shield attack/defence tradeoff readout
- Core Functionality vs Features & Versatility lane split
- effective profile level
- over-band / under-band status

The current Forge cost model may still be useful as an authoring budget, but it
is not the same thing as an output-first table-facing profile check.

Implementation Sequence
-----------------------

### Step 1: Add a read-only Forge item output profile helper

Likely files:

- new `lib/forge/outputProfile.ts`
- optional new `scripts/forgeOutputProfile.smoke.ts`

Purpose:

- Normalize existing Forge item data into table-facing output descriptors.
- Calculate output facts without changing saved item data or FP costs.
- Establish the output bottleneck that later UI and diagnostics can share.

Initial helper should report:

- item level
- rarity
- item type
- per attack profile:
  - profile kind: melee, ranged, AoE
  - physical authored Strength
  - mental authored Strength
  - table-facing wounds per success by lane
  - damage type count
  - total wounds per success
  - target count
  - range category
  - AoE shape / dimensions where relevant
  - greater-success rider count/labels
- defensive profile:
  - PPV
  - MPV
  - aura values
  - defensive effects
  - shieldHasAttack
- debug source:
  - values are table-facing calculations from Forge item data
  - no balance tuning applied

Non-goals:

- no schema migration
- no UI readout yet
- no FP cost rewrite
- no slot weighting
- no effective profile level yet

Validation:

- `npx tsc --noEmit`
- `npx --yes tsx scripts/forgeOutputProfile.smoke.ts` if the smoke script is
  added

Rollback risk:

- low; helper-only and report-only.

Prisma migration:

- no.

Recommended First Patch
-----------------------

The smallest useful first patch is:

1. Add `lib/forge/outputProfile.ts`.
2. Add a focused report-only smoke script, for example
   `scripts/forgeOutputProfile.smoke.ts`.
3. Do not change Forge save behaviour, schema, FP costs, or UI.

The helper should calculate table-facing output from current Forge item-shaped
data:

- Strength x 2 wounds per success
- damage type count
- total wounds per success
- target count and range/AoE facts
- PPV/MPV defensive facts
- shield attack/defence co-presence

This first patch creates the output bottleneck that docs/08 requires, while
leaving the existing Forge cost economy untouched.


Validation Strategy
-------------------

For helper-only work:

- `npx tsc --noEmit`
- `npx --yes tsx scripts/forgeOutputProfile.smoke.ts`

For any later UI work:

- `npx tsc --noEmit`
- existing Forge route/component smoke by manual browser check if needed

For equipped monster validation:

- `npm run calibration:export-active-tuning`
- `npx --yes tsx scripts/outcomeCalibrationHarness.ts --tuning active`
- `npm run diagnostic:level-relative-radar`
- `npm run diagnostic:level-relative-decay`
- `npm run diagnostic:output-origin-parity`
- `npm run diagnostic:combat-smoke` only when the fixture adapter honestly
  supports the tested monsters.

- No gameplay code changes.

- No Power Tuning, Combat Tuning, or Outcome Normalization changes.
