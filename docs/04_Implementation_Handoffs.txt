- ongoing damage uses ongoing value plus duration rather than stacks

- ongoing-damage cleanup removes source damage units per success, where each unit is the source effect's wounds-per-success / wounds-per-band value
- packet-local recipient class may distinguish Primary Targets, Allies, and Self where a Secondary Packet needs a different target class from Packet 1
- Allies includes self by default unless an explicit authored restriction says otherwise
- packet-local recipient class must remain legal within the shared cast context
- range, target legality, and target count are separate; do not reintroduce a hidden rule that Melee or Ranged powers cannot target self
- self can be selected through any range mode only when self is legal, within range / area geometry, and counted against the authored target count
- AoE affects self only when the placed area includes self and self is a legal target
- ambiguous Cleanse targeting of Active Power or Channelled Power may use Cleanse Theme only where a unique cleanup Resist Attribute is not otherwise already settled
- Trigger powers may use Trigger Method to distinguish Arm and then target from Target and then arm while keeping Packet 1 on trigger
- Secondary timing choices must obey both chassis legality and Packet 1 carrier legality
- Immediate remains the only chassis that can honestly have no Lifespan requirement in the current builder surface; Field, Attached, Trigger, and Reserve require Turns or Passive and default to Turns
- cooldown begins only after a power has completely resolved
- Immediate powers enter cooldown after their immediate resolution sequence completes and any continuing duration tracking has been noted
- non-Immediate Lifespan Turns powers treat cast / execution / release as Turn 0 and enter cooldown only after the final resolution window of the last lifespan turn closes
- Passive Lifespan powers enter cooldown only when voluntarily ended by the caster or when Cleanse / Resist reduces currently active successes to 0
- Counters and Assists use the same cooldown lifecycle, treating the executing Response as Turn 0
- Charge is front-loaded and lifespan tracking begins only when the charged power is released, cast, or otherwise executed
- self-origin sphere wording should distinguish static areas centered on the user's current space from self-hosted Attached areas attached to the user
- overlapping areas from the same packet at the same timing use one power roll, pre-defence multiplication by overlap count, and one defence roll
- canonical attribute naming is Guard and Synergy
- Dodge succeeds on match or exceed
- future chats must not reintroduce exceed-only dodge wording
- duration Dodge, Block, and Resist powers create finite Dodge Pool, Block Pool, and Resist Pool resources; they roll once on apply and spend down per matching trigger unless explicitly authored to refresh or reroll

Response Window Continuity Rule

All runtime-facing systems must use one response-stack model:

1. Apply already-active effects.
2. Declare the hostile action and targets.
3. Declare legal Assists.
4. Declare legal Counters.
5. Roll and apply Assists.
6. Roll and apply Counters.
7. Roll the triggering hostile action.
8. If no Counter was declared, roll the chosen normal active defence.
9. Calculate prevention and effects.
10. Apply the incoming result and the Counter result.
11. Process defeats and cleanup.

Combat Lab, Manual Combat Lab, Encounter Confidence Panel, printable runtime guidance, and future digital runtime tooling must not invent separate response rules.

Do not reintroduce Counter plus normal active defence stacking.

Any Counter replaces Dodge, Physical Defence, Mental Defence, or Resist for that trigger unless the Counter explicitly grants or modifies that defence.

Already-active passive effects, static protection, buffs, debuffs, and prepared states still apply because they were established before the trigger.

Preparation stacks. Panic buttons replace.

Movement and Control interrupt Counters require strictly more successes than the triggering action. Ties fail.

Attack Counter damage does not cancel the triggering action unless the Counter explicitly interrupts or cancels.

Counter-enabled powers remain normally usable; Counter: Yes adds an extra response mode, not exclusivity.


Duration Defence Pool Runtime Rule

Runtime-facing tools must represent duration Dodge, Block, and Resist powers as finite pools.

Use these terms consistently:

- Dodge Pool
- Physical Block Pool
- Mental Block Pool
- Resist Pool

Do not model duration defensive powers as repeated full-value rerolls on every incoming attack unless the authored power explicitly says it refreshes or rerolls on each trigger.

On application:

- roll the power once
- calculate poolRemaining = successful power-roll successes x potency
- set perTriggerCap = potency unless explicitly modified
- store the matching lane:
  - Dodge Pool: physical Dodge-legal triggers only
  - Physical Block Pool: physical wound prevention
  - Mental Block Pool: mental wound prevention
  - Resist Pool: one authored resisted Attribute lane per packet
- store sourcePowerId / sourcePowerName / sourcePacketId / sourceActorId where available
- store owner / source actor and protected actor separately where those differ
- store whether the pool is self-only, ally-targeted, or area-applied
- store duration / expiry / cleanup ownership separately from the pool value

On a matching incoming trigger:

1. Resolve incoming declaration and roll first.
2. Let the defender choose the matching legal active defence or Resist path.
3. Before the normal defence / Resist roll, commit pool points up to min(poolRemaining, perTriggerCap).
4. Roll the normal defence / Resist.
5. Add committed points to that result.
6. Spend committed points even if the defence fails.
7. Apply normal defence degradation normally.

Combat Lab and future runtime diagnostics should expose at least:

- pool created
- pool remaining
- points committed per trigger
- points spent per trigger
- per-trigger cap
- prevention or cancellation achieved
- points wasted on failed defence where relevant
- expiry or cleanup reason
- resisted Attribute for Resist Pools
- wound channel for Block Pools
- whether a Counter was one-shot or created a persistent duration pool

Counter Dodge remains one-shot unless it also explicitly creates a duration Dodge Pool. Movement is not a Defence mode and must not be used as a synonym for Dodge Pool.

- source power id/name
- source actor
- protected actor
- target scope
- source chassis: Immediate / Trigger / Reserve / Field / Attached / Channel / Charge
- source packet id/index
- remaining pool points
- generated pool points
- potency-derived per-trigger commit cap
- final charged potency/dice if applicable
- duration/lifespan expiry
- Field-bound availability
- Attached dependency
- Channel dependency
- resistedAttribute for Resist pools
- wound channel for Block pools
- pool points committed per incoming action
- pool points spent
- successful Dodge Pool avoids
- Block Pool wounds prevented
- Resist Pool units cancelled
- points wasted on failed defences
- refresh/replace events
- additive/recharging events if explicitly authored
- expiry due to empty pool, duration end, field exit, attachment end, channel end, cleanse/removal, or defeat cleanup

Combat Lab diagnostics should eventually expose:

- pools created
- pool points generated
- pool points committed
- pool points wasted
- pool points remaining at expiry
- refresh/replace events
- field-bound expiries
- channel-ended expiries
- charged potency/dice used at release
- Dodge avoids
- Block wounds prevented
- Resist units cancelled
- per-action pool contribution

Implementation defaults:

- Trigger and Reserve create the pool when their packet is delivered/released.
- Field creates target-specific pools and refresh/replaces repeated applications from the same source power, packet, and protected target.
- Attached pools follow the attachment and end with it unless explicitly lingering.
- Channelled pools refresh/replace while maintained and end with the channel unless explicitly lingering.
- Charged pools use final release dice/potency; charged potency affects both pool size and per-trigger commit cap.
- A defender commits from only one matching defensive pool per incoming action unless explicit combined spending is authored.
- Additive/recharging pools require explicit authoring and should not be inferred.


Cooldown Lifecycle Rule For Future Work

All future implementation work must preserve the distinction between cooldown value and cooldown lifecycle.

Cooldown value is the number placed on the cooldown die.

Cooldown lifecycle determines when that die is placed and when it is reduced.

Cooldown does not begin when a power is merely cast, armed, attached, placed, held, or released.

Cooldown begins only after the power completely resolves.

Immediate powers completely resolve after the application roll, any defence / protection / dodge / resist resolution, result comparison, unblocked result application, and any required ongoing duration notation.

Non-Immediate powers with Lifespan Turns X treat the cast / execution / release turn as Turn 0. The power remains active through X subsequent caster-turn rounds and enters cooldown only after the final resolution window of the last lifespan turn has closed.

The final resolution window closes once the last legal target has finished the relevant turn, trigger, entry, expiry, or interaction that could be affected by the power during that final lifespan turn.

Passive Lifespan powers do not enter cooldown from time passing. They enter cooldown only when voluntarily ended by the caster as a Response or Power Action, or when Cleanse / Resist reduces currently active successes to 0.

Counters and Assists use the same lifecycle and treat the executing Response as Turn 0.

Charge is front-loaded. Charge turns happen before release or cast. Lifespan tracking begins only when the charged power is actually released, cast, or otherwise executed.

Combat simulation, printable power output, runtime tracking, and future encounter tooling must not treat cooldown as starting on cast unless the power also fully resolves on cast.

- git status / worktree cleanliness

- run count
- seed or seed policy
- scenario names
- actor and monster names

- active Outcome Normalization name
