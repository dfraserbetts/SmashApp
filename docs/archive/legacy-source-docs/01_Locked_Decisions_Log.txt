Purpose


This document records decisions that are considered locked unless explicitly reopened. Its purpose is to prevent drift, repeated debate, and accidental contradiction across future chats.


A locked decision should be treated as binding design truth unless a direct contradiction, severe playtest failure, or clearly superior lower-drift alternative forces a reopen.


Locked Decisions


1. The shared power core uses one consistent three-layer model


A power is built from:


Power-Level Fields


one Descriptor Chassis


one or more Effect Packets


This is the shared conceptual model for the power system.


2. Delivery Method is not a replacement for shared rules grammar


Delivery Method is a structural identity layer. It does not replace Range, Targeting, Dice, Counter, or other common rules grammar. Responses remain the player-facing resource and are not the authored power field. Its job is to define the privileged structure through which the power exists and resolves.


3. Delivery Method will be implemented through descriptor chassis


Each Delivery Method must justify itself by changing descriptor structure, field requirements, and/or mechanical resolution pattern. If it does not do that, it is not earning its place.


4. Packet 1 is always the Primary Packet


The first packet is always the Primary Packet. Packet order is mechanically meaningful, not just presentational.


5. Only the Primary Packet is defended against


A power has one defended entry point. Only Packet 1 sets the defence gate.


6. Secondary packets only land if the Primary Packet lands


If the Primary Packet is fully defended or otherwise applies no operative result, Secondary Packets do not resolve. This is a global simplification rule for speed, clarity, and cleaner authored powers.


7. Secondary packets never create separate defended entry points


Secondary Packets are contingent follow-through effects. They do not set their own separate defence rolls or independent defence gates.


8. Defence selection is gated, not fully freeform


The builder must not offer nonsense defence combinations. Defence options are filtered based on the Primary Packet’s nature. Physical attack effects should not casually fish for irrelevant weak defences.


9. Some packets will need an application-mode concept


For effects such as Control or Debuff, Intention plus Specific is not always enough to determine a sensible defence gate. A restrained application-mode layer will likely be used where required.


10. Lifespan is distinct from Effect Duration


Lifespan refers to how long a chassis-level structure remains active, armed, attached, deployed, sustained, or available to resolve. It is not the same thing as Effect Duration.


11. Lifespan, Effect Timing, and Effect Duration are separate concepts


The system must distinguish between:


how long the chassis exists


when an effect packet applies


how long the result remains after application


These concepts are not interchangeable and must not be collapsed into one fuzzy timing field.


12. Delivery Method must be selected early in the workflow


Delivery Method cannot be left until the end because it shapes required fields, descriptor structure, and the broad resolution pattern.


13. Simple powers should remain simple


The redesign must not force all powers into a bloated advanced workflow. Straightforward powers should remain fast to build and easy to understand.


14. Complexity increases cost


More intentions, longer structural presence, richer behaviour, and stronger action commitment should all increase cost in a rising fashion. Complexity is not free.


15. Trigger powers can use a soft-commit model


If a triggered power is prepared and never fires, it may return to ready state rather than being fully wasted, because the user already paid a tempo cost by spending the Power Action. However, higher-complexity or broader powers may later use stricter cooldown handling.


16. The shared power core will be implemented in Summoning Circle first


Summoning Circle is the first proving ground for the shared power core, but not as a monster-only dead end. The implementation must remain reusable for the Character Builder.


17. Standard monsters do not get the full player Spark / Restriction layer


Standard monsters will use the shared power architecture but not the full player-facing Spark / Restriction system. Legendary monsters may later use a curated variant.


18. Live preview is mandatory


The power builder must support real-time preview rendering. This is foundational, not optional polish.


19. Future runtime support should mirror the same power structure


Printable effect slips and later digital runtime tracking should be based on the same underlying power model rather than inventing a second unrelated state language.


20. The hardened Descriptor Chassis matrix currently uses five core chassis


The current practical chassis matrix is:


Immediate


Field


Attached


Trigger


Reserve


These survive because they materially change descriptor structure, required supporting fields, or broad resolution pattern.


21. Counter is not a separate chassis


Counter is not retained as its own Descriptor Chassis.


Counter is a shared Power-Level Field permission that determines whether a power may be used as a Counter when the user spends a Response on self-defence.


22. Assist is universal and not authored per power


Assist is assumed to be universally available by system rule.


It is not a per-power authored field.


Any power may be used as an Assist where the core response rules allow it.


23. Responses are the player-facing resource, not the authored power field


Responses are a player-facing resource tracked on the character sheet.


They may be spent on Assists or Counters.


The power builder does not author broad Response access per power.


Instead, it authors whether a power is Counter-enabled.


24. Counter and Trigger are separate concepts


Counter is a shared Power-Level Field permission that determines whether the power may be used as a Counter.


Trigger is a structural chassis in which a power is armed and later resolves automatically if its trigger condition is met.


They must not be collapsed into one field or treated as synonyms.


25. Counter is a Power-Level Field


The power builder should expose an always-visible Counter selector with the options:


No


Yes


No is the default unless later costing rules say otherwise.


Counter answers whether the power may be used as a Counter when the user spends a Response on self-defence.


25A. Response Window / Incarnate Stack order is locked


When a hostile action is declared, the response window resolves in this order:


1. Apply already-active effects.


2. Declare the hostile action and targets.


3. Declare legal Assists.


4. Declare legal Counters.


5. Roll and apply Assists.


6. Roll and apply Counters.


7. Roll the triggering hostile action.


8. If no Counter was declared by a defender and the hostile action produced one or more actionable hostile successes, that defender rolls the chosen normal active defence.


9. Calculate prevention and effects.


10. Apply the incoming result and the Counter result.


11. Process defeats and cleanup.


All declared actions, Assists, Counters, and Responses resolve before defeat and cleanup unless a rule explicitly interrupts or cancels the triggering action.


Attack Counter damage does not cancel the triggering action unless the Counter explicitly says it interrupts or cancels.


25A-1. Zero-success hostile actions preserve declaration timing but do not degrade normal defence

A legal hostile action that targets a defender creates the normal declaration/reaction window before the attacker rolls.

Actionable hostile successes are incoming successes or hostile units that create something the defender must actually oppose, prevent, avoid, cancel, or suffer. Examples include attack successes, wound-bearing successes, hostile effect successes, control/debuff units, ongoing-effect application successes, or resistible hostile units. If the incoming hostile action produces zero actionable hostile successes, normal defence does not roll and does not degrade.

A zero-success hostile action still counts as an attack/targeting event for declaration-based triggers. "When attacked" and "when targeted by a hostile action" effects may trigger from a zero-success hostile action.

A zero-success hostile action is not a hit and does not apply successful hostile effects. "When hit", "when damaged", "when wounded", "when affected", or "when successfully struck" effects do not trigger from a zero-success hostile action.

The defender does not roll normal Dodge, Physical Defence, Mental Defence, or active Resist against zero actionable hostile successes. The defender does not commit defensive-pool points and does not suffer resolution-based defence degradation from normal defence against that action.

Counters, Assists, and declaration-triggered responses that were declared still count as used and resolve according to their own rules unless they specifically require the incoming action to hit, wound, damage, affect, or succeed.

Design intent:

- characters react to a declared threat before knowing whether the attacker rolled well
- defence degradation represents meaningful defensive exertion against actual pressure, not merely raising a shield against an attack that whiffs before contact
- zero-success attacks can still trigger "when attacked" or "when targeted" reactions
- zero-success attacks should not tax normal defence degradation or defensive pools
- this preserves cinematic reaction timing without punishing defenders for attacks that produced no actionable pressure


25A-2. Independent defence-lane degradation is locked

Defence degradation is tracked independently by defence lane.

A defence lane degrades only when that specific lane actually resolves against one or more actionable hostile successes.

Physical Defence degradation does not affect Mental Defence.

Mental Defence degradation does not affect Physical Defence.

Dodge uses its own degradation track and does not degrade Physical Defence or Mental Defence.

Active Resist degradation applies only to the relevant resisted lane or attribute unless an explicit rule says otherwise.

Repeated pressure in one defensive channel does not automatically weaken unrelated defensive channels. Blocking or defending against several physical attacks does not make the defender worse at resisting a later mental attack. Resisting several mental attacks does not make the defender worse at blocking or defending against later physical attacks.

Design intent:

- defence degradation represents repeated pressure in the same defensive channel, not generic exhaustion that weakens every defensive response
- raising a shield, blocking weapon strikes, or enduring physical pressure should not make the defender worse at resisting psychic screams, fear, mental assault, or other mental pressure
- resisting repeated mental pressure should not make the defender worse at blocking physical attacks
- this preserves lane identity and prevents mixed-threat groups from creating abusive cross-lane defence stripping
- four physical attackers should not be able to wear down Physical Defence with sword strikes so that a weak later mental attack lands more easily through cross-lane degradation


25B. Any Counter replaces the defender's normal active defence


If a defender declares a Counter against a trigger, that Counter is the defender's active response path for that trigger.


The defender does not also roll Dodge, Physical Defence, Mental Defence, or Resist unless the Counter explicitly grants or modifies that defence.


Already-active passive effects, static protection, buffs, debuffs, and prepared states can still apply because they were established before the response window.


Preparation stacks. Panic buttons replace.


25C. Counter-enabled powers remain normally usable


Counter: Yes means the power may also be used as a Counter when the user spends a Response on self-defence.


It does not make the power Counter-only.


When used normally, the power follows normal Power Action rules and may allow the normal gates generated by its Primary Packet.


When used as a Counter, the power follows Counter response rules and replaces the defender's normal active defence against that trigger.


Counters and Assists use the normal cooldown lifecycle for the power being executed.


25D. Assists are declared before Counters and resolve first


Assists are universal by system rule, not authored per power.


Legal Assists are declared before legal Counters.


Assists roll and apply before Counters roll and apply.


An Assist may modify a Counter roll, a normal defence if no Counter is declared, or another legal response outcome.


The assisted actor still chooses one active response path: a normal active defence or a Counter.


25E. Attack Counters are legal but do not grant active defence to the attacker


An Attack Counter rolls during the Counter step before the triggering hostile action rolls.


Its damage or effect is known before final result application.


The triggering action still rolls and resolves unless the Counter explicitly interrupts or cancels it.


The triggering attacker does not get an active defence roll against the Attack Counter.


Already-active passive or static protection still applies.


25F. Movement Counters and Primary Control interrupt Responses require a strict timing win


Movement Counters can avoid or cancel a trigger only when their timing and geometry are sufficient.


First-pass Movement Counter timing is a Clash against the triggering action successes and must strictly exceed the trigger successes to avoid or cancel the trigger. Ties fail.


Control Counters and Control Assists can cancel the trigger only when Control is the Primary packet / Primary intention, the Control effect could logically stop the incoming action, and the Control result strictly exceeds the trigger successes. Ties fail.


If a Control Counter or Control Assist does not win the interrupt timing, the incoming action resolves normally and is not reduced. Secondary Control is after-effect only by default and may apply after the trigger only if powered by the Primary result and its timing / legality conditions are met.


25G. Dodge and Resist defensive power options are needed


The shared power builder needs explicit defence power options for Block, Dodge, and Resist.


These options must be shared by Summoning Circle and Character Builder rather than forked into one surface.


Dodge and Resist Counter powers do not stack with normal Dodge or Resist. They are chosen active response paths. Resist is authored against one core Attribute lane and remains distinct from Cleanse cleanup/removal powers. One Defence -> Resist packet selects exactly one Resisted Attribute: Attack, Guard, Fortitude, Intellect, Synergy, or Bravery. Potency means Resists applied per successful power-roll success, so total Resists equals roll successes multiplied by Potency. Multi-attribute Resist requires multiple effect packets. Movement powers may be Counter-enabled and use Movement Counter timing, but Movement/Reposition is not a Defence mode.


25H. Natural failures, recoverable failures, and spike eligibility are locked

A natural 1 is always a hard failure.

A natural 1 cannot be modified into a success by positive modifiers.

A natural 1 cannot become a greater success, critical success, or any higher success tier.

A natural 2 or natural 3 is a recoverable failure.

Positive modifiers may raise a natural 2 or 3 to the normal success threshold.

If a natural 2 or 3 is rescued by modifiers, it can produce at most 1 normal success.

A rescued natural 2 or 3 cannot become a greater success, critical success, or higher-tier success, even if the final modified value would otherwise reach that tier.

A natural roll that already meets the normal success threshold is a natural success.

Natural successes may be modified normally.

Natural successes may be upgraded or downgraded into higher or lower success tiers according to the current documented success-tier thresholds.

Negative modifiers may reduce natural successes, including reducing them below the success threshold.

Do not change the existing success thresholds, dice sizes, maximum modifier, or success-tier definitions unless that is explicitly reopened in a separate rules pass.

Design intent:

- modifiers should remain meaningful, especially for low-die characters
- a strong positive modifier should be able to rescue weak rolls of 2 or 3 into competence
- a strong positive modifier should not erase all failure risk
- a buffed weak character may become reliable, but should not gain the spike ceiling of a true specialist
- true spike output belongs to natural successes, preserving the value of high attributes, high skill, and lane mastery


26. Charge and Channel are not Descriptor Chassis


Charge and Channel materially affect authoring, action economy, and cost, but they do not replace base delivery structure.


They are advanced Power-Level Field modifiers, not chassis.


27. Commitment Modifier is a Power-Level Field layer


The power builder should expose an always-visible Commitment Modifier selector with the options:


Standard


Channel


Charge


Standard is the default.


Channel means the power may or must continue consuming later Power Actions to maintain, reapply, or sustain its operation.


Charge means the power must or may consume one or more Power Actions building power before resolution, or may grow in strength while held before release.

In first implementation, only one Charge Type may be active at a time:

Delayed Cast

Build Power

Build Power uses bonus dice per charged turn.

Delayed Cast is not Reserve-only.

Delayed Cast cannot be Counter-enabled.

Reserve + Charge delayed-cast:

becomes primed after the required charge turns

once primed, may be held in reserve for up to its authored Reserve Lifespan

if not released before that Reserve hold window ends, it is lost and goes on cooldown

Non-Reserve delayed-cast:

becomes primed after the required charge turns

must be cast with a Power Action before the end of the user’s next turn

if not cast, is lost

Reserve owns release permission and release flavour.

Non-Reserve delayed-cast uses cast framing, not release framing.

For non-Reserve delayed-cast, Packet 1 timing and defence wording follow the authored Packet 1 Effect Timing rather than a forced ON_RELEASE bridge.

Build Power on non-Reserve chassis uses cast framing rather than release framing.


28. Commitment Modifier does not replace Lifespan, Effect Timing, or Effect Duration


Commitment Modifier answers a different question:


how the user must continue spending turns to operate the power


It must not be collapsed into Lifespan, Effect Timing, or Effect Duration.


29. Immediate does not force instant Effect Duration


Immediate means the chassis itself has no meaningful ongoing structural presence and the Primary Packet resolves now.


It does not mean the applied result of Packet 1 must have instant Effect Duration.


30. Packet 1 lands when it applies at least one operative result unit

Packet 1 lands if, after its required Trial or Clash resolution, it applies at least one operative result unit.

For wound-dealing packets, operative result is measured in net wounds inflicted.

For non-wound packets, operative result is measured in net applied successes, stacks, heals, or other effect units.

31. Secondary Packets survive on landing and may scale from applied result value

If Packet 1 does not land, all contingent Secondary Packets are discarded.

If Packet 1 lands, contingent Secondary Packets may resolve immediately or later according to their own Effect Timing.

Where a Secondary Packet scales from Packet 1’s result:

use net applied successes for non-wound primaries

use Applied Success Bands = CEILING(Net Wounds / Wounds Per Success) for wound-dealing primaries

Secondary Packets are always powered by Packet 1's operative result. If Packet
1 does not land or is fully defended, Secondary Packets do not resolve later
even if their authored timing occurs.

For Field, Attached, Triggered, and other recurring delivery, the first matching
Primary trigger each round rolls once and stores that roll for that round. Later
matching Primary triggers that round reuse the stored roll. Each Primary Target
defends separately, so applied Primary results may differ per target.

If a Secondary Packet targets the Primary Target, use that target's own applied
Primary result. If it targets Self, Allies, Field context, attached-host allies,
or another non-primary recipient, the first successful Primary application in
that round sets the Secondary payload for that round. A fully defended first
trigger sets no payload; the next successful trigger may set it. Once set,
later same-round Primary triggers do not replace, improve, reduce, or add to it
unless explicitly authored. Descriptors for these non-primary Secondary payloads
should refer to the power's Primary result this round rather than cumulative
wounds or successes across all same-round triggers.

For Field trigger descriptors, per-recipient area triggers default to the
triggering recipient. "A target enters / starts / ends / leaves the area" affects
that target unless the packet is explicitly authored as area-wide. A Field
Secondary timed to "an ally starts / ends their turn in the area" affects that
ally unless explicitly authored to affect all allies in the area.

Secondary Packets own their own Effect Timing and Effect Duration. They do not
inherit the parent power duration, Packet 1 duration, or chassis lifespan as
their own packet duration. An Instant Secondary Packet must not render as
lasting for 1 turn.

Individual power descriptors should not repeat the global one-roll-per-round /
stored Primary result rule unless the authored power explicitly changes that
rule. Preferred split-timing wording is "When [recipient] subsequently [timing],
and [Power] [Primary condition] this round..." rather than "then when...".

32. Mixed-intention legality is constrained by hostile-primary ordering

Hostile-primary into hostile-secondary is legal.

Hostile-primary into beneficial-secondary is legal.

Non-hostile-primary into non-hostile-secondary is legal.

Non-hostile-primary into hostile-secondary is illegal.

Mixed intention never creates additional defence gates.

33. Packet timing legality must not smuggle in a different chassis

Immediate Packet 1 timings are on cast / on use now and, where needed, on hit.

Field Packet 1 timings may use on create, on enter, start of turn, end of turn, and on expiry while the field exists.

Attached Packet 1 timings may use gate on attach or later attached-state payload timings such as start of turn, end of turn, movement, action, or expiry.

Trigger Packet 1 resolves on trigger.

Reserve Packet 1 resolves on release, or on reserve expiry only if the authored release behaviour explicitly says so.

34. Commitment Modifier may constrain legal resolution windows without replacing the timing triad

Charge and Channel do not create new packet timing categories.

However, they may constrain when Packet 1 is allowed to resolve and how long a structure may remain operational in a usable pre-resolution or maintained state.

This interaction does not replace the separate meanings of Lifespan, Effect Timing, or Effect Duration.

35. Channel uses one first-pass behaviour model only

In first implementation, Channel means the user may or must spend subsequent Power Actions to keep the power operating after initial use.

Channel is an action-commitment modifier only. It does not automatically reroll Packet 1 or create repeated packet resolution by itself.

Authored Effect Timing determines when each Effect Packet resolves.

Do not offer alternate channel modes in first implementation.

36. First-pass Channel legality is limited to supported chassis pairings

Immediate + Channel is supported.

Field + Channel is supported.

Attached + Channel is supported.

Trigger + Channel is not supported in first implementation.

Reserve + Channel is not supported in first implementation.

37. Channel does not override packet timing

Each authored Effect Packet resolves only at its own Effect Timing.

If a power is intended to apply effects repeatedly while maintained, those repeated effects must be authored through packet timing such as START_OF_TURN, END_OF_TURN, or other legal timings for the selected chassis rather than assumed from Channel alone.

Delayed Secondary Packets remain scheduled for their own later timing and do not automatically re-resolve unless their authored timing says they do.

38. Channel ends when upkeep ends or the sustaining anchor is broken

A Channelled power ends when the user stops maintaining it, can no longer maintain it, its Lifespan ends, or its sustaining anchor is broken.

Target-anchored channels end when all sustaining anchors are broken.

Source-projected or area-anchored channels do not end merely because individual targets fully defend that turn.

39. Reapplying live stacks from the same power/source refreshes them

Where a power reapplies live stacks from the same power/source, those live stacks are refreshed on application rather than requiring separate expiry tracking for each stack instance.

40. Attached powers use initial legal range for attachment, with separate rules for later host-bound persistence and Channelled reinforcement

Targets of Attached powers must be in legal range for the initial application.

Once attached, the attached state may continue and later attached effects may resolve for its Lifespan even if later range changes.

For first implementation, a Channelled Attached power may only add new applications, refreshes, or maintained re-resolution while the attached target remains within the power’s legal cast range from the source. Existing attached effects already on the host continue until their own duration, attachment end, or expiry rules say otherwise.


41. First-pass Immediate timing constraints stay conservative and honest

In the current builder surface, Immediate + Standard Packet 1 uses On Cast only.

If an Immediate Packet 1 resolves On Cast and does not create a beyond-turn carrier, later Immediate packets also remain On Cast only in first implementation.

Secondary scaling from the Primary Packet’s result class is derived rather than manually authored in the current builder surface.

41A. Legal Main Action choices are locked

During a Main Action, an actor may choose exactly one of the following legal options:

- Attack with one currently equipped weapon.
- Interact with an object or use an item. Examples include interacting with an artifact, pulling a lever, reloading a weapon, drinking a medicinal tonic, or accessing a computer.
- Rush, moving an additional distance up to half the actor's maximum move distance.
- Brace, allowing each failed die in a Physical Defence or Mental Defence roll to be rerolled once while the Brace benefit is available. A die rerolled by Brace cannot be rerolled again.
- Resist, rolling the relevant Resist Attribute value against persistent stacks or one ongoing damage effect.

The Main Action Resist option has two cleanup uses:

- Against persistent stacks, roll the authored cleanup Resist Attribute and remove stacks equal to rolled successes.
- Against ongoing damage, choose one ongoing damage effect and roll the cleanup Resist Attribute for that effect. Each success removes 1 ongoing damage unit from the chosen effect.

Power use remains a Power Action or legal Response unless a more specific future rule explicitly says otherwise.

42. Persistent states use stacks and a stack-cleanup model

Persistent states such as Control stacks and other stack-based states are tracked as stacks rather than as ongoing direct-value effects.

A target may use the Main Action Resist option to attempt the authored cleanup Resist Attribute and remove stacks equal to rolled successes.

Cleanse effects remove stacks per success.

At the end of each turn, persistent-state stacks naturally degrade by 1 unless a more specific rule says otherwise.

When the duration of the source power ends, any remaining stacks from that effect are removed.

43. Ongoing direct effects use ongoing value plus duration rather than stacks

Repeating direct effects such as ongoing damage, healing-over-time, or repeated direct restoration are tracked through ongoing value plus duration rather than stack count.

For Immediate + Standard packets that resolve On Cast and use a non-instant duration for a repeating direct effect, the effect applies initially on cast and then reapplies at the start of the affected target’s next turns for the authored duration.

Ongoing damage may be reduced by Cleanse or by the Main Action Resist option against one chosen ongoing damage effect.

Physical ongoing damage uses Fortitude Resist by default for this cleanup roll. Mental ongoing damage uses Bravery Resist by default.

Each cleanup-roll success removes 1 ongoing damage unit from the chosen effect.

Each ongoing damage unit removes the source effect's wounds-per-success / wounds-per-band value from the stored ongoing amount.

Ongoing damage does not naturally degrade by 1 each turn.

Expiry ends the ongoing damage effect.

Same-source reapplication may refresh duration without adding potency unless the power is explicitly authored to stack or escalate.

44. Packet-local secondary target classes may distinguish Primary Targets, Allies, and Self

Where a Secondary Packet needs a different recipient class from the Primary Packet, packet-local targeting may use:

Primary Targets

Allies

Self

Allies includes self by default. A power that should exclude the user must do so through an explicit authored restriction rather than by narrowing the baseline meaning of Allies.

The renderer should explain the contextual reach of that target class through chassis and range language such as inside the field, within the released area, or within range of the attached host rather than bloating the authored dropdown label itself.

44A. Range, target legality, and target count are separate

Range defines reach or area. Target legality defines who or what may be selected. Target count defines how many legal targets may be selected.

Range does not by itself make self legal or illegal.

Self can be selected through Self range, Melee range, Ranged range, or AoE / area placement when all of the following are true:

- self is in the power's legal target set
- self is inside the relevant range or area geometry
- self is selected as one of the allowed target slots
- the selected target count does not exceed the power's authored target count

Self consumes one target slot like any other target unless a power explicitly says otherwise.

A Ranged power that chooses 2 legal targets may choose self plus 1 ally if both are legal targets. It may not choose self plus 2 allies unless the power allows 3 targets.

A Melee power that chooses 1 legal target may choose self only if self is legal and the geometry permits it.

An AoE affects self only if the placed area includes self and self is a legal target. AoE placement does not make self a free or automatic target.

Self-only targeting remains the cheapest and cleanest option. Broader self-plus-other flexibility should be priced through normal range, target legality, target count, and area cost axes rather than blocked by a hidden rule that Melee or Ranged powers cannot target self.

45. Ambiguous Cleanse cleanup uses a narrow Cleanse Theme where needed

Where Cleanse targets an Active Power or a Channelled Power and the current authored specifics do not already settle one unique cleanup Resist Attribute, the builder may surface a packet-local Cleanse Theme disambiguator.

Cleanse Theme resolves to one exact cleanup Resist Attribute and exists only to keep the builder and descriptor honest.

46. Trigger powers may distinguish arm-first versus target-first watched structures

Trigger powers may use a Trigger Method distinction between:

Arm and then target

Target and then arm

Arm and then target means the power is armed now and legal target selection happens when the trigger condition occurs.

Target and then arm means legal target selection happens on cast and the power remains armed against that chosen target or those chosen targets for its Lifespan.

In both cases, Packet 1 still resolves on trigger, not on cast.

47. Secondary packet later timings require a real carrier from Packet 1

For Secondary Packets, chassis legality is not enough by itself.

If Packet 1 has Effect Duration Instant or Until the target's next turn, later recurring timings such as Start of Turn or End of Turn are not legal for Secondary Packets.

Those later timings are only legal where Packet 1 creates a real beyond-resolution carrier such as Turns or Passive, subject to the existing chassis timing rules.

48. Non-Immediate chassis require a real Lifespan in the current builder surface

Immediate remains the only chassis that can honestly have no Lifespan requirement.

Field, Attached, Trigger, and Reserve require a real Lifespan.

In the current builder surface they use Turns or Passive, default to Turns, and do not allow None.

49. Self-origin AoE wording must distinguish static placement from attached presence

When a sphere area uses cast range 0 from the user's own position, static chassis wording must describe the area as centered on the user's current space rather than implying that it moves with the user.

Where the area is hosted through an Attached structure on the user, the wording must describe the area as attached to the user.

50. Overlapping areas from the same packet stack before one defence roll

If multiple distinct areas from the same packet affect the same target at the same timing, the power rolls once, determines the packet's hostile result once, multiplies that hostile result by the number of overlapping authored areas affecting that target, and then resolves one defence roll against that multiplied result.

This preserves one defended entry point while allowing focused overlap from the same packet to matter.

51. Packet-local recipient classes must remain legal within shared cast context

Packet-local recipient classes must be legal within the power's shared cast context unless an explicit advanced local targeting override truthfully authorizes a different recipient lane.

In the current builder surface:

Self-only powers collapse packet-local recipient selection to Self only.

Single-target Melee and single-target Ranged powers do not automatically exclude self. They may select self only when self is legal, within range / geometry, and counted as the one selected target.

If a legally authored packet has no legal recipient when it would resolve, that packet does not apply to that recipient.

52. Duration defensive powers create finite defensive pools

Duration defensive powers do not reroll or reapply their full value for every incoming attack unless the power explicitly says they do.

This applies to:

- duration Dodge powers
- duration Physical Defence Block powers
- duration Mental Defence Block powers
- duration Resist powers

When the power is applied, roll it once.

Pool size = successful power-roll successes x potency.

The pool remains available until one of the following ends it:

- the pool is spent to 0
- the power duration expires
- the effect is removed or cleansed
- the source / target lifecycle ends the effect

Per incoming trigger, the maximum committed from a single defensive pool is the power's potency unless the power explicitly modifies that cap.

Defensive pools may be committed as a pool-only defence or as a supplement to a normal active defence. The defender chooses one legal path for each incoming hostile action: take the hit/effect, commit one matching defensive pool only, roll the matching normal defence only, or commit one matching defensive pool and roll the matching normal defence.

Commit timing is locked:

1. Incoming hostile action is declared and rolled.
2. If the hostile action produced one or more actionable hostile successes, the defender chooses whether to take the result, commit a matching defensive pool only, roll the matching normal Dodge / Physical Defence / Mental Defence / active Resist only, or use both a matching pool and the matching normal defence.
3. If a pool is used, commit pool points up to the per-trigger cap and remaining pool before any normal defence or active Resist roll is made.
4. If a normal defence or active Resist is chosen, roll it against the remaining hostile pressure.
5. Add committed pool points to the relevant result only when a normal defence or active Resist is also rolled; in a pool-only defence, the committed points directly provide their lane-specific cancellation or prevention.
6. Spent pool points are spent whether they fully stop the incoming result or prove ineffective.
7. Pool-only commitment does not roll or resolve Dodge, Physical Defence, Mental Defence, or active Resist, so it does not cause resolution-based degradation.
8. If a normal defence or active Resist is rolled, resolution-based degradation applies only to the independent defence lane that actually resolves against one or more actionable hostile successes.

The defender cannot wait to see the normal defence or Resist roll before deciding whether to spend pool points. Pool spending does not reset normal Dodge, Physical Defence, Mental Defence, or active Resist degradation in a lane that is also rolled. Pool-only commitment is a finite-resource alternative to normal defence, not an automatic normal defence declaration.

Dodge Pool points add to normal Dodge successes when Dodge is also rolled. In pool-only use, they count as Dodge successes for that incoming packet without rolling Dodge or degrading the Dodge lane. They are usable only against physical attacks or actions that explicitly allow Dodge. They do not make mental attacks dodgeable.

Block Pool points add directly to wounds prevented in their matching lane, whether used pool-only or alongside normal Physical Defence / Mental Defence. Physical Block Pool prevents physical wounds. Mental Block Pool prevents mental wounds.

Resist Pool points add to the matching active Resist result when active Resist is also rolled. In pool-only use, each committed point cancels one hostile success, stack, ongoing unit, or equivalent unit in the authored resisted Attribute lane without rolling active Resist or degrading that lane. A Resist Pool is not generic unless explicitly authored as generic. One packet maps to one resisted Attribute.

Counter Dodge remains a one-shot Counter use unless the power also explicitly creates a duration Dodge Pool. Counter use still replaces normal active defence for that trigger unless the power explicitly grants or modifies that defence. Already-active passive and static effects still apply.

53. Defensive pool chassis interactions use refresh/replace by default

Trigger and Reserve change when a packet is delivered. Once a Trigger or Reserve packet delivers a duration Dodge, Block, or Resist effect, that effect follows the normal finite defensive pool rules.

For Trigger-delivered defensive pools:

- create the pool when the Trigger packet is delivered
- pool size = delivered power successes x potency
- per-trigger commit cap = potency unless explicitly modified
- duration begins when the Trigger packet is delivered unless the power explicitly says otherwise
- repeated delivery from the same source power, packet, and protected target uses refresh/replace, not additive stacking by default

For Reserve-delivered defensive pools:

- create the pool when the Reserve packet is released
- use the released packet's final dice and potency
- duration begins on release
- repeated release/application from the same source power, packet, and protected target uses refresh/replace, not additive stacking by default

Field defensive pools use the Field's authored roll and delivery cadence. Each legal target receives its own target-specific pool instance. A Field can maintain only one active pool instance per same source power, packet, and protected target. If the same Field packet reapplies to a target that already has that pool, roll the new application and use refresh/replace:

- active remaining pool becomes the higher of current remaining pool points or newly generated pool points
- duration/availability refreshes from the latest legal application
- pools do not add together by default

Field pools are location-bound by default. They are usable only while the protected actor remains a legal target of the Field unless the power explicitly says the pool lingers after leaving. When the Field ends, pools granted by that Field end unless explicitly authored to linger. If protection should follow the target independently, author it as Attached or as an explicit lingering rider.

Attached defensive pools follow the attached target while the attachment persists. Reapplication uses the Attached power's authored delivery timing and the same refresh/replace rule. When the attachment ends, the pool ends unless the power explicitly says it lingers.

For repeated applications from the same source power and packet to the same protected target, the default refresh/replace rule is:

- do not add pools together
- roll the new pool
- compare new pool points to current remaining pool points
- keep the higher value
- refresh duration/availability from the latest legal application

This avoids making beneficial reapplication worse while also preventing armour/avoidance battery exploits.

Additive or recharging defensive pools are special mechanics, not defaults. They require explicit authoring, added cost, and limiting rules such as maximum pool cap, clear expiry, explicit stack/recharge wording, likely lower per-trigger cap or separate cap, and clear Field/Attached/Channel interaction.

A defender may commit from only one matching defensive pool per incoming action unless a power explicitly allows combined pool spending. If two Dodge Pools are active, choose one Dodge Pool for that incoming attack. If one Block Pool and one Dodge Pool are active, the chosen defence mode determines which pool can be used.

Channelled defensive pools reapply while the channel is maintained according to authored Effect Timing. Each reapplication rolls normally and uses refresh/replace for the same source power, packet, and protected target. Channelled defensive pools do not add by default. If the channel ends, channel-dependent pools end unless the power explicitly says they linger.

Charged defensive pool powers use final charged values at release. If Charge grants bonus dice, those dice increase the release roll. If Charge grants bonus potency, final charged potency increases both pool size and per-trigger commit cap. Duration begins on release; charge turns before release do not count against pool duration.

Descriptor terminology for pool-generating Defence powers is canonical: "creates a Physical Block Pool with N point(s) per success", "creates a Mental Block Pool with N point(s) per success", and "creates a Dodge Pool with N point(s) per success". Individual power descriptors must not repeat global pool operation, refresh/replace, or commitment-cap rules.

54. Codex-run Combat Lab simulation batches are an accepted incremental balance workflow

Manual Combat Lab runs remain important for major milestone confidence, UI/report readability checks, and table-feel judgement.

Codex-run Combat Lab simulation batches may be used for incremental balance passes, variant comparison, canary checks, repeated scenario matrices, outlier detection, and checking whether a patch moved win rates or diagnostics in the expected direction.

Codex-run results are evidence only when their provenance is reported. They are not magic truth.

Every Codex-run simulation report must state:

- commit SHA
- git status / worktree cleanliness
- exact command, script, route, API endpoint, direct runtime call, or other method used
- whether the browser UI was visually inspected
- run count

- scenario names
- actor and monster names
- active Power Tuning name
- active Combat Tuning name
- active Outcome Normalization name
- any temporary in-memory overrides
- confirmation that no database, source-data, tuning, saved-monster, or code changes were made during a read-only simulation pass
- validation or smoke status where relevant

Recommended report fields include:

- player win %
- monster win %
- stalemate %
- average rounds
- player damage per round
- monster damage per round
- action use counts
- cooldown-prevented counts
- defensive choice diagnostics

Simulation reports must distinguish whether results came from UI/browser inspection, API route execution, script execution, direct runtime calls, or another method.

Codex-run results are trusted only when runtime rules are current, active tuning is identified, scenario construction is clear, seed/run count are reported, no hidden data changes occurred, and diagnostics are readable rather than contradictory.

55. Secondary wound-band scaling uses total Primary wounds per success by default

When a Secondary Packet scales from Primary wound bands, the default band size equals the Primary Packet's total simultaneous wounds per success across all damage components. Multi-damage Primary Packets increase the band size; they do not generate separate bands per damage type unless explicitly authored.

A single Primary Packet outputs the same wound total per success for every simultaneous damage component / damage type. Example: Potency 3 means 6 wounds per success. If the Primary has Fire, Ice, and Slashing damage, each damage type outputs 6 wounds per success, for 18 total simultaneous wounds per success. A default wound-band Secondary therefore uses one band per 18 wounds in the Primary result, rounding up. It does not create three separate "for every 6 wounds" band engines.

Default examples:

- Single damage type: Primary Packet deals 6 physical Slashing wounds per success. A Secondary Packet that scales from Primary wound bands uses one band per 6 applied Primary wounds, rounding up.
- Three simultaneous damage types: Primary Packet has Potency 3 and deals Fire, Ice, and Slashing wounds. Each type deals 6 wounds per success. Total simultaneous wounds per success is 18. A default wound-band Secondary uses one band per 18 wounds in the Primary result, rounding up.

Explicit exceptions:

- Component-specific wound-band scaling may be authored as a special rider, such as "for every 6 Slashing wounds in the Primary result". That uses only the Slashing component and must be explicitly authored and priced.
- Per-component additive wound-band scaling may be authored only as a special multi-component band engine, such as "each damage component generates its own wound bands". This is not default and requires additional pricing.

Unresolved future design debt: overkill-scaling and wound-band source.

Should Primary wound-band scaling use actual wounds suffered after remaining-health cap, or net wounds before overkill is discarded? The current recommendation is that default scaling should probably use actual wounds suffered to avoid farming riders from overkilling 1 HP targets. Overkill-scaling may be allowed later only as an explicit premium mechanic. This pass does not lock the final overkill answer.

56. Vulnerability, Resistance, and Protection modify defence rolls, not wounds

Incarnate has no active general Immunity rule. Damage immunities may be added later through Monster Traits, but they are future design debt and must define their own interaction with defence rolls, wound results, and Secondary wound-band scaling when implemented.

Vulnerability, Resistance, and Protection trigger from damage types on the incoming packet. If the incoming packet has a triggering damage type, apply the relevant Vulnerability / Resistance modifiers and Protection dice to the one defence roll against that packet, then calculate the Primary result normally.

Vulnerability applies a negative modifier to the relevant defence roll. It does not increase wounds directly, multiply wounds, or alter the wound-band denominator.

Resistance applies a positive modifier to the relevant defence roll. It does not reduce wounds directly, divide wounds, or alter the wound-band denominator.

Protection adds additional dice to the relevant defence roll. It does not reduce wounds directly, subtract wounds, or alter the wound-band denominator.

Vulnerability and Resistance compare and combine as defence-roll modifiers. Example: Vulnerability Ice 2 and Resistance Ice 1 against an incoming Ice packet yields net -1 to the defence roll. The result is not extra Ice wounds or reduced Ice wounds. It is simply a net defence modifier before the defence roll is resolved.

Multi-damage packets still produce one Primary result. If an incoming packet has multiple damage types, each relevant Vulnerability, Resistance, or Protection entry may trigger. Apply all relevant defence modifiers and bonus dice before the one defence roll against that packet. Do not split the packet into separate defence rolls by damage type unless an explicit future rule says otherwise.

Wound-band interaction:

Vulnerability, Resistance, and Protection modify the defence roll against an incoming packet when their triggering damage types are present. They do not modify wounds directly and do not alter Primary wound-band denominators. For Secondary Packet wound-band scaling, resolve the Primary Packet normally, including any Vulnerability / Resistance / Protection effects on the defence roll, then calculate wound bands from the resulting Primary wound total using the Primary Packet's authored total simultaneous wounds per success as the default band size.

Examples:

- Vulnerability and Resistance: an incoming Primary Packet includes Ice. The target has Vulnerability Ice 2 and Resistance Ice 1. Net defence modifier is -1. Resolve one defence roll with that modifier. The resulting Primary wound total is then used for any Secondary wound-band scaling.
- Protection: an incoming Primary Packet includes Fire. The target has Protection Fire 2. Add 2 dice to the relevant defence roll. Protection does not subtract wounds directly. After defence is resolved, the resulting Primary wound total is used for Secondary wound-band scaling.
- Multi-damage Primary: a Primary Packet deals 6 Fire, 6 Ice, and 6 Slashing wounds per success. Total authored simultaneous wounds per success is 18. The target has Vulnerability Ice 2, Resistance Fire 1, and Protection Slashing 1. Apply -2 defence modifier from Ice Vulnerability, +1 defence modifier from Fire Resistance, and +1 defence die from Slashing Protection, then resolve one defence roll against the packet. After the Primary result is calculated, default Secondary wound-band scaling still uses one band per 18 wounds in the Primary result, rounding up.
- No active immunity rule: do not say "Fire immunity reduces Fire wounds to 0" as an active rule. Damage immunity is not currently part of the active Vulnerability / Resistance / Protection system. Future Monster Trait immunities must define their interaction with defence rolls, wound results, and Secondary wound-band scaling when implemented.

57. Mixed defensive Counters choose one active defence path on declaration

If a Counter contains multiple active defence packets, the actor chooses one eligible active defence path when declaring the Counter, before the incoming roll and before the Counter roll are made. Only the chosen active defence path resolves.

Unchosen active defence packets do not apply, do not create pools, do not provide successes, do not mitigate, and do not act as fallback protection against that incoming packet. This is the default for mixed defensive Counters.

Counter declaration includes the active defence path choice when the Counter has more than one active defence path. The choice is made after the incoming hostile action is declared, but before incoming and Counter rolls are made. The actor may use known declared information, such as attack type, target, damage type, range, and visible threat context, to choose the mode. The actor may not wait to see the roll result before choosing unless the power explicitly has Adaptive Counter wording.

Terminology:

- Single-lane Counter: a Counter with one active defence path, such as a Dodge Counter, Physical Defence / Block Counter, Mental Defence / Block Counter, or Resist Counter.
- Modal Defence Counter: the default for a Counter with multiple active defence paths. The actor chooses one eligible active defence path when declaring the Counter. Only the chosen path resolves. Unchosen active defence packets do not apply.
- Adaptive Counter: a premium / explicit exception. The power explicitly allows the actor to choose the active defence path after seeing the relevant roll/result. This is not default behaviour and has no implementation support from this documentation pass.
- Layered Defence Counter: a premium / explicit exception. The power explicitly allows more than one active defence path to apply against the same incoming packet, such as "try to Dodge; if Dodge fails, also Block". This is not default behaviour and has no implementation support from this documentation pass.

Example:

A Counter contains both a Dodge packet and a Physical Defence / Block packet. When the Counter is declared, the actor must choose Dodge mode or Block mode. If Dodge mode is chosen and the Dodge result is not enough to avoid the incoming packet, the actor may not then switch to Block. If Block mode is chosen, the Dodge packet does not apply.

Rationale:

Dodge and Block have different lane identities. Dodge is high-ceiling, all-or-nothing avoidance. Block / Defence is more reliable partial mitigation. Allowing the actor to roll first and choose afterwards would create post-result optimisation. That would turn mixed Dodge + Block Counters into Dodge upside plus Block insurance, making single-lane Counters easier to obsolete and harder to price fairly. Mixed defensive Counters are still valuable because they provide matchup versatility at declaration time, not post-roll insurance.

Passive/static mitigation remains governed by existing mitigation rules; this lock controls only active defence packets inside the Counter. This rule does not change Attack Counters, Movement Counters, Control Counters, Assist rules, or the global Counter timing order except where a power also contains multiple active defence packets. The Modal Defence Counter choice happens inside the existing "Counter is declared" step.

58. Interrupting Control Responses require Control Primary and strict exceed

Control Counters and Control Assists can halt an incoming action only when Control is the Primary packet / Primary intention of that Counter or Assist, and only when the chosen Control effect could logically stop the incoming action.

Control as a Secondary packet is after-effect Control by default. It cannot interrupt, cancel, halt, weaken, reduce, or partially mitigate the triggering incoming action.

An interrupting Control Counter or Assist succeeds only if its Control result strictly exceeds the incoming action's successes. A tie is not enough. If the interrupting Control result does not strictly exceed the incoming action, the interrupt fails completely and the incoming action resolves normally. Its successes, wounds, potency, stacks, duration, movement, target count, and other outputs are not reduced. Failed interrupting Control does not become weaker Block, Resist, Dodge, tax, damage reduction, or any other partial mitigation.

This mirrors Dodge's all-or-normal shape: Dodge succeeds and avoids the packet, or fails and the packet resolves normally. Interrupting Control succeeds and halts the incoming action, or fails and the incoming action resolves normally.

The attacker does not make a separate Resist roll against an interrupting Control Counter or Assist by default. The incoming action's successes are the interruption contest value because they represent the strength, momentum, or quality of the action being interrupted. Do not describe this as the attacker "magically getting a Resist roll", and do not add a separate attacker defence roll unless a future explicit rule does so.

Primary / Secondary cases:

- Attack Primary + Control Secondary Counter: If a Counter has Attack Primary + Control Secondary, the Control does not use strict-exceed interruption. The Attack may resolve as counter damage, and the Control is an after-effect only; the incoming action is not halted. The Control Secondary may apply after the exchange only if it is powered by the successful Primary Attack result and its own timing / legality conditions are met.
- Control Primary + Attack Secondary Counter: If a Counter has Control Primary + Attack Secondary, the Control may use strict-exceed interruption. The Attack Secondary does not resolve regardless; it resolves only if powered by the successful Primary Control result, unless the power explicitly says otherwise. If Control fails to strictly exceed, the incoming action resolves normally and secondaries do not fire from the failed Primary unless explicitly authored otherwise.
- Control Secondary in any Counter or Assist is after-effect Control by default. It cannot cancel the triggering action, reduce incoming potency, reduce incoming successes, or partially mitigate damage.

Assist follows the same law. An Assist can halt an incoming action only if Control is the Primary packet / Primary intention, the Control effect is logically capable of stopping that incoming action, the Assist is being used as an interrupting Response, and the Control result strictly exceeds the incoming action's successes. If an interrupting Control Assist fails to strictly exceed, it provides no mitigation and the incoming action resolves normally.

Logical relevance examples:

- Force No Main Action can interrupt an incoming Main Action if it strictly exceeds.
- Force No Power Action can interrupt an incoming Power Action if it strictly exceeds.
- A movement-stopping Control may interrupt a charge or movement-dependent action if it logically prevents the action.
- A Control that only affects later movement cannot cancel a melee attack that is already in range unless explicitly authored.
- A Control that only applies an after-effect condition cannot retroactively stop the triggering action.

If a successful interrupting Control also applies persistent Control stacks, duration, or an ongoing effect, those follow normal Primary-result scaling. Intended default: interruption is checked by strict exceed; persistent follow-up Control magnitude scales from applied / net interrupt success, not raw successes, unless explicitly authored otherwise.

This rule preserves the global Response Window / Incarnate Stack order. The interrupt decision is part of Response / Counter / Assist resolution. The Control packet must be Primary to interrupt; Secondary Control cannot hijack timing and cancel the triggering action.
