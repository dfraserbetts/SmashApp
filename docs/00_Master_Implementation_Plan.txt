Purpose

This document is the top-level source of truth for the SMASH TTRPG power-system redesign. Its purpose is to keep future design and implementation work aligned across multiple chats and development sessions. It is intentionally strategic rather than exhaustive. Detailed rules, matrices, and edge cases belong in the more specific source documents.

Project Goal

SMASH is moving from a legacy power-building approach toward a shared power framework that supports both monster powers and player powers through a single underlying architecture.

The goal is to preserve the original speed and clarity of the descriptor engine while adding enough structure for powers to feel distinct, premium, and mechanically expressive rather than just baseline actions with bigger numbers.

Thematic Scope and the Heart of the Game

SMASH TTRPG allows players to create the characters and powers they imagine.

Unlike games that say, “Pick from these classes and we will tell you how your character plays,” SMASH asks the player, “How do you imagine your character? How do they play? What can they do?”

The player imagines the character, how they would behave, and what powers they would have. SMASH then provides the tools to create those results in a mechanically viable and balanced way.

This is a foundational design truth and should not be forgotten during implementation. The system exists to support authored character imagination, not to force players into narrow pre-shaped identities.

Core Direction

The agreed direction is that powers should be built from four implementation-relevant layers:

Shared Rules Grammar

These are the common fields that keep the system readable and consistent, such as Range, Targeting, Dice, and Counter. Responses remain the player-facing resource; Assist is universal by system rule; Counter is the authored power permission field.

Descriptor Chassis

Delivery Method is treated as a descriptor chassis. It changes sentence structure, required fields, and broad resolution pattern. It is not just a flavour tag.

Effect Packets

Each intention inside a power is represented by an effect packet. Packets define the actual mechanical outcomes of the power. Packet outcomes, timing, duration, scaling, and most effect specifics belong here.

Commitment Modifiers

Standard, Channel, and Charge are Power-Level Field commitment modifiers, not Descriptor Chassis. They change action commitment and operation cost, but do not replace Lifespan, Effect Timing, or Effect Duration.

Design Intent

The redesign exists to solve several problems at once:

Powers should feel structurally different from weapon attacks and defence strings.

The system must remain mechanically readable and descriptor-driven.

Simple powers must remain quick to build.

Complex powers must become possible without collapsing into ambiguous text.

The same foundation must support both Summoning Circle monster powers and later Character Builder player powers.

Agreed Structural Principles

The following principles are core to the redesign:

The first packet in a power is always the Primary Packet.

Only the Primary Packet is defended against.

If the Primary Packet does not land, contingent Secondary Packets do not land.

Delivery Method must be chosen early enough in the workflow to shape the form correctly.

Not every power needs every layer of complexity. Straightforward powers should stay fast and simple.

Complexity has a meaningful cost in SMASH and must continue to do so. More intentions, longer lifespans, longer effect durations, richer timing structures, and stronger action commitment should all consume more budget.

Packet Resolution Direction

Packet 1 lands if, after its required Trial or Clash resolution, it applies at least one operative result unit. Secondary Packets remain contingent on Packet 1 landing. Where a Secondary Packet scales from Packet 1, it scales from the Primary Packet’s applied result value rather than bypassing the result of the defended entry point.

Stack effects are tied to packet resolution, not just chassis existence. Repeating packet timings such as START_OF_TURN, END_OF_TURN, or the channel-gated recurring timings create repeating stack application opportunities only when those packets actually resolve.

Implementation Strategy

The shared power core is now actively implemented behind Summoning Circle.

Summoning Circle is no longer just the proving ground in theory; it is the live implementation surface for the shared architecture. The immediate objective is to finish cost integration, preserve descriptor trust, keep builder clarity high, and complete the remaining implementation-facing hard rules without reopening solved architecture.

The Character Builder will later reuse the same foundation, then layer player-only systems such as Spark and Restriction on top.

Monster vs Player Approach

Monsters and players should use the same shared power architecture.

Standard monsters should not use the full player-facing Spark / Restriction subsystem.

Legendary monsters may eventually gain a curated monster-facing equivalent, but not the raw full player implementation.

Runtime Direction

Tracking debt is a known system risk.

Long-term support will likely include:

printable effect slips

later optional digital encounter runtime support

These are future layers and should mirror the same underlying power structure rather than introducing a separate tracking model.

Current Implementation Reality

The monster power builder now exists as a real implementation surface.

The current implementation already includes:

- canonical chassis selection for Immediate, Field, Attached, Trigger, and Reserve
- packet-local potency, timing, duration, recipient, and specific-effect authoring
- visible Lifespan where structurally relevant
- Counter and Commitment as Power-Level authoring controls
- first-pass Charge support using the player-facing Charge Types Delayed Cast and Build Power
- first-pass Channel support as action commitment only
- timing-gated descriptor rendering for current packet timing values
- normalized Trigger Condition, Release Behaviour, Host / Anchor, and Movement Theme authoring controls
- Reserve delayed-cast using release framing while non-Reserve delayed-cast uses cast framing
- current descriptor logic distinguishing persistent state effects from repeating direct effects such as ongoing damage
- preview honesty fixes across all five chassis and all three commitment modifiers
- live pressure-test fixtures and an approved shared canary catalog
- control-theme-based resolution for genuinely ambiguous control effects

Phase 6 cost integration is no longer hypothetical. The current codebase now includes:

- a canonical Phase 6 power-cost resolver
- a versioned Power Tuning system with active / draft / archived sets
- a separate versioned Outcome Normalization system for radar interpretation and display
- a separate versioned Combat Tuning / Monster Baseline Tuning system for non-power monster baseline contribution
- a Canary Harness that compares active baseline versus candidate Power Tuning, Outcome Normalization, and Combat Tuning sets
- a shared stripped Summoning Circle baseline used by both the live editor and the harness
- a cleaned radar pipeline where canonical power contribution is derived first, then combined with non-power monster contribution, then passed through outcome normalization

This means the current implementation now has three distinct tuning layers:

Power Tuning
- the canonical underlying power-cost truth

Combat Tuning / Monster Baseline Tuning
- the non-power monster baseline contribution that feeds the outcome calculator

Outcome Normalization
- the downstream radar interpretation and display layer

Success Criteria

This redesign is successful if:

powers remain fast to read and mechanically solid

delivery methods create genuinely different descriptor structures

multi-intention powers become clean to author and resolve

the same shared foundation supports monsters first and players later

the cost model remains one shared underlying truth rather than splitting into contradictory systems

the runtime implications of powers can eventually be tracked in print and digital forms without inventing a separate rules language
