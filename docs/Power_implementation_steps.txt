Immediate practical next-step direction

The architecture is no longer the main risk.

The practical next work should stay focused on:

- implementing the locked Response Window / Incarnate Stack in runtime-facing tools
- preserving that Counter use replaces normal active defence unless explicitly modified by the Counter
- keeping Counter-enabled powers normally usable; Counter: Yes adds response use rather than exclusivity
- adding shared Block, Dodge, and Resist defensive power options for Summoning Circle and Character Builder; Movement Counter remains a Movement-power rule, not a Defence mode
- treating duration Dodge, Block, and Resist powers as finite Dodge Pool, Block Pool, and Resist Pool resources that roll once on apply, spend down per matching trigger, and do not refresh their full value unless explicitly authored
- using refresh/replace higher-of-current-or-new for repeated defensive pool applications from the same source power, packet, and protected target; additive/recharging pools require explicit special authoring and pricing
- preserving the simplified targeting rule: target self legality is selected-target legality, not range exclusivity; descriptors and validation must count self against the authored target count
- applying strictly-exceed timing for Movement and Control interrupt Counters; ties fail
- checking runtime and tests for the locked modifier/failure ruling in a separate implementation pass: natural 1 hard failure, natural 2/3 recoverable failure capped at 1 normal rescued success, and natural successes as the only spike-eligible rolls
- checking runtime and tests for the zero-success hostile action ruling in a separate implementation pass: declaration/reaction triggers still fire and declared Counters/Assists remain spent, but normal defence, defensive-pool commits, and resolution-based defence degradation require one or more actionable hostile successes
- checking runtime and tests for independent defence-lane degradation in a separate implementation pass: Physical Defence, Mental Defence, Dodge, and active Resist must degrade only in the lane that actually resolves
- using the now-aligned Canary Harness as the comparative tuning workspace
- using Codex-run Combat Lab simulation batches for incremental balance passes, variant comparison, canary checks, scenario matrices, and outlier detection
- keeping manual Combat Lab runs for major milestone confidence, UI/report readability, public/demo confidence, playtest packet confidence, and table-feel judgement
- reporting commit SHA, worktree cleanliness, command/script/route/direct-runtime method, UI inspection status, run count, seed policy, scenario names, actor/monster names, active Power Tuning, active Combat Tuning, active Outcome Normalization, temporary in-memory overrides, no-mutation confirmation, and validation/smoke status when Codex runs simulations
- extending the existing scripts/combatLab.scenarioMatrix.ts runner with the defensive-pool canaries defined in docs/14_Defensive_Pool_Canary_Matrix.txt before larger defensive-pool balance passes
- tuning canonical power values against the current builder surface
- tuning non-power monster baseline contribution where stripped monsters prove too weak or too strong
- tuning radar display curves only after canonical power and baseline contribution are understood
- keeping the three-layer split clear:
  - Power Tuning
  - Combat Tuning
  - Outcome Normalization
- beginning the later monster-side realization modifier layer only after the current tuning layers behave honestly
- keeping the source docs aligned with current code reality
