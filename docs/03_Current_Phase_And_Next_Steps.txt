What Is Now Locked

The following points should now be treated as stable design truth:

- the shared power core uses one three-layer model
- Delivery Method is structural and does not replace shared rules grammar
- Delivery Method is implemented through Descriptor Chassis
- Packet 1 is always the Primary Packet
- only the Primary Packet is defended against
- Secondary Packets only resolve if the Primary Packet lands
- Secondary Packets do not create separate defended entry points
- Packet 1 lands if it applies at least one operative result unit
- wound-based Secondary scaling uses Applied Success Bands = CEILING(Net Wounds / Wounds Per Success)
- Lifespan, Effect Timing, and Effect Duration are separate concepts
- Summoning Circle is the first implementation surface for the shared core
- Character Builder is expected to reuse the same foundation later
- live preview is mandatory
- the practical Descriptor Chassis matrix is Immediate, Field, Attached, Trigger, and Reserve
- Counter is an authored Power-Level Field permission with No / Yes
- Assist is universal by system rule and is not authored per power
- Responses are the player-facing resource and are not the authored power permission field
- Commitment Modifier uses Standard / Channel / Charge
- first-pass Channel support is limited to Immediate, Field, and Attached
- hostile delayed-entry Attached powers must explicitly distinguish gate on attach from gate on payload
- Immediate + Standard Packet 1 currently uses On Cast only in the builder surface
- if an Immediate Packet 1 resolves On Cast and creates no beyond-turn carrier, later Immediate packets remain On Cast only in first implementation
- secondary scaling from the Primary result class is derived in the current builder surface rather than manually guessed
- persistent states use a stack-cleanup model distinct from ongoing direct effects
- ongoing damage uses ongoing value plus duration rather than stacks
- physical ongoing-damage cleanup defaults to Fortitude Resist and mental ongoing-damage cleanup defaults to Bravery Resist
- packet-local recipient class may distinguish Primary Targets, Allies, and Self where a Secondary Packet needs a different target class from Packet 1
- Allies includes self by default unless an explicit authored restriction says otherwise
- packet-local recipient classes must remain legal within the shared cast context
- Trigger powers may use Trigger Method to distinguish Arm and then target from Target and then arm while keeping Packet 1 on trigger
- Secondary timing choices must obey both chassis legality and carrier legality
- Immediate remains the only chassis that can honestly have no Lifespan requirement in the current builder surface
- Field, Attached, Trigger, and Reserve require Turns or Passive and default to Turns
- overlapping areas from the same packet at the same timing use one power roll, pre-defence multiplication by overlap count, and one defence roll
- canonical attribute naming is Guard and Synergy, not Defence and Support
- Bravery is the mental defence dice stat
- Dodge now succeeds on match or exceed, not exceed-only
- Dodge is physical-only in monster survivability baseline work
- Resist dice do not create offensive radar axes by themselves
- natural attacks and equipped weapons with the same authored at-will attack profile must produce the same non-power threat contribution
- natural protection and equipped protection with the same authored defensive profile must produce the same non-power survivability contribution
- mental protection has its own calculator-economy seam and must not be forced back into the exact same direct-budget curve as physical protection
- source defaults are smoke-test fallback values only
- active tuning snapshots are the current live balance truth for Outcome Calculator calibration


What Phase 6 Now Includes

Phase 6 no longer means only “design the cost model.”

The current implementation now includes:

- a canonical Phase 6 power-cost resolver
- a Power Tuning system with versioned active / draft / archived sets
- an Outcome Normalization system with versioned active / draft / archived sets
- a Combat Tuning / Monster Baseline Tuning system with versioned active / draft / archived sets
- a Canary Harness for comparing active and candidate tuning sets
- an active Outcome Calibration Harness for behaviour regression testing
- active tuning snapshot export workflow
- snapshot-backed harness mode for live balance truth
- source-default harness mode for fallback smoke testing only
- a shared stripped Summoning Circle baseline so live editor and harness comparisons align
- a cleaned monster radar pipeline where:
  - canonical power contribution is derived first
  - first-pass derived cooldown is now resolver output after BasePowerValue
  - per-power availability/cooldown adjustment is applied for monster outcome contribution
  - non-power monster contribution is added second
  - outcome normalization is applied last
- a real-monster manual validation checklist for field-testing authored campaign monsters

The tuning model now has three distinct layers:

Power Tuning
- canonical power truth

Combat Tuning / Monster Baseline Tuning
- non-power monster baseline contribution

Outcome Normalization
- final radar interpretation and display


Current Calibration Reality

The active calibration workflow now exists.

The active workflow is:

1. export active tuning snapshots
2. run the Outcome Calibration Harness in active mode
3. treat active-mode results as live balance truth
4. treat source-default mode as smoke-test fallback only

The current command sequence is:

npm run calibration:export-active-tuning

npx --yes tsx scripts/outcomeCalibrationHarness.ts --tuning active

The active harness is currently calibrated against active snapshots.

The current target state is:

29 fixtures
29 pass
0 warn
0 fail
0 mismatches

This is the current regression shield for the live Outcome Calculator.

Defaults mode is not balance truth.

Defaults mode may warn and still be acceptable as long as it runs successfully as a smoke test.

The source-default harness exists to catch code breakage and fallback configuration errors, not to decide live balance.

Near-Term Priority Questions

The next real questions are no longer broad architecture questions.

They are:

- whether real authored monsters consistently match their intended archetype shape
- whether existing monsters authored before the calibrated Pressure / Presence model need re-authoring rather than tuning
- whether any remaining manual mismatch belongs to monster authoring rather than the calculator
- which powers are still over- or under-valued in canonical Power Tuning
- whether stripped monsters are too weak or too strong before powers are added
- whether radar curves are saturating too early or too late once power plus non-power contribution are combined
- which current costs should remain cost-only versus later translated through cadence, realization, or structural pressure
- how the future monster-side realization modifier layer should gate powers, weapons, armor, and other typed contributors through the relevant monster attributes
- what the correct long-term low-end and mid-band physical dodge valuation should be under match-or-exceed rules
- how much cheaper or more efficient MPV should be than PPV in Forge and in monster baseline contribution without breaking the table-facing defence experience


Important Anti-Drift Rule

Do not let future chats collapse these into one blurred balance system.

If a problem belongs to:

- the power itself -> fix in Power Tuning
- the stripped monster body / non-power baseline -> fix in Combat Tuning
- the radar display curve -> fix in Outcome Normalization
- authored monster composition -> fix monster authoring
- a stale fixture expectation -> fix the fixture expectation
- a wrong axis route -> fix code / routing law

Only escalate to code changes when the issue is routing law, realization law, or a genuinely missing implementation seam rather than a numeric tuning problem.

Do not tune from one real monster unless it exposes a clear routing/code bug.

If a mismatch seems real, add it as a future calibration fixture first, then tune against the fixture set.
