03_Builders_And_Implementation_Roadmap.txt

Purpose

This is the active roadmap for Character Builder, Summoning Circle, Forge, and
remaining app work. It consolidates useful current roadmap content and removes
stale step-by-step prompt chains.

This document is implementation roadmap authority, not combat rule authority or
final costing law. Combat rules live in 01; power grammar and costing live in
02.


Current Builder Surfaces

Summoning Circle:

- First live implementation surface for the shared power architecture.
- Uses the shared power core in first-pass form.
- Supports practical chassis selection, packet-local authoring, Counter,
  Commitment Modifier, Lifespan where relevant, timing-gated descriptor
  rendering, and power-cost integration.
- Remains the reference authoring surface for shared power legality.

Character Builder:

- Reuses the shared power builder direction rather than a character-only fork.
- Stores player-authored powers in CampaignCharacter.builderData.
- Uses the shared power descriptor renderer and canonical Phase 6 cost resolver.
- Uses a player-facing Power Point pool: Character Level x 50.
- Converts BasePowerValue to player Power Point spend through the current
  player power spend scalar.
- Has current runtime support for Signature Move with a separate pool:
  Character Level x 20.
- Signature Move should remain one authored move, use the same power builder
  rules, and render at the top of the Power Sheet.
- Equipment and Backpack flows are campaign-local and must hydrate current item
  details rather than copying stale item data into the character.

Forge:

- Forge creates and renders campaign-local items.
- Forge output work should compare item output by table-facing profile, not raw
  backend fields or source origin.
- Forge implementation should not be rewritten merely to support Character
  Builder. Reuse descriptor and output helpers where possible.


Current Complete Or Substantially Complete Areas

- Shared power foundation phases 1 through 5 are complete enough to close as
  design phases.
- Summoning Circle shared power implementation is substantially complete.
- Monster power builder has a minimal but real implementation.
- Builder-surface validation and preview formatting are well advanced.
- Phase 6 infrastructure exists: Power Tuning, Combat Tuning, Outcome
  Normalization, Canary Harness, and Combat Lab batch paths.
- Character Builder has real save/load/editor/preview/print paths and shared
  power-builder usage.
- Character Builder Signature Move is implemented in current runtime, despite
  older roadmap text that said it was deferred unless reopened.


Remaining Work

- Keep Character Builder and Summoning Circle power legality in sync through
  shared pure helpers where practical.
- Continue validating Character Builder save/load, budget validation, preview,
  and print parity.
- Continue removing duplicated or stale power-builder logic between Summoning
  Circle and Character Builder.
- Preserve Packet 1 as the sole defence gate.
- Keep secondary packet dice ownership out of Character Builder V1 unless a
  later rule changes it.
- Keep packet duration turns capped according to current validation rules.
- Continue real-monster and real-character validation after mechanical changes.
- Build printable power reference output and effect-slip output when scheduled.
- Add runtime active-effect model and optional encounter runner later.
- Build future Setting Package policy and tooling after campaign-local workflows
  are stable.


Launch Roadmap Backlog: Builder And Toolkit

These items are captured roadmap/backlog, not locked rule law.

Character Creator:

- Character Creator is still missing Spark support.
- Character Creator is still missing Power Restrictions support.
- Spark and Restriction scope remains OPEN / NEEDS ARCHITECT DECISION.
- Develop Signature Move uniqueness remains OPEN / NEEDS ARCHITECT DECISION.
  Character Builder should eventually support any Signature-specific
  presentation, restrictions, scaling hooks, cooldown/friction display, and
  preview/print treatment required by the final design. Current implementation
  truth remains one Signature Move with a separate Character Level x 20 pool.
- Develop Signature Move Priming remains OPEN / NEEDS ARCHITECT DECISION.
  Character Builder / Power Builder may eventually need to display and validate
  Signature start state, Priming X, conditional priming, or over-limit-gated
  availability if the final design requires it. Preview and print output may
  need to show priming clearly. Do not implement or lock this now.
- Character Builder must eventually support optional Archetypes, Backstory
  Points, Personal Goals, and any character-sheet tracking needed for roleplay
  resources.
- Party resource tracking may belong at campaign / party level rather than
  character level; keep this open.

Power Builder:

- Full Summon intention implementation remains future work.
- Full Transform intention implementation remains future work.
- Summon / Transform should likely reuse or open a constrained Summoning
  Circle-style creation flow, but this is not locked.
- Summon creation should be constrained by originating power value, potency,
  dice, or equivalent power budget rather than ordinary level / tier creation,
  pending design.
- Transform creation should likely create a transformed profile or form with
  changed Attributes and Powers, pending design.
- Summon / Transform must preserve shared power-builder legality and avoid
  character-only or monster-only forks unless a future explicit rule requires
  one.

Closed Beta Vertical Slice / Roleplay Ability Testbed:

- Intended v0 direction: separate Roleplay Ability Builder section using shared
  visual/component language with the Power Builder.
- Builder entry points:
  - Add Power opens Power Builder.
  - Add Ability opens Roleplay Ability Builder.
- Roleplay Abilities should avoid unnecessary combat-specific v1 fields in v0.
- v0 Roleplay Ability builder fields:
  - Name
  - Theme / Description
  - Roleplay Intention
  - Specific
  - Scene Impact
  - Scope
  - Outcome Contract
  - Dice Count
- Optional / conditional fields:
  - Restriction
  - Counter permission where the selected standard contract permits it
  - proposed Outcome Lane and Custom Outcome Request for Custom Outcome review
- Standard-contract-derived fields:
  - Outcome Lane
  - Success Outcome
  - Descriptor
  - contract compatibility and Counter eligibility
  - eventual privilege-cost adjustment and cooldown pressure
- Theme / Description is unrestricted narrative method and manifestation text.
  It is not costed or parsed and cannot enlarge or rewrite the selected Outcome
  Contract.
- The structured Outcome Contract is mechanically authoritative and defines the
  reusable purchased privilege, derived lane/outcome, compatible authoring,
  Counter eligibility, and eventual costing/cooldown pressure.
- Scene Manifestation is narrated by the Game Director from current context.
  "Theme explains how. The Outcome Contract defines what."
- Remove free-text Success Outcome from standard contract authoring.
- Initial standard registry only:
  - HIDE_FROM_IMMEDIATE_DANGER: Help; Intervention / Rescue / Minor / One
    Target; Counter ineligible.
  - DENY_IMMINENT_HOSTILE_ACT: Hinder; Intervention / Interrupt / Major / One
    Target; Counter eligible.
- Filter standard contracts by Intention, Specific, Scene Impact, and Scope.
  Keep Custom Outcome - GD Review Required available for all combinations.
- Custom Outcome stores a proposed lane and request, remains Pending GD
  Approval, cannot be automatically costed, and has pending final Cooldown.
- Powers calculate quantified effects. Roleplay Abilities establish outcomes;
  they do not convert into wounds, Block, Dodge, Resist, stacks, units,
  measured movement, or other quantified Power mechanics.
- Roleplay Abilities v0 do not author Ability Shape or Duration.
- Ability Shape is removed from v0 because Sustained, Prepared, and Triggered
  are deferred / removed, and a lone Immediate shape would be clutter.
- Standard Use in combat consumes the user's Power Action. Out-of-combat use is
  an ordinary scene declaration with no combat action cost.
- Roleplay Window use costs 1 Response, is universal to Roleplay Abilities, and
  is not an authored or purchased permission. Only one Roleplay Ability may be
  declared in each Game Director-opened Roleplay Window.
- Roleplay Abilities cannot be used as normal Assists. Preserve ordinary Power
  Assist behaviour unchanged.
- Counter uses 1 Response and requires the actor to be personally targeted by a
  hostile action / effect plus authored paid Counter permission. Counter adds
  flexibility; it does not make the Ability Counter-only.
- Attribute should not be a builder field.
- Builder should not suggest typical Attributes.
- The Game Director validates the selected contract, derived lane/outcome,
  Theme / Description, Scope, Scene Impact, fictional position, and Restriction,
  then declares Difficulty and Attribute. On success, no additional roll or
  target action is required to establish the contract outcome.
- Potential costing remains non-numeric in this slice: Scene Impact x Scope;
  Outcome Contract privilege adjustment; Dice Count reliability; optional
  contract-permitted Counter surcharge; Cooldown adjustment; Restriction
  discount applied last. Custom Outcome has no automatic costing.
- Cooldown should derive from pre-Restriction / gross cost, not discounted net
  Potential spend.
- Show non-judgemental Combat/Roleplay investment profile and a shared remaining
  Potential summary:
  - Powers spend
  - Roleplay Abilities spend
  - Remaining Potential
- Avoid warning language implying roleplay-heavy builds are suboptimal.
- For the first slice, test-card representation may be used temporarily where
  implementation timing requires it, but roleplay field model and intent should stay
  aligned with the v0 framework.


Campaign-Local Asset Model

At the current development stage, Characters, Monsters, and Forge Items are
campaign-local.

Character Builder should equip from the character's Backpack, not from every
campaign item.

Equipped selections should be slot-based references to assigned Backpack rows.
The Builder should hydrate current item details from Party Inventory / Forge item
templates and sanitize slot selections against current Backpack quantities on
load/save.

Backpack items do not affect derived combat stats unless equipped.

Future Setting Packages may introduce global protected source assets. When that
happens, active/global package assets require Architect approval before mutation.
Campaign-local clones remain editable inside approved balance-test campaigns.


Source Mutation Policy Summary

Codex may create and edit Characters, Monsters, and Forge Items only inside an
approved balance-testing scope.

Inside the dedicated balance campaign, campaign-local calibration characters,
monsters, Forge items, temporary variants, cloned test assets, and
simulation-only overrides may be edited for calibration.

Codex may not freely alter:

- assets marked baseline
- assets marked canonical
- assets marked locked
- accepted calibration anchors
- demo/playtest campaign assets outside the balance-test scope
- any campaign or asset explicitly reserved by the Architect

If an asset is used as a comparison baseline, clone it before mutation.


Builder Rules To Preserve

- Summoning Circle remains the reference implementation for shared power
  authoring legality.
- Character Builder must not invent a character-only targeting fork.
- Range defines reach/area; target legality defines who may be selected; target
  count limits selected targets; self consumes one target slot when legal.
- Shared builders inherit Block, Dodge, and Resist defensive power options.
- Movement powers may be Counter-enabled, but Movement/Reposition is not a
  Defence mode.
- Cleanse remains distinct from Resist.
- Attack packets require damage type selection.
- Primary Defence Gate is derived from Packet 1, not chosen freely by default.
