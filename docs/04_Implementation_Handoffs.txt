Rule for Future Builder Work

All future builder work should assume:

- one shared power core
- Summoning Circle as the first implementation surface
- Character Builder later reusing the same foundation
- standard monsters do not use the full player Spark / Restriction layer
- live preview is mandatory
- simple powers must stay simple
- only Packet 1 is defended against
- Lifespan, Effect Timing, and Effect Duration are separate concepts
- Counter is an authored Power-Level Field with No / Yes
- Assist is universal by system rule and is not authored per power
- Responses are the player-facing resource and are not the authored power permission field
- Trigger remains structurally separate from Counter
- Commitment Modifier uses Standard / Channel / Charge
- Channel uses one first-pass behaviour only as an action-commitment modifier
- first-pass Channel support is limited to Immediate, Field, and Attached
- hostile delayed-entry Attached powers must explicitly distinguish gate on attach from gate on payload
- Immediate + Standard Packet 1 currently uses On Cast only in the builder surface
- if an Immediate Packet 1 resolves On Cast and creates no beyond-turn carrier, later Immediate packets remain On Cast only in first implementation
- secondary scaling from the Primary result class is derived in the current builder surface rather than manually guessed
- persistent states use a stack-cleanup model distinct from ongoing direct effects
- ongoing damage uses ongoing value plus duration rather than stacks
- physical ongoing-damage cleanup defaults to Fortitude Resist and mental ongoing-damage cleanup defaults to Bravery Resist
- packet-local recipient class may distinguish Primary Targets, Allies, and Self where a Secondary Packet needs a different target class from Packet 1
- Allies includes self by default unless an explicit authored restriction says otherwise
- packet-local recipient class must remain legal within the shared cast context
- ambiguous Cleanse targeting of Active Power or Channelled Power may use Cleanse Theme only where a unique cleanup Resist Attribute is not otherwise already settled
- Trigger powers may use Trigger Method to distinguish Arm and then target from Target and then arm while keeping Packet 1 on trigger
- Secondary timing choices must obey both chassis legality and Packet 1 carrier legality
- Immediate remains the only chassis that can honestly have no Lifespan requirement in the current builder surface; Field, Attached, Trigger, and Reserve require Turns or Passive and default to Turns
- self-origin sphere wording should distinguish static areas centered on the user's current space from self-hosted Attached areas attached to the user
- overlapping areas from the same packet at the same timing use one power roll, pre-defence multiplication by overlap count, and one defence roll

Future builder work should also preserve the following implementation-facing direction:

- shared cast context should remain centered in Power-Level Fields
- structural state should be represented through the selected chassis
- packet outcomes should be represented through Effect Packets
- resolved gates, legality warnings, and preview wording should be treated as derived metadata rather than co-equal authored truth
- local origin and local targeting overrides should stay behind advanced controls


Rule for Future Phase 6 Work

All future Phase 6 work should preserve the current three-layer split:

Power Tuning
- canonical power truth

Combat Tuning / Monster Baseline Tuning
- non-power monster baseline contribution

Outcome Normalization
- downstream radar interpretation and display

Do not collapse these into one fuzzy tuning system.

Use this ownership rule:

- if the power itself is wrong, tune Power Tuning
- if stripped monsters are too weak or too strong before powers, tune Combat Tuning
- if the combined result displays wrongly on the radar, tune Outcome Normalization
- if the issue is routing law, realization law, or a missing implementation seam, then it is a code / architecture task rather than a numeric tuning task

The current live implementation already uses:
- canonical power contribution from the Phase 6 resolver
- non-power contribution from the stripped monster baseline and other monster systems
- final outcome normalization on top

Do not resurrect old legacy power-side heuristics beside this cleaned pipeline.


Rule for Future Packet Resolution Work

All future packet-resolution work should assume:

- Packet 1 is always Primary
- only Packet 1 is defended against
- Secondary Packets remain contingent
- Packet 1 lands if it applies at least one operative result unit
- wound-based Secondary scaling uses Applied Success Bands = CEILING(Net Wounds / Wounds Per Success)
- mixed-intention legality is constrained by hostile-primary ordering
- packet timing legality must not smuggle in a different chassis
- Attached powers distinguish initial legal range from later host-bound persistence
- Channelled Attached reinforcement requires the target to remain within legal cast range from the source
- hostile delayed-entry Attached structures must keep gate on attach and gate on payload explicit


Rule for Future Runtime Work

Runtime and tracking tools are later work.

Do not let future chats accidentally design a second contradictory state model.

Printable effect slips and digital effect tracking should grow from the same power structure as the builder outputs.

Do not let runtime speculation contaminate builder truth before the builder is stable.


Rule for Future Implementation Chats

Implementation chats should work from current files only.

Do not reason from memory if a newer file version exists.

Do not produce schema or UI assumptions against unseen code.

If current files are missing, request them.

If a remaining unresolved design boundary does not materially block implementation, do not stop to philosophize about it.

If a remaining unresolved design boundary does materially block implementation, resolve only the smallest hard rule needed to unblock honest execution.


Current Open Questions That Are Real

The following questions are still genuinely open and should be treated honestly:

- the final schema ownership boundary for Lifespan
- the final schema boundary between Power-Level Fields, chassis-local fields, packet-local fields, and derived metadata
- the exact long-term treatment of costs that are currently cost-only versus later translated through cadence, realization, or structural pressure
- the exact monster-side realization modifier layer that should gate typed contributor output through monster attributes without contaminating canonical power truth

These are real open questions.

They are not permission to pretend the whole architecture is still vague.
