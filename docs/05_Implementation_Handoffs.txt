Purpose

This document contains durable implementation notes for Codex and developers. It
does not replace the rule-law document. It exists to prevent future work from
assuming runtime compliance without checking code and tests.


General Implementation Rule

Inspect current local files before patching. Do not assume older docs, prior
chat memory, or GitHub state is source of truth.

When docs and runtime disagree:

- If the doc is a locked rule, treat runtime mismatch as an implementation issue.
- If the doc is a roadmap or historical prompt chain, verify current code before
  preserving it.
- If the issue is unclear, mark OPEN / NEEDS ARCHITECT DECISION.

Rules That Need Runtime Protection

Future implementation work must keep checking:

- Natural 1s are hard failures.
- Natural 2s and 3s are recoverable failures.
- Only natural successes can spike.
- Zero-success hostile actions trigger declaration-based effects but do not
  cause normal defence roll/degradation without actionable hostile successes.
- Defence degradation is tracked independently by lane.
- Pool-only defensive-pool commitment does not degrade normal defence.
- Counter use replaces normal active defence unless the Counter explicitly
  grants or modifies that defence.
- Generic Assist cannot create an independent hostile action or independent
  hostile payload. Offensive Assist output becomes Assist pressure against the
  triggering ally's opposition, not damage or hostile effects. Assist pressure
  is output-stage opposition suppression, not a roll modifier or payload boost:
  it reduces only legal matching opposition after rolls become outputs, floors
  at 0, wastes excess pressure, does not create wound bands, does not apply
  damage types, and does not trigger damage/wound effects.
- Multiple generic Assists on the same trigger stack only through distinct Power
  Intentions unless explicit group-assist, combo, formation, ritual, squad
  tactic, or similar authoring says otherwise.
- Dual-Lane Defence Responses are a narrow Counter/Assist exception: roll once
  and apply complementary Physical Defence and Mental Defence packets to their
  matching wound lanes from the same trigger.
- Independent eligible Secondary Packets resolve as one simultaneous result
  bundle after the Primary Packet; authored order is presentation order unless
  explicit sequential/dependent/triggered language says otherwise.
- Multiple responders use acting-side-first support assembly: acting-side
  Assists declare and resolve first, target-side Assists declare and resolve
  second, targeted defenders then declare defence or Counter choices, then the
  response window closes.
- Passive defensive pools do not naturally tick down.
- Movement/Reposition is not a Defence mode.
- Dodge succeeds on match-or-exceed.
- Mental attacks require Mental Defence unless a specific action explicitly
  allows Dodge.
- Main Action Resist cleanup remains distinct from active defence Resist,
  Defence -> Resist powers, and Cleanse.
- Full damage-prevention order must preserve the layer split: Protection adds
  dice, Resistance/Vulnerability modify defence rolls, Block and defensive
  pools prevent output-stage wounds, Dodge avoids legal packets, and Assist
  pressure suppresses opposition without becoming damage.
- Cleanse must use declared target/source/category eligibility. Do not let a
  generic Cleanse silently remove every hostile state on the target.
- Same-source effect reapplication refreshes/replaces by default; different
  sources can coexist; same-condition binary burdens resolve as one burden while
  their source instances remain individually tracked.
- Default wound-band scaling uses actual wounds suffered after prevention and
  remaining-health cap. Overkill wounds do not count unless explicitly authored
  as a premium overkill-scaling exception.
- Action-level wound-band Secondaries may pool actual wounds across valid
  Primary targets once. Target-local Secondaries use only that target's own
  actual Primary wounds.
- Health-channel defeat, Major Injury, Blaze of Glory, and defeat cleanup must
  follow actor type: non-Legendary Monsters die at 0 in either health channel;
  Player Characters and Legendary Monsters make affected-channel Major Injury
  Trials after the current simultaneous packet/result bundle unless same-bundle
  healing restores the pool above 0. Player Character Blaze is available only
  before a Terminal Major Injury Trial; Legendary Monsters retain their separate
  simplified Blaze permission. Runtime
  should represent this through an explicit defeat model / actor category rather
  than inferring only from monster tier.


Combat Lab Handoff Notes

Combat Lab and future runtime tools should implement one Response Window /
Incarnate Stack model.

When both sides have Responses to the same triggering action, runtime-facing
tools should assemble support in acting-side-first, target-side-second order.
Acting-side support declares and resolves first to establish the final incoming
threat. Target-side support declares and resolves second to establish the final
answer. Targeted defenders then declare normal defence or Counter choices,
including legal pool commitments. The window then closes unless a future
explicit rule allows Responses to Responses.

Normal active defence should only roll if no Counter was declared by that
defender and the hostile action produced one or more actionable hostile
successes.

Defensive-pool state should track at least:

- source actor
- source power
- source packet/action where available
- protected actor
- pool type
- lane / resisted Attribute where relevant
- remaining points
- initial points
- per-trigger cap
- remaining duration or passive lifecycle
- chassis/commitment metadata where available
- refresh/replace key

Reports should expose defensive-pool diagnostics clearly enough to see pools
created, generated, committed, spent, wasted, remaining at expiry,
refresh/replace events, Dodge avoids, Block prevented, and Resist cancelled.

Secondary Packet simultaneous-bundle runtime/test alignment should be checked
in a separate implementation pass. Do not assume current runtime already nets
same-target, same-timing, same-lane independent Secondary damage/healing before
defeat, Major Injury, or cleanup checks unless code/tests prove it.

Assist pressure and distinct-Power-Intention Assist stacking runtime/test
alignment should be checked in a separate implementation pass. Do not assume
current runtime already converts hostile Assist payload into Assist pressure,
prevents independent hostile Assist payload, or enforces duplicate generic
Assist Power Intention limits unless code/tests prove it.

Pool-with-Counter runtime/test alignment should be checked whenever Counter or
defensive-pool code is touched. A Counter does not automatically spend existing
defensive pools. Pool points may contribute only when the Counter resolves a
matching defensive path or explicitly grants pool compatibility. Attack-only,
Movement-only, and Control-only Counters should leave nonmatching pools active
and unspent.

Cleanse taxonomy runtime/test alignment should verify declared effect/source
selection, source-category eligibility, potency/success scaling, and separation
from cooldown immunity, source lifecycle cleanup, passive carrier ending, field
ending, and attachment breaking.

Duplicate effect runtime/test alignment should verify same-source
refresh/replace, different-source coexistence, same-condition combined burden
presentation, source-specific removal, lowest-stack-first removal where no
source is declared, and oldest-first tie-breaks.

Wound-band runtime/test alignment should verify actual-wounds-suffered scaling,
remaining-health cap, no default overkill farming, action-level multi-target
pooling where explicitly authored, and target-local isolation by default.

Health-channel runtime/test alignment should verify Non-Legendary Monsters die
at 0 in either health channel; PC / Legendary Major Injury Trials by affected
channel; overflow severity modifier of -1 per full actor level of overflow,
rounded down; exactly 3 dice of a chosen relevant-side Attribute; normal success
counting of 1-3 = 0, 4-9 = 1, and 10+ = 2 successes; overflow modifier applied
to each die result before success counting; 0-1 total successes = Major
Injury, 2 = Minor Injury, and 3+ = No Injury; separate Physical and Mental
injury tracks; dual-channel 0 in one bundle creates both Major Injury events for
PC / Legendary actors; Player Characters may replace only terminal decisions
with the locked single/Shared Blaze procedure, while Legendary Monsters retain
their simplified separate Blaze rule; Mental injury events
resolve before Physical injury events as a runtime ordering rule; same-bundle
healing can prevent a trial if the relevant pool is restored above 0 before
injury processing; Blaze of Glory timing and final removal; and no mid-bundle
defeat cleanup. Overflow must be captured from the triggering damage event and
must not be recomputed from accumulated negative HP; after injury resolution,
the affected Health pool should be normalized to 0 unless the actor is defeated.

Zone Range And AoE Template Handoff

LOCKED DESIGN DIRECTION / DETAILED SPATIAL RULES AND IMPLEMENTATION OPEN. Do not
implement this direction until a dedicated Spatial System Design project has
resolved zone ontology, range bands, movement/transitions, engagement and
line-of-sight/effect, AoE dimensions and inclusion, forced movement/teleport/
flight/verticality, large creatures/moving scenes, costing validation, and an
explicit data/code migration.

The eventual implementation must preserve:

- relative zones for ordinary movement, melee/ranged/Power reach, position, and
  scene travel without ordinary square counting;
- 1 inch / one standard square = 5 feet for battlemaps, miniatures, terrain,
  template footprints, and digital overlays;
- meaningful miniature position inside zones for templates, terrain, barriers,
  sight/effect, features, and tactical placement;
- map zones and connections established before combat or mechanical relevance,
  with changes driven by changed fiction rather than tactical convenience;
- Sphere, Cone, and Line, each with three ordered template Size bands;
- remote origins for all three shapes, with no baseline Placed/Emitted split;
- only the centre/apex/start origin required inside legal origin range, while
  the footprint may project beyond range and cross zones;
- actual barriers rather than abstract zone boundaries governing future line
  of effect;
- physical/digital template coverage rather than automatic whole-zone or
  subjective-group coverage; and
- equivalent physical, printable, and digital delivery with no premium
  accessory requirement.

The current Builder, stored data, descriptors, calculators, Combat Lab, and
runtime use exact-foot and geometric fields. Preserve those fields and all
current exact-distance data until the migration is separately authorized. This
handoff authorizes no Range-enum, movement, dimension, template, cost,
descriptor, Summoning Circle, Character Builder, Monster Builder, Forge, Combat
Lab, schema, or data change.

Keep Origin Range and AoE Size as separate tuning seams. Size is the heavier
spatial cost driver; Origin Range changes placement freedom but does not enlarge
the template. Do not implement numeric costs, multipliers, dimensions,
inclusion rules, shape-specific values, or furthest-edge Range pricing from this
direction alone.


Balance Handoff Notes

Do not tune from one scenario unless it exposes a clear code/routing bug or
matches an established canary failure.

North Star sequencing for balance work:

- Official conclusions require runtime-resolved evidence using Balance
  Environment-authored assets through Combat Lab/runtime resolution.
- Balance Environment is the authored test-data scope, not the simulator.
- Start with attack string calibration for the level.
- Then perform defence string calibration for the level.
- Then tune the attack/defence offset so mitigation matters while preserving
  the Attacks-to-Kill bands.
- Only then tune real role assets: characters, monsters, powers, Forge items,
  Summoning Circle monsters/summons, and other app-authored combat assets.
- Align costs last. Cost values follow runtime truth.
- If a real role asset value pass happens before this sequence is complete,
  label it an experimental value pass, record before/after values, and wait for
  review before treating it as accepted doctrine.

Do not tune against the current post-c89338d formula shape as final truth. The
formula-shape audit found that current data paths are useful, but serious
balance tuning is blocked until formula architecture is corrected or explicitly
accepted as an interim model.

Keep the split between:

- Power Tuning
- Combat Tuning
- Outcome Normalization

Layer A deterministic canaries are mandatory mechanical guardrails. Layer B
synthetic batches are diagnostic only. Real authored scenarios are primary
table-truth balance evidence.

B. Add formula unit/example tests without changing formulas.

   impact.

- Signature Move uses the same base formula as ordinary powers unless a future

- pure Movement powers stay out of automated balance evidence until Movement

Monster Tier Reporting And Tuning Handoff

Before tuning Balance Environment assets heavily, add or improve reporting for
attacks-to-defeat / meaningful offensive actions-to-defeat if feasible.
Official PASS / LOW / HIGH durability judgement should use the Average Medium
Strength Attack profile: Minion 1-2 Medium Strength Attacks, Soldier 2-3, Elite
4-6, Boss/Legendary 16+. Low and High attack profiles are diagnostic range
checks only. Reports should distinguish attacks from turns wherever possible and
state any one-attack-per-turn assumption.
Reports should distinguish:

- weapon/natural attack damage
- Power attack damage
- same-turn burst where a weapon/natural attack and Power attack both land in
  the same actor turn
- successful hits-to-defeat
- turns-to-defeat
- rounds-to-defeat
- overkill
- first-turn deletion
- stale rate

Do not tune Soldiers to 1v1 Elite standards. Soldier solo scenarios are
diagnostics for role expression and whether a Soldier survives around the
expected 2-3 Medium Strength Attacks. Add future presets for Soldier group
pressure and Elite duel core before using results as stronger balance evidence.

Mental Wailer remains a specific Soldier red flag if it beats a PC heavily in
1v1. That does not mean all Soldiers should be tuned to win 1v1; it means the
Mental Wailer lane, control/damage pressure, or counter-matchup shape needs
inspection against the locked Soldier attacks-to-defeat doctrine.

is owned by docs/02_Power_System_And_Costing.txt; commitment-time runtime law is

Power Burdens And Blaze Implementation Handoff

LOCKED DOCTRINE / CURRENT IMPLEMENTATION INCOMPLETE. Runtime law is owned by
docs/01_Locked_Rules_And_Combat_Bible.txt and non-numeric authoring/economic law
by docs/02_Power_System_And_Costing.txt. Do not infer implementation completion
from those locks.

Future implementation must preserve:

- zero/one whole-Power Burden, with Activation Cost and Backlash mutually
  exclusive, for both Player Character and Monster Powers;
- Power-only Burdens and the Roleplay no-Burden boundary;
- separate normal and Burden descriptors;
- pre-Response Activation Cost, post-initial-execution Backlash, absolute
  application, and the committed-use cooldown obligation;
- terminal-only Player Character Blaze, Activation-Cost-triggered Blaze, Blaze
  opening state, Assists, special actions, Limit Break donation, permanent
  death, and Shared Blaze; and
- during Shared Blaze, only an eligible non-Blazing allied Player Character may
  sacrifice one future turn, choose one Blazing recipient, apply one selected
  special action only to that recipient, and donate dice only to that same
  recipient; and
- separate simplified Legendary Monster authority until later design approves
  anything more.

Monster Burden implementation must reuse the shared mechanical timing and
absolute-application rules, with no Monster-only variant, stacked Burden, or
packet-local constraint. GDs author Monster Activation Costs and Backlash
directly under structural validation and normal Monster edit permissions; do
not invent an approval lifecycle for them. Preserve authenticated immediate
approval-on-save provenance for valid GD-authored Monster Restrictions.

No Monster constraint may feed Player Character credit equations, Gross/Net
BPV discounts, cooldown reduction, threat budgets, the Outcome Calculator,
tier or encounter allowances, Effective Encounter Load, or an automatic
balance/fairness claim. Combat Lab/runtime evidence for the exact Monster,
Player Characters, and encounter assumptions remains matchup authority.

Prohibited shortcuts:

- no packet-local Activation Cost or Backlash;
- no several Burdens and no Cost-plus-Backlash Power;
- no sequential percentage application to an already reduced subtotal;
- no prevention, redirection, conversion, refund, or same-Power mitigation of a
  Burden;
- no Cleanse, source removal, expiry, interruption, failure, miss, resistance,
  or early ending cancelling owed Backlash (no success fishing);
- no mixed donor and recipient Limit Break die sizes in one Blaze roll;
- no literal extra turns, turn-starts, refreshes, or duration ticks from Make It
  Count! action packages;
- no Power Action silently granted by Hit It Again!;
- no Make It Count! package, Hit It Again! cooldown removal, or same-declaration
  Limit Break donation split between or duplicated across Blazing characters;
- no Blazing character sacrificing a nonexistent future turn;
- no automatic merged attack or universal combined defence for a Shared Blaze
  simultaneous beat; and
- no implementation prompt applied without inspecting current files, schemas,
  runtime order, tests, and migrations first.

Current gaps include structured Burden registries, numeric tuning authority,
Activation Cost/Backlash runtime state, terminal and Shared Blaze orchestration,
special actions/turn debt/donation, coordinated-beat resolution, persistence,

Codex Tooling Guidance

Use attached prompts as task instructions, but still inspect local files before
patching.

Use local terminal commands for repo inspection and validation.

When running simulations, report provenance. When modifying files, report files
changed and validation results.

Live-authored balance testing must report campaign provenance. The official
balance campaign is Balance Environment
(250aee5e-632f-405c-ba36-a49ed12a5afc). Future scripts and reports that use
live-authored Characters, Monsters, Forge Items, or encounters should include
campaignId, campaignName, assetSource, exact runtime/adapter path, repo commit,

balance conclusions require runtime evidence using Balance Environment-authored

outputs, passives, control effects, conditions, reactions, resources, riders,
