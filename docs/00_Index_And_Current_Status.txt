Current Known Implementation State

The shared power architecture is no longer hypothetical. Current source docs and
runtime inspection show:

- Summoning Circle is the first live implementation surface for the shared power
  model.
- The power model uses Power-Level Fields, one Descriptor Chassis, and one or
  more Effect Packets.
- The practical Descriptor Chassis set is Immediate, Field, Attached, Trigger,
  and Reserve.
- Counter and Commitment Modifier are power-level controls.
- Commitment Modifier uses Standard, Channel, and Charge.
- Phase 6 cost integration exists in code through a canonical power-cost
  resolver.
- Versioned Power Tuning, Combat Tuning / Monster Baseline Tuning, and Outcome
  Normalization exist as separate tuning layers.
- A Canary Harness and scenario catalogs exist for baseline/candidate
  comparisons.
- scripts/combatLab.scenarioMatrix.ts exists and should be extended rather than
  bypassed for scenario-matrix work.
- Combat Lab can be driven by local scripts or runtime calls for incremental
  balance checks when provenance is reported.
- Character Builder now uses the shared power builder direction and has runtime
  support for player-authored powers.
- Character Builder Signature Move support is present in current code with a
  separate Character Level x 20 pool. Older docs that say Signature Move is
  deferred are superseded by current implementation state, but save/load,
  validation, preview, and print parity should continue to be verified.
- Character Builder power pool remains Character Level x 50 unless explicitly
  changed later.
- Character Builder power spend derives from BasePowerValue through the
  player-facing spend scalar. Formula harnesses must load the same active
  Character Builder tuning source as the live app, currently
  characterBuilderTuning/default, instead of falling back to stale audit values.
- Campaign-local Backpack and equipped gear flows exist as the current asset
  model; future Setting Packages remain a later protected-asset layer.


Current Balance And Tuning State

The current tuning model has three distinct layers:

- Power Tuning: canonical underlying power-cost truth.
- Combat Tuning / Monster Baseline Tuning: non-power monster baseline
  contribution.
- Outcome Normalization: downstream radar interpretation and display.

Anti-drift rule:

- If a problem belongs to the power itself, fix it in Power Tuning.
- If a problem belongs to stripped monster body or non-power baseline, fix it in
  Combat Tuning.
- If a problem belongs to radar display curve, fix it in Outcome Normalization.
- Escalate to code only when the issue is routing law, realization law, or a
  missing implementation seam.

Current defensive-pool pricing work should not begin from raw win rate alone.
Layer A deterministic canaries must protect mechanics first, Layer B synthetic
batches are targeted diagnostics, and real authored scenarios remain primary
table-truth balance evidence.

Serious numeric balance tuning is additionally gated by formula architecture.
The current formula work has corrected Monster Outcome Calculator Phase 1
expected output for supported natural/equipped attacks, but packet-level power
expected output still depends on callers supplying packet data, and structured
trait reliability, Forge enforcement, and final scenario evidence remain open.
Balance Environment assets are currently useful for data-path testing but are
not final balance evidence until campaign-scoped runs validate the current
formula/runtime state.
