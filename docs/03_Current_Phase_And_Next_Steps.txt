What Is Now Locked

The following points should now be treated as stable design truth:

- the shared power core uses one three-layer model
- Delivery Method is structural and does not replace shared rules grammar
- Delivery Method is implemented through Descriptor Chassis
- Packet 1 is always the Primary Packet
- only the Primary Packet is defended against
- Secondary Packets only resolve if the Primary Packet lands
- Secondary Packets do not create separate defended entry points
- Packet 1 lands if it applies at least one operative result unit
- wound-based Secondary scaling uses Applied Success Bands = CEILING(Net Wounds / Wounds Per Success)
- default wound-band size uses the Primary Packet's total simultaneous wounds per success across all damage components; multi-damage primaries do not create separate free rider bands per damage type
- Secondary Packets are always powered by the Primary Packet result; if Packet 1 applies no operative result, later Secondary timing does not fire
- Field, Attached, Triggered, and recurring Primary delivery rolls once on the first matching trigger each round and reuses that stored roll for later same-round triggers
- non-primary-recipient Secondary payloads use the first successful Primary application in the round; Primary Target secondaries use that target's own applied Primary result
- Secondary Packets own their own Timing and Duration and must not inherit a parent / Primary / chassis duration
- Lifespan, Effect Timing, and Effect Duration are separate concepts
- Summoning Circle is the first implementation surface for the shared core
- Character Builder is expected to reuse the same foundation later
- live preview is mandatory
- the practical Descriptor Chassis matrix is Immediate, Field, Attached, Trigger, and Reserve
- Counter is an authored Power-Level Field permission with No / Yes
- Assist is universal by system rule and is not authored per power
- Responses are the player-facing resource and are not the authored power permission field
- Commitment Modifier uses Standard / Channel / Charge
- first-pass Channel support is limited to Immediate, Field, and Attached
- hostile delayed-entry Attached powers must explicitly distinguish gate on attach from gate on payload
- Immediate + Standard Packet 1 currently uses On Cast only in the builder surface
- if an Immediate Packet 1 resolves On Cast and creates no beyond-turn carrier, later Immediate packets remain On Cast only in first implementation
- secondary scaling from the Primary result class is derived in the current builder surface rather than manually guessed
- persistent states use a stack-cleanup model distinct from ongoing direct effects
- ongoing damage uses ongoing value plus duration rather than stacks
- physical ongoing-damage cleanup defaults to Fortitude Resist and mental ongoing-damage cleanup defaults to Bravery Resist
- packet-local recipient class may distinguish Primary Targets, Allies, and Self where a Secondary Packet needs a different target class from Packet 1
- Allies includes self by default unless an explicit authored restriction says otherwise
- packet-local recipient classes must remain legal within the shared cast context
- Trigger powers may use Trigger Method to distinguish Arm and then target from Target and then arm while keeping Packet 1 on trigger
- Secondary timing choices must obey both chassis legality and carrier legality
- Immediate remains the only chassis that can honestly have no Lifespan requirement in the current builder surface
- Field, Attached, Trigger, and Reserve require Turns or Passive and default to Turns
- cooldown begins only after a power has completely resolved
- Immediate powers enter cooldown after their immediate resolution sequence completes and any continuing duration tracking has been noted
- non-Immediate Lifespan Turns powers treat cast / execution / release as Turn 0 and enter cooldown only after the final resolution window of the last lifespan turn closes
- Passive Lifespan powers enter cooldown only when voluntarily ended by the caster or when Cleanse / Resist reduces currently active successes to 0
- Counters and Assists use the same cooldown lifecycle, treating the executing Response as Turn 0
- Charge is front-loaded and lifespan tracking begins only when the charged power is released, cast, or otherwise executed
- overlapping areas from the same packet at the same timing use one power roll, pre-defence multiplication by overlap count, and one defence roll
- canonical attribute naming is Guard and Synergy, not Defence and Support
- Bravery is the mental defence dice stat
- Dodge now succeeds on match or exceed, not exceed-only
- natural 1s are hard failures, natural 2s and 3s are recoverable failures, and only natural successes can produce greater / critical / higher-tier spike results
- duration Dodge, Block, and Resist powers create finite Dodge Pool, Block Pool, and Resist Pool resources rather than rerolling or refreshing their full value on every incoming attack
- repeated defensive pool applications from the same source power, packet, and protected target use refresh/replace higher-of-current-or-new by default, not additive stacking
- Field, Attached, and Channel defensive pool delivery uses authored timing/cadence while preserving finite pool limits; Trigger and Reserve use normal pool rules once delivered/released; Charge uses final release dice/potency

What Phase 6 Now Includes

Phase 6 no longer means only “design the cost model.”

The current implementation now includes:

- a canonical Phase 6 power-cost resolver
- first-pass derived cooldown values from BasePowerValue, with cooldown lifecycle rules treated separately from cost derivation
- a Power Tuning system with versioned active / draft / archived sets
- an Outcome Normalization system with versioned active / draft / archived sets
- a Combat Tuning / Monster Baseline Tuning system with versioned active / draft / archived sets
- a shared canary catalog and curated scenario catalog
- a Canary Harness that compares:
  - Active Power Tuning + Active Outcome Normalization + Active Combat Tuning
  - versus selected candidate tuning sets
- Codex-run Combat Lab simulation batches as an accepted incremental balance workflow when provenance is reported
- a shared stripped Summoning Circle baseline so live editor and harness comparisons now align
- a cleaned monster radar pipeline where:
  - canonical power contribution is derived first
  - non-power monster contribution is added second
  - outcome normalization is applied last

This means the tuning model now has three distinct layers:

Power Tuning
- canonical power truth

Combat Tuning / Monster Baseline Tuning
- non-power monster baseline contribution

Outcome Normalization
- final radar interpretation and display

The current non-power baseline work also now includes:

- normalized natural vs equipped at-will attack parity through one shared calculator-owned profile path
- normalized natural vs equipped protection parity through one shared calculator-owned defensive profile path
- physical-only dodge routing in monster survivability baseline work
- a separate mental protection contribution seam in Combat Tuning so MPV can be tuned independently from PPV in the monster calculator
- updated combat-tuning admin copy to remove stale shared-dodge language

Immediate Practical Next-Step Direction

The practical next work should stay focused on:

- implementing shared power-builder support for Block, Dodge, and Resist defensive power options in both Summoning Circle and Character Builder
- keeping Resist authored against one core Attribute lane per packet, while Cleanse remains the cleanup/removal mechanic
- treating Defence -> Resist Potency as Resists per successful power-roll success; multi-attribute Resist requires multiple packets
- implementing duration defensive powers as finite pools: roll once on apply, pool size equals successes x potency, commit up to potency per matching trigger as pool-only protection or before an optional matching normal defence / active Resist roll, and spend committed points whether they fully stop the incoming result or prove ineffective
- implementing chassis-specific pool handling: Field target-specific pools, Attached following pools, Channel refresh/replace pools, Trigger/Reserve delivered pools, Charge release pools, and one matching pool commit per incoming action unless explicitly modified
- updating descriptor wording so normal use and Counter use are clearly distinct: Counter-enabled powers remain normally usable, while Counter use replaces normal active defence
- preserving the locked targeting split across shared builders, descriptors, and Combat Lab: range defines reach / area, target legality defines who may be selected, target count limits selected targets, and self consumes one target slot when self is legal
- applying the same range-vs-target-legality rule to Dodge and Resist power implementation; Movement Counter remains a Movement-power rule, not a Defence mode
- implementing Combat Lab and future runtime handling through one Response Window / Incarnate Stack model
- preserving the locked response order: active effects, hostile declaration, Assist declaration, Counter declaration, Assists, Counters, trigger roll, normal active defence only if no Counter and one or more actionable hostile successes were produced, prevention/effects, result application, cleanup
- checking runtime/tests for the zero-success hostile action ruling: declaration/reaction triggers still fire, declared Counters/Assists remain spent, but normal defence, defensive-pool commits, and defence degradation do not occur without actionable hostile successes
- checking runtime/tests for independent defence-lane degradation: Physical Defence, Mental Defence, Dodge, and active Resist tracks must degrade only in the lane that actually resolves against actionable hostile successes
- checking runtime/tests for the defensive-pool-only commitment ruling: pool-only spending is legal against matching actionable hostile pressure, spends finite pool points, and does not degrade Dodge, Physical Defence, Mental Defence, or active Resist unless a normal active defence is also rolled
- rerunning the 16-test response / defence / Counter matrix after runtime changes
- revisiting Counter, Movement Counter, and Control interrupt Counter costs only after runtime behaviour faithfully follows the source rules
- using the Canary Harness for baseline/candidate tuning comparisons alongside Codex-run Combat Lab batches for repeated scenario evidence
- preserving the three-layer split between:
  - Power Tuning
  - Combat Tuning
  - Outcome Normalization
- using attack and defence anchor cases before broader family-wide tuning
- tuning physical dodge and PPV tradeoffs so the physical survivability lane has a believable breakpoint shape
- tuning mental protection contribution so the mental lane can stand beside the physical lane while preserving asymmetry
- resolving only genuinely blocking remaining builder / descriptor edge cases
- using Codex-run Combat Lab batches for incremental balance passes, scenario matrices, variant comparison, canary checks, and outlier detection where possible
- reserving manual Combat Lab runs for major milestone confidence, UI/report readability checks, playtest-packet confidence, public/demo build confidence, and table-feel judgement
- beginning the monster-side realization modifier layer only after the current tuning layers are behaving honestly
- keeping the source docs aligned with current code reality

Codex-Run Simulation Provenance Rule

Codex-run Combat Lab results are evidence only when the report states:

- commit SHA
- git status / worktree cleanliness
- exact command, script, route, API endpoint, direct runtime call, or other method used
- whether browser UI was visually inspected
- run count
- seed or seed policy
- scenario names
- actor and monster names
- active Power Tuning name
- active Combat Tuning name
- active Outcome Normalization name
- any temporary in-memory overrides
- confirmation that no database, source-data, saved-monster, tuning, or code changes were made during read-only simulation
- validation or smoke status where relevant

Recommended report fields are player win %, monster win %, stalemate %, average rounds, player and monster damage per round, action use counts, cooldown-prevented counts, defensive choice diagnostics, counter candidate diagnostics, ongoing pressure diagnostics, defensive pool diagnostics, transcript anomalies, and unsupported runtime notes.

Simulation reports must say whether results came from UI/browser inspection, an API route, a script, direct runtime calls, or another method.

Important Anti-Drift Rule

Do not let future chats collapse these into one blurred balance system.

If a problem belongs to:

- the power itself -> fix in Power Tuning
- the stripped monster body / non-power baseline -> fix in Combat Tuning
- the radar display curve -> fix in Outcome Normalization

Only escalate to code changes when the issue is routing law, realization law, or a genuinely missing implementation seam rather than a numeric tuning problem.
